# home-decoration DESIGN.md

本文件是本仓库的默认设计决策层。

作用边界：
- `AGENTS.md` 负责开发规则、验证规则、模块边界。
- `DESIGN.md` 负责页面气质、视觉层级、交互取向、页面选型。
- 未明确指定风格时，所有 UI 相关实现默认先遵守本文件。

本文件的唯一外部风格来源是：
- [`VoltAgent/awesome-design-md`](https://github.com/VoltAgent/awesome-design-md)

使用原则：
- 不原样复制某个站点的整份 `DESIGN.md`。
- 只提取其稳定、可复用、可执行的设计语言。
- 单页面只允许一个主风格方向，必要时最多叠加一个辅风格。
- 优先保证业务可读性、交易信任感、流程清晰度，再追求“好看”。

## 1. 项目设计总目标

`home-decoration` 不是纯营销站，也不是通用后台模板。它是一个覆盖用户、商家、管理、履约、交易的多端装修平台。

默认目标：
- 用户侧：建立信任、降低决策压力、突出阶段感和结果感。
- 商家侧：提高执行效率、突出状态变化、减少操作噪音。
- 管理侧：强调审查、风险、状态、批量决策能力。
- 多端统一：同一业务对象在不同端视觉语言一致，但页面形态允许因场景而异。

默认气质：
- 专业
- 温暖
- 克制
- 有秩序
- 不炫技

默认避免：
- 千篇一律的 AI 卡片风
- 紫色主导的泛科技风
- 过度玻璃拟态
- 无意义的大渐变与悬浮阴影
- 把后台页做成营销海报
- 把交易页做成空洞 landing page

## 2. 设计来源如何使用

`awesome-design-md` 在本项目中不作为“照抄模板库”，而作为“风格来源库”。

允许提取的内容：
- 页面层级组织方式
- 标题与副标题的节奏
- 留白策略
- 区块分隔方式
- 表单和表格的信息密度控制
- 按钮层级与 CTA 结构
- 动效是否克制、是否强调结果反馈
- 品牌感呈现方式

禁止直接照搬的内容：
- 与本项目业务无关的品牌文案
- 明显不适合装修平台的纯 SaaS 营销套路
- 不符合现有组件栈的完整组件实现
- 与当前端内现有品牌色/资源冲突的强品牌色系统

## 3. 页面类型与默认风格路由

### 3.1 营销 / 品牌 / 下载 / 介绍页

适用范围：
- `website/`
- `web/` 中首页、介绍页、落地页、下载页、引导页

默认风格：
- 来自 `awesome-design-md` 中偏品牌展示、强层级、强首屏控制、排版驱动型风格

执行要求：
- 首屏必须有明确主叙事，不允许堆很多等权重卡片
- 标题必须短、准、可复述
- 优先使用图片区、场景图区、流程图区建立信任
- CTA 数量控制在 1 个主 CTA + 1 个辅 CTA
- 区块数宁少勿碎
- 动效只服务于节奏，不服务于炫耀

默认视觉取向：
- 大标题
- 强对比留白
- 真实场景感
- 品牌感图像优先于装饰性图形

### 3.2 用户交易 / 订单 / 支付 / 项目流程页

适用范围：
- `web/` 用户端业务页
- `mini/` 用户端交易与流程页
- `mobile/` 用户端核心业务流

默认风格：
- 来自 `awesome-design-md` 中偏产品化、强调状态流与可信反馈的风格

执行要求：
- 先解释“当前处于哪一步”，再展示“下一步做什么”
- 金额、节点、风险提示、时间信息必须高辨识
- 单页主任务只能有一个，不允许多个并列主 CTA
- 重要状态必须可扫描，不依赖长文字理解
- 步骤条、节点卡、状态徽标、支付确认区必须层次明显

默认视觉取向：
- 结果导向
- 安全感
- 明确状态反馈
- 轻装饰、重流程

### 3.3 商家工作台 / 履约后台

适用范围：
- `merchant/`

默认风格：
- 来自 `awesome-design-md` 中偏高信息密度、结构稳定、状态驱动的产品后台风格

执行要求：
- 信息密度高，但必须有清晰的主次层级
- 筛选区、列表区、详情区、操作区要职责清楚
- 不堆砌卡片，优先使用面板、表格、时间线、状态区块
- 重点状态变化必须突出，不要被普通信息淹没
- 页面必须适合长时间使用，不能靠强视觉刺激维持注意力

默认视觉取向：
- 稳定
- 高效
- 清晰
- 可连续作业

### 3.4 Admin 治理 / 审核 / 风险控制页

适用范围：
- `admin/`

默认风格：
- 来自 `awesome-design-md` 中偏运营台、治理台、审查台的严肃产品风格

执行要求：
- 风险信息优先于装饰效果
- 审核信息要支持快速对比与判断
- 危险操作、驳回原因、异常状态必须显著
- 列表页优先考虑批量处理效率
- 详情页优先考虑审查链路完整性

默认视觉取向：
- 克制
- 审慎
- 强状态
- 弱装饰

### 3.5 表单 / 提交流 / 配置页

适用范围：
- 全仓库所有端

默认风格：
- 来自 `awesome-design-md` 中偏表单产品、强调输入节奏与错误反馈的风格

执行要求：
- 表单必须分段
- 每段只解决一类信息
- 标签、说明、错误提示、禁用态必须清楚
- 复杂输入要优先做结构简化，再做视觉美化
- 不允许用装饰掩盖字段复杂度

### 3.6 详情 / 时间线 / 状态页

适用范围：
- 订单详情
- 项目详情
- 方案详情
- 支付详情
- 验收详情

默认风格：
- 来自 `awesome-design-md` 中偏信息编排、时间结构、事件可追踪的产品风格

执行要求：
- 顶部先给结论
- 中部给关键状态与下一步动作
- 下部再给详细记录、说明、附件、时间线
- 页面内要有“概览层”和“明细层”

## 4. 多端设计一致性规则

同一业务对象在不同端必须共享以下语义一致性：
- 状态名称一致
- 风险等级一致
- 关键节点命名一致
- 金额、时间、支付、验收、退款等关键语义一致

允许不一致的部分：
- 导航结构
- 交互密度
- 列表呈现方式
- 布局方向

端差异原则：
- `mini/`：优先成交与快速操作，层级更短，反馈更直接
- `mobile/`：优先沉浸与连续流程体验
- `web/`：兼顾展示与交易说明
- `merchant/`：优先效率和履约视图
- `admin/`：优先治理与审查

## 5. 视觉系统硬规则

### 5.1 品牌与资源

品牌资源以仓库现有资产为准：
- `docs/branding/`
- `shared/design-tokens/`

约束：
- 优先复用已有 logo、图标、tokens
- 不私自发明第二套品牌主色
- 不绕开现有 tokens 随意定义全局色板

### 5.2 颜色

默认要求：
- 颜色先服务语义，再服务装饰
- 重要语义色必须稳定：成功、警告、错误、进行中、已完成、待处理
- 品牌色用于强调，不用于铺满全屏

禁止：
- 用多个高饱和主色争夺主导权
- 在后台页使用大面积花哨渐变
- 仅靠颜色表达关键状态，必须配合文字或图标

### 5.3 排版

默认要求：
- 标题层级必须明显
- 主标题短句优先
- 长说明要拆段，不堆成大块文本
- 中文界面优先考虑阅读节奏，不做英文式过窄排版

禁止：
- 各区块标题权重接近，看不出层级
- 说明文字比操作按钮更抢眼
- 用过多小字承载核心说明

### 5.4 间距与布局

默认要求：
- 页面先确定主结构，再填组件
- 同一页面的横向和纵向节奏要统一
- 留白必须服务分组与层级，不是单纯“空”

禁止：
- 一屏内出现太多边框盒子
- 卡片间距、标题间距、模块间距各自为政
- 不同区块使用完全不同的圆角、阴影、边框语言

### 5.5 组件取向

默认取向：
- 按钮要分主次
- 卡片要有用途，不是为了“看起来丰富”
- 表格优先可读与可操作
- 时间线、步骤条、状态块优先反映业务阶段

禁止：
- 为了“高级感”滥用玻璃、模糊、浮层
- 把所有信息都卡片化
- 把后台操作做成展示型橱窗

### 5.6 动效

默认要求：
- 只在进入、状态变化、反馈确认时使用轻量动效
- 动效要服务理解，不服务表演
- 后台与流程页动效更克制

禁止：
- 大面积持续运动背景
- 与业务无关的过多 hover / spring / parallax
- 影响表单、列表、审批效率的动画

## 6. 实现决策规则

当用户没有明确指定风格时，执行顺序如下：

1. 判断页面属于哪一类
2. 使用本文件中该类页面的默认风格
3. 参考 `awesome-design-md` 对应气质做风格抽取
4. 保持与仓库现有品牌资源和 tokens 一致
5. 优先完成可用性和业务清晰度

当用户有明确偏好时：
- “更高级”：提高排版控制、留白、图像主导权，不增加花哨装饰
- “更大胆”：允许更强对比、更明确主视觉，但不牺牲业务可读性
- “更稳重”：减少装饰、增强信息秩序和边界清晰度
- “更像品牌官网”：采用营销页规则，不把后台规则混进来
- “更像工作台”：采用后台规则，不做营销式夸张首屏

## 7. 冲突处理规则

如果出现以下冲突，按此优先级决策：

1. 业务正确性
2. 可读性与可操作性
3. 多端一致性
4. 现有品牌资产与 tokens
5. `awesome-design-md` 风格参考

这意味着：
- 如果某个参考风格很好看，但不适合装修交易场景，不采用
- 如果某种表现很高级，但会影响审核或履约效率，不采用
- 如果视觉想法与现有 tokens 冲突，优先落回仓库主设计系统

## 8. 默认禁令

未明确要求时，默认不要做这些：
- 通篇紫色科技风
- 纯模板化 shadcn 拼贴感
- 过量阴影和圆角堆叠
- 过多无意义 icon 装饰
- 空洞大 Hero + 三列卡片 + 一排 logo 的通用 AI 套路
- 营销页和后台页使用同一套视觉节奏

## 9. 维护方式

本文件应在以下情况下更新：
- 项目品牌方向明确变化
- 用户端、商家端、管理端的设计策略发生系统性变化
- 从 `awesome-design-md` 提炼出更适合本项目的新默认风格

不需要因为单个页面的小改动频繁更新。

如果只是一次性实验页或跳跃性设计：
- 可以临时覆盖本文件
- 但应在交付说明中明确“本次为局部 override，不代表全局默认设计方向已改变”
