# Draft: 多角色 App 端集成升级

## 需求背景
用户提出：将业主、设计师、工长、公司都集成到 App 端，需要对前后端数据库做升级开发。

## 当前系统分析

### 现有架构
- **用户角色**：业主(owner)、服务商(provider)、工人(worker)、管理员(admin)
- **服务商类型**：设计师(designer)、装修公司(company)、工长(foreman)
- **多身份系统**：已实现 `user_identities` 表，支持一账户多身份切换
- **前端平台**：
  - Admin Panel (React 18.3.1) - 管理后台
  - Mobile App (React Native 0.83) - 移动应用
  - WeChat Mini (Taro 3.x) - 微信小程序

### 现有数据模型
- `users` - 用户基础表，`user_type` 字段区分角色
- `user_identities` - 身份表，支持多身份
- `providers` - 服务商表，`provider_type` 区分设计师/公司/工长
- `workers` - 工人表

### 现有身份切换功能
- 后端 API：`/api/v1/identities/*`
- 前端组件：`IdentitySwitcher` (Admin/Mobile/Mini 都有)
- Token 结构：包含 `active_role` 和 `identity_ref_id`

## 已确认需求

### 1. 集成模式
- **选择**：单 App 多角色切换
- **说明**：一个 App，用户切换身份后看到不同的界面和功能

### 2. 新增角色功能
- [x] 设计师端：接单、方案管理、作品展示、收入管理
- [x] 工长端：接单、工人调度、进度汇报、收入管理
- [x] 装修公司端：项目管理、团队管理、财务管理、客户管理
- [ ] 工人端：（未选择，暂不开发）

### 3. 数据库设计
- **用户期望**：需要我推荐表结构设计
- **现有基础**：`user_identities` 多身份系统已实现

### 4. UI/导航设计
- **底部导航**：完全独立的 Tab 结构（切换角色后 Tab 完全变化）
- **业主端 Tab**：首页/灵感/进度/消息/我的（现有）
- **设计师端 Tab**：首页/订单/作品/收入/我的（待设计）
- **工长端 Tab**：首页/订单/进度/收入/我的（待设计）
- **公司端 Tab**：首页/项目/团队/财务/我的（待设计）

### 5. 订单流程
- **选择**：完整订单流程
- **流程**：业主预约 → 服务商接单 → 确认方案 → 签约 → 施工 → 验收

### 6. 开发策略
- **选择**：MVP 先行
- **策略**：先完成所有角色的核心功能，再迭代完善

### 7. MVP 功能范围
- [x] 接单管理：查看新预约、接单/拒单、查看预约详情
- [x] 收入管理：查看收入统计、收入明细、申请提现
- [x] 项目进度：查看进行中项目、更新进度、上传日志
- [x] 作品管理：管理作品集、上传新作品（设计师专属）
- [x] 团队管理：查看团队成员、分配任务（公司/工长专属）
- [x] 消息中心：与业主沟通、查看消息列表

### 9. 测试策略
- **选择**：TDD 测试驱动
- **基础设施**：项目已有 Jest 测试框架
- **流程**：先写测试 → 实现功能 → 重构

### 10. Metis 分析后的决策
- **订单表设计**：复用 bookings 表，新增 status 字段支持完整流程
- **团队管理范围**：简化版（仅查看成员列表 + 指派到项目）
- **消息中心**：简单通知列表（系统消息 + 状态变更通知）
- **提现流程**：人工打款（做申请记录，实际打款由人工处理）

### 11. MVP 明确排除项
- ❌ 实时聊天（WebSocket）
- ❌ 任务看板/甘特图
- ❌ 复杂的数据分析图表
- ❌ 发票管理
- ❌ 评价系统
- ❌ 地图导航
- ❌ 视频通话
- ❌ 自动打款（支付 API 对接）
