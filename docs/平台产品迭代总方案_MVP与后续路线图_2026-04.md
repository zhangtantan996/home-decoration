# 平台产品迭代总方案（MVP 与后续路线图，2026-04）

> 文档目的：统一产品、研发、运营对“先做什么、哪些先不上、各端分别承担什么角色、MVP 如何收口、后续如何迭代”的判断口径。  
> 真相基线：`产品需求文档(PRD).md`、`BUSINESS_FLOW.md`、`平台阶段判断与主商业模型_2026-04.md`、`小程序优先增长与成交操作系统_2026-04.md`、`试运营整体开发计划_2026-04.md`。  
> 适用阶段：当前试运营收口 + 下一阶段产品迭代规划。

---

## 1. 核心结论

当前平台不是“普通装修信息展示小程序”，而是正在收敛中的 **家装交易平台**。  
当前最重要的不是横向铺更多功能，而是把以下四件事做深：

1. **成交做深**：让用户从预约稳定走到设计确认，再走到工长确认
2. **履约做稳**：让项目执行、验收、退款、争议能稳定在平台内收口
3. **平台赚钱**：先把设计成交佣金、施工成交佣金、托管/治理服务费跑顺
4. **增长复利**：让完工结果沉淀为案例、评价、履约标签和转介绍资产

因此，未来产品路线固定为四层结构：

- **第 1 层：交易平台**  
  设计成交 + 施工成交 + 履约收口
- **第 2 层：供给平台**  
  让设计师、工长、装修公司长期依赖平台接单与履约
- **第 3 层：金融/资金平台**  
  托管、节点付款、分期、保险、账期
- **第 4 层：数据平台**  
  报价数据库、履约标签、区域价格指数、经营数据

但当前阶段只做 **第 1 层打透 + 第 2 层基础能力建立**，第 3、4 层放到后续。

## 2. 端面战略与角色分工

## 2.1 当前正式端面战略

- **小程序 = 主成交端**
  - 主获客
  - 主预约
  - 主设计成交
  - 主施工桥接前台
  - 主支付入口

- **Merchant Web = 主履约端**
  - 线索承接
  - 量房/预算/方案提交
  - 工长报价
  - 项目执行
  - 收入与提现

- **Admin Web = 主治理端**
  - 商家审核
  - 项目/订单监控
  - 退款/争议/关闭
  - 提现审核与打款确认
  - 风险治理与最小经营看板

- **Web/H5 = 辅助承接端**
  - 活动落地页
  - 分享回流页
  - 支付结果页
  - 轻量浏览页

## 2.2 当前不作为主端的端面

- **App**：当前不上主线，不进入 MVP 或近期重点
- **独立监理端**：当前不单做，先以 Admin / Merchant 流程能力承接
- **主材商城独立前台**：当前不进主交易主线

## 2.3 监理能力策略

监理不是当前阶段的独立业务面，而是履约治理能力的一部分。

当前处理原则：

- **不单独开发监理端**
- 监理相关动作先拆到两个面里：
  - **Admin**：开工协调、异常协调、验收/争议治理
  - **Merchant**：提交进度、补材料、响应整改
- 等平台项目量、监理动作频次、独立岗位需求真正成立后，再评估单独监理工作台

## 3. 当前主商业模型与产品目标

## 3.1 当前唯一主商业模型

当前主商业模型必须固定为：

- **主模型**：设计成交佣金 + 施工成交佣金
- **辅模型**：托管 / 验收 / 风控 / 治理服务费

这是当前最贴合既有主链、最容易形成真实收入闭环的路径。

## 3.2 当前阶段产品目标

当前阶段只需要证明 4 件事：

1. 用户愿意在平台内提交真实需求并进入预约
2. 用户愿意完成设计确认，并继续推进到工长确认
3. 商家愿意在平台规则下持续响应、提案、报价、履约
4. 完工、退款、投诉、争议等异常可以被平台稳定收口

## 3.3 当前暂不作为主模型的方向

以下方向保留，但当前不能抢主线资源：

- 广告位 / 流量位收费
- 主材商城主线
- 重 SaaS 收费
- 会员 / 积分体系
- 金融分期 / 贷款
- 材料供应链利润
- 数据服务商品化

原因：这些能力都建立在 **成交闭环和履约闭环已经稳定** 的前提上。

## 4. MVP（试运营版）必须做什么

## 4.1 MVP 目标

MVP 的目标不是“页面齐全”，而是让平台最核心的交易链跑通：

**预约 → 设计确认 → 工长确认 → 项目创建 → 开工 → 阶段验收 / 异常收口**

## 4.2 用户端（小程序）MVP

### A. 获客与预约

必须具备：

- 首页价值表达与强引导 CTA
- 服务商列表 / 详情
- 预约创建 / 预约列表 / 预约详情
- 报价引流页 / 预约前预估抓手
- 登录 / 留资 / 表单收敛

目标是让用户从“看看”变成“提交真实需求”。

### B. 设计成交

必须具备：

- 量房 / 预算确认结果查看
- 设计方案详情查看
- 设计方案确认 / 拒绝
- 方案版本历史与重提结果查看
- 设计确认后的下一步指引

目标是让用户从“咨询”走到“确认设计”。

### C. 施工桥接

必须具备：

- 报价基线说明
- 工长 / 装修公司对比
- 施工报价确认页
- 施工责任边界 / 风险提示 / 工期说明
- 工长确认后的项目承接

目标是让用户从“设计确认”顺利走到“工长确认”。

### D. 项目与售后最小可用

必须具备：

- 订单详情
- 项目详情
- 进度摘要
- 阶段验收入口
- 退款入口
- 支付结果页 / 退款结果承接页

### E. 设置与账号安全最小可用

必须具备：

- 个人资料维护
- 账号安全：修改手机号、修改登录密码、注销账号
- 实名认证：支付和资金动作前置门禁，一期身份证二要素，二期再评估人脸/活体
- 核心通知开关：系统通知、项目提醒、订单与支付提醒
- 隐私设置轻量入口：用户协议、隐私政策
- 意见反馈与客服入口

设置功能按“小程序上线前置”判断，不按当前完成度判断。隐私协议、账号注销、修改手机号、实名认证支付门禁、核心通知开关、意见反馈和客服入口属于上线前置；登录设备、字体/语言、定位/推荐/手机号可见等能力按业务必要性和真实消费链路决定是否后置。详见 [Mini 设置功能 MVP 上线范围说明](./mini/settings-mvp-scope.md)。

## 4.3 商家端（Merchant Web）MVP

### A. 线索到成交

- 新预约 / 新需求承接
- 量房与预算记录
- 设计方案提交
- 方案版本重提
- 驳回原因回看
- 工长施工报价提交

### B. 履约执行

- 项目列表 / 项目详情
- 开工动作
- 阶段提交
- 进度材料补充
- 完工申请

### C. 收入与通知

- 收入查看
- 提现申请
- 提现状态查看
- 通知中心

## 4.4 平台端（Admin Web）MVP

### A. 审核与监控

- 商家审核
- 订单 / 项目 / 阶段监控
- 开工协调与状态监控

### B. 异常治理

- 退款审核
- 争议处理
- 项目关闭
- 提现审核与打款确认
- 风险预警最小页

### C. 经营可见性

- 最小经营看板
  - 预约数
  - 设计确认数
  - 工长确认数
  - 项目完工数
  - 退款 / 争议积压

## 4.5 Web/H5 MVP

只做以下辅助能力：

- 活动落地页
- 分享回流页
- 支付结果页
- 轻量服务商 / 案例浏览页

明确不承担主成交职责。

## 4.6 MVP 明确先不上

MVP 阶段以下能力不进入主线：

- 开放式抢单大厅
- 工地直播
- 节点资金托管完整产品化
- 售后工单系统完整版
- 小区口碑系统
- 老带新返现体系
- 裂变分销体系
- 主材商城升级
- 独立监理端
- App
- 金融分期
- 商家 CRM SaaS 收费版

## 5. MVP 之后的后续迭代方向

## 5.1 迭代一：成交率提升

目标：把“设计确认 → 工长确认”的掉点做深。

### 核心能力

- 智能报价器升级
  - 半包 / 全包报价区间
  - 工期预估
  - 风险提醒
- 多主体对比增强
  - 价格
  - 工期
  - 评分
  - 履约标签
- 限时成交机制
  - 报价有效期
  - 阶段优惠
  - 催单抓手
- 在线签约 + 定金支付
- 客服 IM / 咨询链路收口

### 端面重点

- 小程序优先
- Merchant 补报价效率工具
- Admin 补签约与订单状态监控

## 5.2 迭代二：履约与资金确定性提升

目标：让平台从“能成交”升级为“能控履约、敢控风险”。

### 核心能力

- 项目执行流程标准化
- 节点提交 / 阶段验收体系
- 支付计划与结算强绑定
- 退款 / 争议 / 关闭深收口
- 提现链路规范化

### 端面重点

- 小程序：进度、验收、退款体验
- Merchant：阶段提交、收入、提现
- Admin：退款、争议、提现、风控

## 5.3 迭代三：增长复利建设

目标：让一次履约结果变成下一次获客与成交资产。

### 核心能力

- 评价系统
- 完工案例沉淀
- 履约标签系统
- 转介绍入口
- 轻量装修工具页（预算计算器、面积估价器等）

### 端面重点

- 小程序：案例、评价、转介绍
- Merchant：标签与案例结果可见
- Admin：案例审核、标签治理、增长数据看板
- Web/H5：承接分享与外部流量

## 5.4 迭代四：利润升级

目标：在主交易跑顺后，扩展高利润业务能力。

### 核心方向

- 金融分期
- 托管 / 保险 / 风控服务
- 材料供应链
- 商家会员 / 流量位 / 轻 SaaS
- 数据服务

### 进入条件

只有在以下前提同时成立后才进入：

- 小程序成交链稳定
- Merchant 履约链稳定
- Admin 治理链稳定
- 平台已有稳定 GMV 和订单量

## 7. 迭代优先级判断规则

后续每个新需求都先判断 5 个问题：

1. 是否直接服务设计成交或施工成交
2. 是否提升成交前抓手、桥接能力或履约稳定性
3. 是否会稀释“小程序主成交、Merchant 主履约、Admin 主治理”的边界
4. 是否会把主材商城、广告、社区、会员、金融等旁路线抬成主线
5. 是否能映射到现有 PRD / BUSINESS_FLOW，而不是引入另一套业务真相

如果前 2 条回答不强，或者后 3 条明显偏离，就不应进入当前主线。

## 8. 建议的阶段里程碑

## P0：MVP 收口（当前）

成功标志：

- 小程序完成预约 → 设计确认 → 工长确认 → 项目创建
- Merchant 完成提案 → 报价 → 项目执行 → 提现申请
- Admin 完成审核 → 退款/争议/关闭 → 提现审核

## P1：成交率强化

成功标志：

- 报价器成为真实成交抓手
- 设计确认到工长确认转化率提升
- 定金支付 / 签约开始锁单

## P2：履约可控

成功标志：

- 项目执行、验收、退款、争议都能平台内稳定收口
- 平台具备“可信裁判”能力

## P3：增长复利

成功标志：

- 案例、评价、标签开始持续带来新成交
- 完工项目不再一次性沉没

## P4：利润扩展

成功标志：

- 收入不再只依赖撮合佣金
- 平台开始形成资金 / 供应链 / SaaS / 数据型利润

## 9. 结语

当前平台最关键的一句话是：

> 不是在继续堆页面，而是在把家装行业里最难在线化的“设计成交、施工成交、履约收口、异常治理”做成平台基础设施。

因此，产品路线必须始终围绕：

- **先成交**
- **再履约**
- **再增长**
- **最后做利润扩展**

任何偏离这条主线的功能，都不应抢当前主线资源。
