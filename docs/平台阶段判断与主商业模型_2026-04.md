# 平台阶段判断与主商业模型（2026-04）

> 依赖真相源： [产品需求文档(PRD).md](./产品需求文档(PRD).md)、[BUSINESS_FLOW.md](./BUSINESS_FLOW.md)

## 1. 结论

当前项目处于：**平台型产品中前期**。

核心矛盾不是“功能太少”，而是**经营闭环还不够深**。具体表现为：

- 成交前抓手还不够强
- 设计确认到工长确认之间桥接不足
- 商业模型还不够收敛
- 多端复杂度已经开始压主线打磨

因此，当前最重要的不是继续横向铺功能，而是把平台收敛成一个能稳定成交、稳定履约、稳定运营的系统。

## 2. 对外部评论的判断

### 2.1 判断正确的部分

以下判断方向是对的：

- 项目已经不是 idea 或演示型 MVP，而是进入复杂系统管理阶段
- 主问题不是页面数量，而是商业、供给、运营闭环深度
- 多端并行会持续放大复杂度，如果不收敛会稀释主线
- 平台真正的价值不在商家展示，而在信任与履约控制能力

### 2.2 需要修正的部分

以下说法需要修正：

- 项目不是“只有骨架”。现有 [产品需求文档(PRD).md](./产品需求文档(PRD).md) 和 [BUSINESS_FLOW.md](./BUSINESS_FLOW.md) 已把双成交点、项目创建条件、异常收口定义得比较清楚。
- 当前也不是“重新写一份大 PRD”能解决问题。真正缺的是一套商业/运营判断文档，把产品、运营、技术的建议统一到同一口径。
- 主材商线不是当前平台价值核心，不应因为“模块已经存在”就误判为要同时推进的主线。

### 2.3 当前最该优先处理的

当前优先级应固定为：

1. 强化小程序主成交端
2. 补设计确认到工长确认的桥接抓手
3. 建立商家供给分层与治理规则
4. 统一经营指标和驾驶舱口径
5. 再决定是否扩更多端面或更多商业化能力

## 3. 当前阶段定义

### 3.1 阶段标签

当前阶段不是“流量平台”，也不是“成熟履约平台”，而是：

**正在验证并收敛可信成交与可控履约的交易平台。**

### 3.2 阶段目标

当前阶段只需要证明 4 件事：

- 用户愿意在平台内提交真实需求并进入预约
- 用户愿意完成设计确认，并继续推进到工长确认
- 商家愿意在平台规则下持续响应、提案、履约
- 完工、退款、投诉等异常能被平台稳定收口

## 4. 主商业模型

### 4.1 当前主模型

当前唯一主商业模型应定义为：

- **主模型**：设计成交 + 施工成交撮合佣金
- **辅模型**：托管 / 验收 / 风控服务费

这与现有双成交点主链完全一致：

- 设计确认对应设计成交价值
- 工长确认对应施工成交价值
- 项目执行、验收、退款、仲裁构成平台服务价值

### 4.2 为什么这样定

因为这是当前最贴合 PRD 主链、最容易形成真实收入闭环的路径：

- 不依赖大规模广告流量
- 不要求主材商城先跑通
- 不要求先做重 SaaS 再收费
- 能直接绑定当前最重要的交易与履约节点

### 4.3 当前不作为主模型的方向

以下方向可以保留为未来备选，但当前不能抢主线资源：

- 广告位 / 曝光位收费
- 主材商城主线
- 重 SaaS 收费
- 会员积分体系

原因很简单：这些方向都不能直接解决当前最核心的问题，即**如何稳定成交、稳定履约、稳定形成平台信任**。

## 5. 平台护城河定义

当前平台护城河不能定义为“商家展示更多”或“信息更多”，而必须定义为：

- **可信成交**：用户敢在平台内完成设计与施工决策
- **可控履约**：项目执行、验收、放款、退款有明确规则与留痕
- **可追责异常**：争议、投诉、退款、审计能被平台有效收口

如果做不到这三点，平台只是信息聚合页；做到了，平台才具备交易基础设施价值。

## 6. 端面战略判断

当前正式战略基线固定为：

- **小程序 = 主成交端**
- **Merchant Web = 主履约端**
- **Admin Web = 主治理端**
- **Web/H5 = 承接页 / 支付结果页 / 辅助浏览端**

这意味着：

- 小程序优先承担用户增长、预约、设计确认、施工桥接前半段
- Merchant Web 优先承担商家资料、接单、提案、履约协作
- Admin Web 优先承担审核、争议、资金、治理动作
- Web/H5 不再承担当前阶段的主成交战略角色

## 7. 功能去留判断规则

以后判断一个功能要不要做，先问 5 个问题：

1. 是否直接服务设计成交或施工成交
2. 是否能提升成交前抓手、桥接能力或履约稳定性
3. 是否会稀释小程序主成交端或 Merchant/Admin 主角色边界
4. 是否会把主材商、广告、内容、积分之类的旁路能力抬成主线
5. 是否能被现有 PRD / BUSINESS_FLOW 映射，而不是引入另一条业务真相

只要前 2 条答不出来，或后 3 条明显偏离，就不应进入当前主线。

## 8. 本文档以后怎么用

后续遇到下面这些问题，默认优先引用本文档：

- 要不要做某个新功能
- 这个功能是否服务当前商业模型
- 主材商是否该升到主线
- 某个端面是否还应该继续加码
- 某个运营动作是不是当前阶段的重点

如果问题继续深入到交易状态、角色权限、项目创建条件，再回到 [产品需求文档(PRD).md](./产品需求文档(PRD).md) 和 [BUSINESS_FLOW.md](./BUSINESS_FLOW.md)。
