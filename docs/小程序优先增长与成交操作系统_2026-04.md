# 小程序优先增长与成交操作系统（2026-04）

> 依赖真相源： [产品需求文档(PRD).md](./产品需求文档(PRD).md)、[BUSINESS_FLOW.md](./BUSINESS_FLOW.md)

## 1. 结论

当前增长与成交策略必须围绕一个前提展开：**小程序是主成交端，不是辅助展示端。**

因此，小程序不是简单承接信息浏览，而是要承担 4 件事：

- 把用户拉进来并完成留资/登录
- 把用户推到预约和设计成交
- 把用户从设计确认桥接到工长确认
- 把完工结果沉淀为平台可复用资产

## 2. 四条核心漏斗

### 2.1 拉新漏斗

路径：**进入 -> 留资/登录 -> 预约**

当前目标：让用户从“只是看看”进入“愿意留下真实需求”。

关键掉点通常出现在：

- 首页价值表达不够强
- 详情页只有展示，没有信任抓手
- 预约动作成本太高
- 没有把城市、风格、预算、案例说清楚

当前抓手应优先补：

- 首屏价值表达
- 服务商/案例/项目结果背书
- 预约页表单收敛
- 预约前的信任解释

### 2.2 设计成交漏斗

路径：**预约 -> 量房/预算 -> 方案提交 -> 设计确认**

当前目标：让用户从“咨询”走到“确认设计方案”。

关键掉点通常出现在：

- 量房和预算价值解释不足
- 设计方案对比不清楚
- 价格、交付物、修改边界不透明
- 用户不知道确认设计后会发生什么

当前抓手应优先补：

- 方案价值说明
- 预算边界解释
- 设计师资质与案例背书
- 设计确认后的下一步指引

### 2.3 施工桥接漏斗

路径：**设计确认 -> 工长对比/施工方案 -> 工长确认 -> 项目创建**

当前目标：把“设计成交”顺利转成“施工成交”。

这是当前最关键、也是最容易断的漏斗。

### 2.4 完工增长漏斗

路径：**完工 -> 评价 -> 案例沉淀 -> 转介绍**

当前目标：把一次履约结果沉淀为平台长期资产。

如果这一段不做，平台每次都只能重新获客，无法建立持续复利。

## 3. 设计确认 -> 工长确认的桥接问题

### 3.1 用户为什么会在这里掉

用户不是因为“流程不存在”才掉，而是因为这里缺少成交桥接抓手。常见原因有：

- 用户已经认可设计，但不确定谁来施工更靠谱
- 施工报价、工期、责任边界不够可解释
- 不知道设计图能否顺利落地
- 害怕后续增项、扯皮、延期、验收不过
- 看不到足够真实的施工案例与评价支撑

### 3.2 平台缺的不是流程，而是桥接能力

平台在这个阶段必须提供的不是更多按钮，而是更强的桥接抓手：

- 工长筛选与对比
- 工期、预算、责任边界可解释
- 风险提示
- 阶段验收承诺
- 真实案例与评价背书

### 3.3 平台应该怎么提供桥接能力

当前应优先把以下内容做成标准化表达：

- 同设计方案下，不同工长的报价、工期、能力标签对比
- 施工范围、增项边界、谁负责什么的明确说明
- 阶段验收、整改、退款、争议处理规则说明
- 已完工项目与真实评价的背书展示

只有桥接能力足够强，设计确认后的用户才会继续推进到工长确认。

## 4. 小程序主成交端产品职责

### 4.1 首页承担什么

首页不应追求内容堆叠，而应承担 3 个职责：

- 讲清平台价值：为什么要在这里做装修决策
- 引导进入关键漏斗：预约、找设计师、看真实案例
- 快速建立信任：展示规则、结果、标签、口碑

### 4.2 详情页承担什么

详情页不只是展示资料，而应承担成交前信任转换：

- 看懂服务对象、风格、预算区间、交付能力
- 看真实案例、履约标签、评价结果
- 看平台规则下的责任与保障
- 明确下一步 CTA：预约、咨询、提交需求

### 4.3 预约页承担什么

预约页的任务不是收集尽可能多的信息，而是完成最小必要留资：

- 让用户愿意提交真实需求
- 减少表单阻力
- 解释提交后会发生什么
- 给用户可预期的下一步

### 4.4 支付前承担什么

支付前页面要解决的是“为什么现在值得确认”，而不是只催支付。必须把以下内容说清：

- 当前支付对应什么业务承诺
- 支付后平台和商家各自要做什么
- 如果后续不满意，退款/争议怎么走
- 下一步如何继续推进到成交或履约

### 4.5 哪些内容暂不应占主入口

当前不应在主入口抢资源的内容包括：

- 与成交无关的泛内容流
- 纯浏览型灵感堆叠
- 主材商城主入口
- 不直接服务预约或成交的复杂社区功能

## 5. 完工后的资产沉淀

完工不是结束，而是平台增长资产的起点。当前必须沉淀 4 类资产：

- 评价
- 完工案例
- 商家履约标签
- 转介绍入口

### 5.1 评价

评价不只是打星，而是为后续成交提供可信背书，重点应看：

- 设计满意度
- 施工履约度
- 工期兑现情况
- 沟通与整改体验

### 5.2 完工案例

完工案例是后续获客素材，应沉淀：

- 项目类型
- 风格与面积
- 预算区间
- 前后对比与关键节点
- 结果图片与用户评价

### 5.3 商家履约标签

案例和评价最终要收敛成可复用标签，例如：

- 响应快
- 预算稳定
- 工期稳定
- 验收通过率高
- 纠纷少

### 5.4 转介绍入口

完工后应引导用户做二次传播，而不是让项目就此沉没。重点是：

- 分享案例
- 推荐给熟人
- 回流查看项目成果
- 为下一位用户提供真实背书

## 6. 当前端面分工

在增长与成交层面，端面职责应固定为：

- **小程序**：主获客、主预约、主设计成交、施工桥接前台
- **Merchant Web**：提案、履约、材料上传、订单处理、项目协作
- **Admin Web**：审核、治理、争议、资金与异常收口
- **Web/H5**：活动承接、分享回流、支付结果、辅助浏览
