### 1. 合同签署缺少防重放攻击保护

**文件**: `server/internal/service/contract_service.go`  
**位置**: 第 248-289 行（SignContractByUser）、第 292-333 行（SignContractByProvider）

**问题描述**:
合同签署接口使用了悲观锁（`clause.Locking{Strength: "UPDATE"}`），但在检查签署状态后到更新数据库之间存在时间窗口，可能导致并发签署问题。

**风险**:
- 在高并发场景下，可能出现重复签署
- 签名数据（signatureData）未进行任何验证，攻击者可以提交任意字符串

**修复建议**:
```go
// 在 SignContractByUser 中添加签名验证
func (s *ContractService) SignContractByUser(userID, contractID uint64, input *SignContractInput) (*model.Contract, error) {
    // 1. 验证签名数据格式
    if strings.TrimSpace(input.SignatureData) == "" {
        return nil, errors.New("签名数据不能为空")
    }
    
    // 2. 验证签名数据长度（防止超大数据攻击）
    if len(input.SignatureData) > 10000 {
        return nil, errors.New("签名数据过大")
    }
    
    // 3. 在事务中使用悲观锁并立即检查状态
    var contract model.Contract
    err := repository.DB.Transaction(func(tx *gorm.DB) error {
        if err := tx.Clauses(clause.Locking{Strength: "UPDATE"}).First(&contract, contractID).Error; err != nil {
            return errors.New("合同不存在")
        }
        
        // 立即检查是否已签署（在锁内）
        if contract.UserSignedAt != nil {
            return errors.New("您已签署该合同")
        }
        
        // ... 其他验证和更新逻辑
        return nil
    })
    
    return &contract, err
}
```

### 2. 支付金额计算存在精度丢失风险

**文件**: `server/internal/service/wechat_pay_gateway_service.go`  
**位置**: 第 324-330 行

**问题描述**:
金额转换函数 `amountYuanToFen` 使用浮点数运算，可能导致精度丢失。

```go
func amountYuanToFen(amount float64) int64 {
    return int64(normalizeAmount(amount)*100 + 0.5)  // ❌ 浮点数精度问题
}
```

**风险**:
- 在大额支付时可能出现分币级别的误差
- 累积误差可能导致对账不平

**修复建议**:
```go
// 使用 decimal 库进行精确计算
import "github.com/shopspring/decimal"

func amountYuanToFen(amount float64) int64 {
    d := decimal.NewFromFloat(amount)
    fen := d.Mul(decimal.NewFromInt(100))
    return fen.IntPart()
}

// 或者在数据库层面使用整数存储（分为单位）
type PaymentOrder struct {
    AmountCent int64 `json:"amountCent"` // 以分为单位存储
}
```

### 3. SQL 注入风险（低风险但需修复）

**文件**: `server/internal/service/milestone_payment_service.go`  
**位置**: 第 194 行

**问题描述**:
虽然使用了 GORM，但在生成订单号时直接拼接了用户输入的 ID，存在潜在风险。

```go
orderID := fmt.Sprintf("MP%d%d%d", input.ProjectID, input.MilestoneID, time.Now().Unix())
```

**风险**:
- 如果 ProjectID 或 MilestoneID 来自不可信来源，可能导致订单号冲突
- 订单号可预测，可能被恶意利用

**修复建议**:
```go
// 使用 UUID 或加密哈希生成订单号
import "github.com/google/uuid"

orderID := fmt.Sprintf("MP-%s", uuid.New().String())
// 或使用时间戳+随机数
orderID := fmt.Sprintf("MP%d%04d", time.Now().Unix(), rand.Intn(10000))
```

**风险**:
- 在网络重试或并发场景下，可能重复创建出款单
- 可能导致资金损失

**修复建议**:
```go
func (s *PayoutRoutingService) ExecutePayoutTx(tx *gorm.DB, settlementOrderID uint64) (*model.PayoutOrder, error) {
    // 1. 先检查是否已存在出款单（幂等性保护）

        model.PayoutBizTypeSettlementOrder, 
        settlementOrderID,
        []string{model.PayoutStatusFailed}).
        First(&existingPayout).Error
    
    if err == nil {
        // 已存在出款单，直接返回
        return &existingPayout, nil
    }
    
    if !errors.Is(err, gorm.ErrRecordNotFound) {
        return nil, err
    }
    
    // 2. 继续创建新出款单...
}
```

### 6. 定时任务缺少错误恢复机制

**文件**: `server/internal/cron/payment_reconciliation_cron.go`  
**位置**: 第 158-188 行（StartPaymentReconciliationCron）

**问题描述**:
定时任务如果 panic 会导致整个 goroutine 退出，后续任务不再执行。

**风险**:
- 单次任务失败会导致后续所有任务停止
- 缺少监控和告警机制

**修复建议**:
```go
func StartPaymentReconciliationCron() {
    job := NewPaymentReconciliationJob()
    
    go func() {
        defer func() {
            if r := recover(); err != nil {
                log.Printf("[Cron] PANIC recovered: %v", r)
                // 重启定时任务
                time.Sleep(1 * time.Minute)
                StartPaymentReconciliationCron()
            }
        }()
        
        ticker := time.NewTicker(24 * time.Hour)
        defer ticker.Stop()
        
        for {
            select {
            case <-ticker.C:
                // 使用 recover 保护每次执行
                func() {
                    defer func() {
                        if r := recover(); r != nil {
                            log.Printf("[Cron] Job execution panic: %v", r)
                        }
                    }()
                    job.Run()
                }()
            }
        }
    }()
}
```

- 事务回滚逻辑不清晰

**修复建议**:

```go
// 方案1：移除外层事务包装，直接使用 ExecutePayoutTx
func (s *PayoutRoutingService) ExecutePayout(settlementOrderID uint64) (*model.PayoutOrder, error) {
    var result *model.PayoutOrder
    err := repository.DB.Transaction(func(tx *gorm.DB) error {
        payout, err := s.ExecutePayoutTx(tx, settlementOrderID)
        result = payout
        return err
    })
    return result, err
}

// 方案2：将 ExecutePayoutTx 改为不依赖外部事务

    })
```

### 8. 前端缺少错误边界处理

**文件**: `merchant/src/pages/merchant/MerchantQuoteTasks.tsx`  
**位置**: 第 59-74 行（handleSubmit）

**问题描述**:
表单提交失败后，Modal 不会自动关闭，用户可能重复提交。

**风险**:
- 用户体验差
- 可能导致重复提交

**修复建议**:
```tsx
const handleSubmit = async () => {
  if (!selectedTask) return;

  try {
    const values = await form.validateFields();
    setSubmitting(true);
    await merchantQuotePKApi.submitQuote(selectedTask.id, values);
    message.success('报价提交成功');
    setSubmitModalVisible(false);
    form.resetFields(); // 重置表单
    loadTasks();
  } catch (err: any) {
    // 区分不同类型的错误
    if (err.response?.status === 409) {
      message.error('该任务已被其他工长抢单');
      setSubmitModalVisible(false);
      loadTasks();
    } else if (err.response?.status === 400) {
      message.error(err?.message || '参数错误，请检查输入');
    } else {
      message.error(err?.message || '提交失败，请稍后重试');
    }
  } finally {
    setSubmitting(false);
  }
};
```
