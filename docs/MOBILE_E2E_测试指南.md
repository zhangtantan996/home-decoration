## 测试场景

### 场景 2.1: 登录和连接

**目标**: 验证 Tinode 初始化和 WebSocket 连接

**步骤**:
1. 在模拟器中打开 App
2. 进入登录界面（如果还没有）
3. 输入手机号: `13800138000`
4. 输入验证码: `123456`
5. 点击"登录"按钮
6. **观察 Metro 终端的控制台日志**

**预期的控制台日志**:
```
[Tinode] 初始化中...
[Tinode] WebSocket 已连接
[Tinode] 登录成功: {ctrl: {...}}
[Tinode] 初始化成功
```

**预期的界面行为**:
- 登录界面消失
- App 跳转到首页/消息界面
- 没有错误提示或弹窗

**验收清单**:
- [ ] 登录成功，无错误
- [ ] 控制台显示所有 4 条预期日志
- [ ] TinodeService 初始化成功
- [ ] 控制台无错误日志
- [ ] App 跳转到主界面

**代码参考**: 
- `mobile/src/services/TinodeService.ts:91-186` (init 方法)
- `mobile/src/screens/LoginScreen.tsx` (登录流程)

**故障排除**:
- **如果没有显示"WebSocket 已连接"**: 检查 Tinode 容器状态 `docker ps | grep tinode`
- **如果"登录成功"失败**: 验证登录 API 响应中是否包含 tinodeToken
- **如果连接超时**: 检查 `mobile/.env` 中的 `TINODE_SERVER_URL`（可选覆盖）以及 `mobile/src/config/tinode.ts` / `mobile/src/services/TinodeService.ts` 的 host 选择逻辑

### 场景 2.2: 查看会话列表

**目标**: 验证会话列表加载和显示

**步骤**:
1. 登录成功后（场景 2.1）
2. 进入"消息"标签页
3. 等待会话列表加载
4. **观察界面**

**预期的界面元素**:
- 连接状态指示器（右上角）: 绿色 WiFi 图标 = 已连接
- 会话卡片显示:
  - 头像图片（圆形，48x48）
  - 名称（粗体，15px）
  - 角色标签（例如"服务商"）
  - 最后一条消息预览
  - 时间戳（格式: "HH:MM"、"昨天"、"周X" 或 "M/D"）
  - 未读数角标（如果 > 0）
- 会话按最近活跃时间排序

**预期的控制台日志**:
```
[Tinode] 会话列表: <数量>
```

**验收清单**:
- [ ] 会话列表加载成功
- [ ] 每个会话显示完整信息
- [ ] 头像正确渲染
- [ ] 最后消息预览显示正确
- [ ] 时间戳格式正确
- [ ] 未读数显示正确
- [ ] 连接状态显示绿色 WiFi 图标

**代码参考**:
- `mobile/src/services/TinodeService.ts:248-275` (getConversationList)
- `mobile/src/screens/MessageScreen.tsx:127-253` (loadConversations)

**故障排除**:
- **如果列表为空**: 检查数据库中是否有现有会话
- **如果显示"IM 未登录"**: Tinode 连接失败，检查日志
- **如果头像不加载**: 检查网络连接和图片 URL

### 场景 2.3: 发送文本消息

**目标**: 验证文本消息发送和状态更新

**步骤**:

4. 在输入框输入文本: "测试消息1"
5. 点击发送按钮（纸飞机图标）
6. **观察消息显示和状态**

1. 消息立即显示在聊天界面（乐观更新）
2. 消息显示在右侧（isMe=true）
3. 消息显示时间戳
4. 消息状态变化: pending → sent（通过 seq 号分配）
5. 发送后输入框清空

**预期的控制台日志**:
```
[Tinode] 消息已发送: 测试消息1
```

**验收清单**:
- [ ] 消息发送成功
- [ ] 消息立即显示在界面
- [ ] 消息显示在右侧（发送方）
- [ ] 时间戳显示正确
- [ ] 状态从 pending 变为 sent
- [ ] 输入框清空
- [ ] 无错误日志

**代码参考**:
- `mobile/src/services/TinodeService.ts:404-415` (sendTextMessage)
- `mobile/src/screens/ChatRoomScreen.tsx` (发送消息处理)

**故障排除**:
- **如果消息不发送**: 检查 topic 订阅状态
- **如果消息显示但没有 seq**: 检查 Tinode 服务器日志
- **如果出现"Tinode not initialized"错误**: 重新连接或重启应用

### 场景 2.4: 接收消息（实时）

**目标**: 验证实时消息接收

**前置条件**:
- 两台设备/模拟器 或 一台模拟器 + 网页管理后台
- 设备1: 账号A 已登录
- 设备2: 账号B 已登录

**步骤**:
1. 设备1: 账号A 停留在与账号B 的聊天室
2. 设备2: 账号B 发送消息: "你好，收到了吗？"
3. **观察设备1** 实时消息到达

**设备1 预期行为**:
- 消息在 2 秒内到达
- 消息显示在左侧（isMe=false）
- 发送者名称/头像显示正确
- 消息内容完整正确
- 时间戳准确
- 未读数更新（如果不在聊天室）

**预期的控制台日志（设备1）**:
```
[Tinode] 📩 收到消息: {data: {...}}
```

**验收清单**:
- [ ] 消息实时接收（<2 秒）
- [ ] 消息内容完整正确
- [ ] 发送者信息显示正确
- [ ] 消息显示在左侧（接收方）
- [ ] 时间戳准确
- [ ] 未读数更新正确

**代码参考**:
- `mobile/src/services/TinodeService.ts:220-223` (onMessage 回调)
- `mobile/src/screens/ChatRoomScreen.tsx` (topic.onData 监听器)

**故障排除**:
- **如果延迟 > 2 秒**: 检查网络延迟、Tinode 服务器负载
- **如果消息不到达**: 检查 WebSocket 连接状态
- **如果消息重复**: 检查 topic 订阅逻辑

### 场景 2.5: 已读回执

**目标**: 验证已读状态同步

**前置条件**:
- 两台设备，账号A 和账号B
- 账号B 已向账号A 发送未读消息

**步骤**:
1. 设备1: 账号A 打开与账号B 的会话
2. 设备1: 滚动查看所有未读消息
3. **观察设备1 的未读数**
4. 设备2: **观察账号B 侧的消息状态**

**预期行为**:
- 设备1: 未读数角标消失或显示 0
- 设备2: 消息状态变为"已读"
- 状态更新在 2 秒内发生

**预期的控制台日志（设备1）**:
```
[Tinode] 标记已读: <seqId>
```

**验收清单**:
- [ ] 设备1 未读数清零
- [ ] 已读状态同步到设备2
- [ ] 状态更新实时（<2 秒）
- [ ] 无错误

**代码参考**:
- `mobile/src/services/TinodeService.ts:472-479` (markAsRead)
- `mobile/src/screens/ChatRoomScreen.tsx` (已读回执逻辑)

**故障排除**:
- **如果未读数不清零**: 检查 markAsRead 调用时机
- **如果状态不同步**: 检查两台设备的 WebSocket 连接

### 场景 2.6: 离线消息

**目标**: 验证离线消息送达和同步

**步骤**:
1. 设备1: 账号A 在线
2. 设备1: **完全关闭应用**（从应用切换器滑走）
3. 设备2: 账号B 向账号A 发送 3 条消息:
   - "离线消息1"
   - "离线消息2"
   - "离线消息3"
4. 等待 5 秒
5. 设备1: **重新打开应用**（账号A）
6. 进入与账号B 的会话
7. **观察消息送达**

**预期行为**:
- 应用重新打开后自动拉取所有 3 条离线消息
- 消息按正确的时间顺序显示
- 未读数显示 3（或总未读数）
- 消息在打开会话时自动加载
- 无消息丢失

**预期的控制台日志（设备1）**:
```
[Tinode] 初始化中...
[Tinode] WebSocket 已连接
[Tinode] 登录成功
[Tinode] 订阅会话: usr...
```

**验收清单**:
- [ ] 所有离线消息送达
- [ ] 消息按正确顺序显示
- [ ] 未读数准确
- [ ] 无消息重复
- [ ] 重连后自动同步

**代码参考**:
- `mobile/src/services/TinodeService.ts:322-382` (subscribeToConversation)
- Tinode SDK 自动处理离线消息队列

**故障排除**:
- **如果消息丢失**: 检查 Tinode 服务器消息保留设置
- **如果顺序错误**: 检查 seq 号排序逻辑
- **如果出现重复**: 检查消息去重逻辑

### 场景 2.7: 断线重连

**目标**: 验证自动重连处理

**步骤**:
1. 设备1: 账号A 在线并已连接
2. **模拟网络断开**:
   - iOS 模拟器: 在设置中开启飞行模式
   - Android 模拟器: 禁用 WiFi/移动数据
   - 或: 停止 Tinode 容器: `docker stop decorating_tinode`
3. 等待 3 秒
4. **观察控制台日志和界面**
5. **恢复网络连接**:
   - 重新启用 WiFi/移动数据
   - 或: 重启 Tinode: `docker start decorating_tinode`
6. 等待自动重连
7. **观察重连行为**

**预期的控制台日志**:
```
[Tinode] ❌ 已断开: <error>
[Tinode] 🔄 尝试重连...
[Tinode] 初始化中...
[Tinode] WebSocket 已连接
[Tinode] 登录成功
[Tinode] ✅ 已连接
```

**预期的界面行为**:
- 连接状态指示器变为红色 WifiOff 图标
- 重连后: 指示器变回绿色 Wifi 图标
- 重连后消息自动同步
- 无数据丢失

**验收清单**:
- [ ] 3 秒内检测到断线
- [ ] 控制台显示"❌ 已断开"
- [ ] 3 秒后自动尝试重连
- [ ] 重连成功
- [ ] 控制台显示"✅ 已连接"
- [ ] 重连后消息同步
- [ ] 界面状态指示器更新正确

**代码参考**:
- `mobile/src/services/TinodeService.ts:206-215` (onDisconnect)
- `mobile/src/services/TinodeService.ts:228-243` (reconnect)

**故障排除**:
- **如果重连反复失败**: 检查 Tinode 服务器可用性
- **如果重连不触发**: 检查 reconnectTimer 逻辑
- **如果消息不同步**: 检查重连后的 topic 订阅
