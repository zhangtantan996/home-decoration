## 1.1 关键结论（摘要）

- 索引与事务：已在模型与 migrations 中广泛使用；关键业务存在事务封装（同时存在 service 与部分 handler 直接开事务的现状）。
- 并发一致性：未发现显式隔离级别设置；escrow 关键写路径已使用 `FOR UPDATE` 行锁，其余场景是否足够取决于具体高并发写场景。
- 会话体系：JWT claims 含 `token_use/jti/sid`，刷新 token 有 Redis 重放检测与会话撤销入口（`session:{sid}:*` 写入点需进一步确认）。
- 消息体系：既有站内通知（notifications）也有短信验证码（SMS）与第三方 IM（Tinode/Tencent IM）；未发现通用 MQ 与后端自研 WebSocket 模块。
- 吞吐/观测：有应用层限流（Redis 为主，内存回退），但未发现通用 metrics/pprof/Tracing；DB 连接池已配置默认值（可通过 config/env 覆盖）。

**现状结论**：有（模型定义与迁移脚本均存在索引/唯一约束）。  

**证据**：
- GORM 模型 tag（`index/uniqueIndex`）：
  - `server/internal/model/dictionary.go`（`uniqueIndex`、`index`）
  - `server/internal/model/region.go`（`uniqueIndex`、`index`）

  - `server/migrations/001_create_notifications_table.sql`（多列/单列索引）

**备注（边界）**：
- 索引“是否已落库”取决于迁移策略：`server/internal/repository/database.go` 中 `AutoMigrate` 受配置开关控制；生产更常见做法是执行 `server/migrations/*.sql`（本次不验证实际 DB 状态）。
- 补充（非穷举示例）：核心实体普遍使用 `index/uniqueIndex`（如用户 `publicId/phone` 唯一约束、微信绑定复合唯一约束、RBAC 账号/角色唯一约束、字典/行政区划索引等）。

## 7. 泄漏（内存/资源/goroutine）

**现状结论**：部分（存在后台 goroutine、ticker、全局 context 等模式；未发现统一泄漏检测/治理机制）。  

**证据**：
- 请求结束后异步 goroutine（用于日志/同步等）：
  - `server/internal/middleware/audit_logger.go`（请求完成后异步写审计日志；已改为仅传入拷贝字段，不依赖 `*gin.Context` 生命周期）
  - `server/internal/service/user_service.go`（注册后异步同步到 IM：`go func(...)`）
- ticker 生命周期：
  - `server/internal/cron/*.go`（多数 `NewTicker` 伴随 `defer ticker.Stop()`）
  - `server/internal/middleware/rate_limit.go`（cleanup ticker 不显式 stop，符合“全局单例常驻”模式）
- 内存 map 缓存（容量增长风险点，是否可控取决于调用频率与清理策略；当前已优先迁移到 Redis，但仍保留回退逻辑）：
  - `server/internal/middleware/rate_limit.go`（Redis 计数限流为主；`requests map[string][]time.Time` 为回退实现）
  - `server/internal/service/sms_service.go`（Redis 频控为主；`phoneRecords/ipRecords` 为回退实现；Redis TTL 自动回收，无需定时清理）
- Redis 操作 context：
  - `server/internal/repository/redis.go`（提供 `RedisContext()` 操作级超时；仍保留历史 `Ctx` 但已不在业务调用中使用）

**备注（边界）**：
- 静态检索只能识别“可能导致泄漏的代码形态”，无法证明“没有泄漏”；若要确认需引入运行时 profiling/指标（见“可选下一步”）。
- 额外观察（已处理）：`audit_logger.go` 之前将 `*gin.Context` 直接传入异步 goroutine，存在对象复用/并发安全风险；现已改为构造审计条目后异步写入。

## 11. TPS/QPS（限流/指标/观测/连接池）

**现状结论**：部分（有应用层限流；缺少通用观测；DB 连接池已配置默认值但需结合环境调参）。  

### 11.1 限流（Rate Limit）

**证据**：
- `server/internal/middleware/rate_limit.go`（按 IP 限流；优先 Redis 计数（多实例共享）+ 内存回退；并提供 `LoginRateLimit`/`SensitiveRateLimit`）
- `server/internal/router/router.go`（全局 `middleware.RateLimit()`；登录/注册/验证码等敏感入口额外限流）
- `server/internal/service/identity_service.go`（身份切换限流：Redis `INCR` + `Expire`，约 5 次/分钟）
- `server/internal/service/sms_service.go`（短信验证码发送频控：手机号 60 秒/次、IP 每日 20 次；优先 Redis + TTL，内存回退）

**备注（边界）**：
- `rate_limit.go` 已迁移为 Redis 计数限流（多实例共享）；当 Redis 不可用时回退为进程内 map 限流（单实例有效）。

### 11.2 指标/观测（metrics/pprof/tracing）

**现状**：
- 未发现通用 Prometheus/pprof/OpenTelemetry 等接入。
- 有少量进程内计数/快照（不等同于完整观测体系）：
  - `server/internal/monitor/public_id_rollout.go`（rollout snapshot/metrics store）
  - `server/internal/monitor/public_id_rollback.go`

### 11.3 DB 连接池与超时（sql.DB pool / timeout）

**现状**：
- 生产代码已显式设置 `SetMaxOpenConns/SetMaxIdleConns/SetConnMaxLifetime/SetConnMaxIdleTime`（默认值 + 可配置）：
  - `server/internal/repository/database.go`
  - `server/internal/repository/tinode_db.go`
  - `server/internal/config/config.go`（`database.max_open_conns/max_idle_conns/conn_max_lifetime_minutes/conn_max_idle_time_minutes`）
- 测试代码中存在连接池设置（用于测试稳定性/资源控制）：
  - `server/internal/service/user_service_test.go`
  - `server/internal/handler/tinode_handler_test.go`

### 11.4 请求超时/取消（Context timeout/cancel）

**现状**：
- Redis 已引入操作级 `context.WithTimeout`（默认 3s，可通过配置/环境变量覆盖）：
  - `server/internal/repository/redis.go`（`RedisContext()` + `redis.operation_timeout_ms`）
  - `server/internal/service/token_service.go` / `server/internal/service/sms_verification.go` / `server/internal/service/identity_service.go`

- 仍未发现统一的 request context 传递规范；DB/GORM 查询与第三方 HTTP 调用未见系统性超时/取消封装（本报告仅静态盘点）。

## 13. 可选下一步（若要升级为“清单 + 风险”）

1. 选取关键业务链路（支付/托管/订单/身份切换）做并发一致性审计：幂等、重试、读改写竞态、事务边界合理性。  
2. 针对“需要强一致读”的场景评估是否需要显式隔离级别/行锁（并沉淀规范/模板）。  
3. 建立可观测性：请求耗时/错误率/慢查询/pprof/指标面板，形成 TPS/QPS 与容量基线。  
4. 审计 Redis 用法（KEYS/SCAN、会话撤销策略、错误处理）并形成一致的封装规范。  
5. Tinode 直连 DB 的消息操作评估迁移到官方 API，以减少一致性与兼容性风险。
