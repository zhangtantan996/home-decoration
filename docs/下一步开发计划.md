## 一、开发策略总览

### 1.1 核心原则

根据PRD v2.3的基线声明，当前阶段的开发策略是：

**先把主链路做实，再做增项**

- **主链路9个节点** = 必做（基线）
- **产品迭代方案** = 可选（增项池）

### 1.2 当前阶段定位

- **阶段判断**: 可信成交 + 可控履约 + 可追责异常的收敛期
- **端面战略**:
  - 小程序 = 主成交端
  - Merchant Web = 主履约端
  - Admin Web = 主治理端
  - Web/H5 = 承接页/支付结果页
- **不进入当前MVP**: App、独立监理端

### 1.3 优先级排序

```
基线主链路 > 主链路异常收口 > 支撑性能力 > 增项
```

## 二、第一优先级：基线主链路完善（P0）

**目标**: 确保主链路9个节点完整可用
**工期**: 2-3周（16-24人天）
**负责端**: 小程序 + Merchant Web + Admin Web

### 2.1 设计确认→工长确认桥接优化

**问题**: PRD规则1明确要求两个独立成交点必须分离，当前实现可能存在混淆

**任务清单**:
- [ ] 审查现有流程：设计确认后是否清晰引导到工长选择
- [ ] 优化小程序UI：设计确认页面增加"下一步：选择施工方"引导
- [ ] 数据模型检查：确保设计确认和工长确认是两个独立状态
- [ ] 商家端优化：设计师和工长的任务列表分离展示

**技术实现**:
- 小程序端：`mini/src/pages/proposal/` 增加桥接页面
- 后端：检查 `server/internal/service/proposal_service.go` 和 `project_service.go` 的状态流转
- 数据库：检查 `projects` 表的状态字段是否支持双成交点

**验收标准**:
- 用户完成设计确认后，必须进入工长选择流程
- 设计师和工长在商家端看到的任务清晰分离
- 状态机支持：预约 → 设计确认 → 工长确认 → 项目创建

**工期**: 3-5天

---

### 2.2 分阶段验收放款系统

**问题**: PRD规则2要求必须支持按阶段验收，也支持整体验收

**任务清单**:
- [ ] 检查现有 `Milestone` 模型是否支持分阶段验收
- [ ] 实现分阶段验收流程：开工付30% → 水电完付30% → 中期付30% → 验收付10%
- [ ] 实现整体验收流程：一次性验收放款
- [ ] 托管账户支持分期放款
- [ ] 小程序端：用户验收界面（按阶段/整体）
- [ ] 商家端：商家提交验收申请界面

**技术实现**:
- 后端：扩展 `server/internal/service/escrow_service.go`
- 数据库：新增 `milestone_payments` 表（如果不存在）
- 小程序端：`mini/src/pages/project/` 新增验收页面
- 商家端：`merchant/src/pages/merchant/` 新增验收申请页面

**验收标准**:
- 支持按阶段验收：每个阶段独立验收、独立放款
- 支持整体验收：一次性验收、一次性放款
- 托管账户正确冻结和释放资金
- 验收记录可追溯

**工期**: 5-7天

---

### 2.3 异常收口流程（退款/关闭/审计）

**问题**: PRD规则3明确退款、关闭、审计属于主链路异常收口，不是附属售后能力

**任务清单**:
- [ ] 实现退款流程：用户申请 → Admin审核 → 退款执行
- [ ] 实现项目关闭流程：异常关闭、正常关闭
- [ ] 实现审计流程：项目审计、资金审计
- [ ] Admin端：退款审核、关闭审核、审计管理
- [ ] 小程序端：用户申请退款入口
- [ ] 商家端：商家查看退款/关闭状态

**技术实现**:
- 后端：扩展 `server/internal/service/project_service.go` 和 `escrow_service.go`
- 数据库：检查 `projects` 表的关闭状态、`transactions` 表的退款记录
- Admin端：`admin/src/pages/finance/` 新增退款审核页面
- 小程序端：`mini/src/pages/project/` 新增退款申请页面

**验收标准**:
- 用户可以申请退款，Admin可以审核
- 项目可以正常关闭或异常关闭
- 所有异常操作有审计记录
- 资金流转可追溯

**工期**: 5-7天

## 八、风险与应对

### 8.1 技术风险

**风险**: 电子签章API集成复杂度高

**应对**:
- 提前调研e签宝、法大大等第三方服务
- 预留2-3天的集成测试时间
- 准备备选方案（如简化版电子合同）

---

### 8.2 业务风险

**风险**: 多工长报价PK可能导致商家流失

**应对**:
- 前期只推送给优质工长
- 提供商家培训和工具
- 建立商家激励机制

---

### 8.3 合规风险

**风险**: 托管支付需要支付牌照

**应对**:
- 接入第三方支付（如连连支付、汇付天下）
- 申请支付牌照（长期）
