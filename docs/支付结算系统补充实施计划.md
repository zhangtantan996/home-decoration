# 支付结算系统补充实施计划

## 背景

根据 `/Volumes/tantan/AI_project/home-decoration/docs/支付对接与阶段性结算落地方案.md` 的要求，项目已有完整的支付结算基础设施，但缺失关键的对账机制和自动化补偿能力。

## 现状评估

### 已实现
- ✅ 微信小程序支付下单 (`WechatPayGateway.CreateMiniProgramPayment`)
- ✅ 支付回调解析 (`WechatPayGateway.ParseNotifyRequest`)
- ✅ 阶段支付计划 (`PaymentPlan`)
- ✅ 节点验收结算 (`SettlementService.ReleaseMilestone`)
- ✅ 商家收入账 (`MerchantIncome`)
- ✅ 提现申请与审核 (`MerchantWithdraw`, `AdminWithdrawHandler`)
- ✅ 退款申请与执行 (`RefundService`)

### 缺失
- ❌ 查单补偿定时任务
- ❌ 超时订单自动关闭
- ❌ 支付/退款/结算对账机制
- ❌ 自动出款通道（当前为线下打款）

## 实施优先级

### P0（资金安全）
1. 创建对账数据模型
2. 补充微信支付网关缺失方法
3. 实现支付查单补偿定时任务
4. 实现超时订单自动关闭
5. 实现支付对账定时任务

### P1（业务完整性）
6. 实现退款对账定时任务
7. 实现结算对账定时任务
8. 实现异常订单告警
9. 实现对账报表API

## 详细实施方案

### Task 1: 创建对账数据模型

**文件**: `server/internal/model/reconciliation.go`

**数据模型**:
```go
// ReconciliationRecord 对账记录主表
type ReconciliationRecord struct {
    Base
    ReconcileDate    time.Time `json:"reconcileDate" gorm:"index"`
    ReconcileType    string    `json:"reconcileType" gorm:"size:30;index"` // payment/refund/settlement
    Channel          string    `json:"channel" gorm:"size:20"`
    TotalCount       int       `json:"totalCount"`
    MatchedCount     int       `json:"matchedCount"`
    DifferenceCount  int       `json:"differenceCount"`
    TotalAmount      float64   `json:"totalAmount"`
    DifferenceAmount float64   `json:"differenceAmount"`
    Status           string    `json:"status" gorm:"size:20;index"` // processing/completed/failed
    ErrorMessage     string    `json:"errorMessage" gorm:"size:500"`
    CompletedAt      *time.Time `json:"completedAt"`
}

// ReconciliationDifference 对账差异明细表
type ReconciliationDifference struct {
    Base
    ReconciliationID uint64  `json:"reconciliationId" gorm:"index"`
    DifferenceType   string  `json:"differenceType" gorm:"size:30"` // missing_in_platform/missing_in_channel/amount_mismatch/status_mismatch
    OutTradeNo       string  `json:"outTradeNo" gorm:"size:64;index"`
    ProviderTradeNo  string  `json:"providerTradeNo" gorm:"size:64;index"`
    PlatformAmount   float64 `json:"platformAmount"`
    ChannelAmount    float64 `json:"channelAmount"`
    PlatformStatus   string  `json:"platformStatus" gorm:"size:30"`
    ChannelStatus    string  `json:"channelStatus" gorm:"size:30"`
    Resolved         bool    `json:"resolved" gorm:"default:false"`
    ResolvedAt       *time.Time `json:"resolvedAt"`
    ResolvedBy       uint64  `json:"resolvedBy"`
    ResolveNotes     string  `json:"resolveNotes" gorm:"size:500"`
}
```

**注册模型**: 在 `server/internal/repository/database.go` 的 `AutoMigrate` 中添加

### Task 2: 补充微信支付网关缺失方法

**文件**: `server/internal/service/wechat_pay_gateway_service.go`

**新增方法**:
1. `QueryOrderByOutTradeNo(ctx, outTradeNo)` - 查询支付单
2. `CloseOrder(ctx, outTradeNo)` - 关闭订单
3. `QueryRefund(ctx, outRefundNo)` - 查询退款
4. `DownloadBill(ctx, billDate, billType)` - 下载对账单

**接口定义**: 在 `server/internal/service/payment_channel_service.go` 补充接口

### Task 3: 实现支付查单补偿定时任务

**文件**: `server/internal/cron/payment_compensation_cron.go`

**执行逻辑**:
1. 查询状态为 `pending` 且创建时间超过5分钟的支付单
2. 调用微信查单API
3. 根据查单结果更新支付单状态
4. 记录补偿日志到 `payment_callbacks` 表
5. 触发业务回调（如果状态变更为 `paid`）

**执行频率**: 每5分钟

**注册**: 在 `server/cmd/api/main.go` 注册定时任务

### Task 4: 实现超时订单自动关闭

**文件**: `server/internal/cron/payment_timeout_cron.go`

**执行逻辑**:
1. 查询状态为 `pending/launching` 且 `expired_at` 已过期的支付单
2. 调用微信关单API
3. 更新支付单状态为 `closed`
4. 释放关联业务对象的锁定状态

**执行频率**: 每10分钟

### Task 5: 实现支付对账定时任务

**文件**: `server/internal/cron/payment_reconciliation_cron.go`

**执行逻辑**:
1. 调用微信下载账单API
2. 解析账单CSV文件
3. 对比平台 `payment_orders` 表
4. 生成对账报告，记录差异到 `reconciliation_records` 和 `reconciliation_differences` 表
5. 差异类型: 漏单、多单、金额不符、状态不符
6. 发送告警通知

**执行频率**: 每日凌晨2点

### Task 6: 实现退款对账定时任务

**文件**: `server/internal/cron/refund_reconciliation_cron.go`

**执行逻辑**:
1. 查询状态为 `processing` 且创建时间超过1小时的退款单
2. 调用微信退款查询API
3. 更新退款单状态
4. 记录对账差异

**执行频率**: 每小时

### Task 7: 实现结算对账定时任务

**文件**: `server/internal/cron/settlement_reconciliation_cron.go`

**执行逻辑**:
1. 查询状态为 `paid` 的结算单
2. 对比关联的 `payout_orders` 和 `merchant_income` 记录
3. 检查金额一致性、状态一致性
4. 记录对账差异

**执行频率**: 每日凌晨3点

### Task 8: 实现异常订单告警

**文件**: `server/internal/service/alert_service.go`

**告警场景**:
- 对账发现差异
- 支付回调验签失败
- 退款失败
- 结算出款失败

**告警渠道**: 系统通知 + 可选的企业微信/钉钉webhook

### Task 9: 实现对账报表API

**文件**: `server/internal/handler/admin_reconciliation_handler.go`

**API端点**:
- `GET /api/v1/admin/reconciliation/list` - 对账记录列表
- `GET /api/v1/admin/reconciliation/:id/differences` - 差异明细
- `POST /api/v1/admin/reconciliation/:id/resolve` - 标记差异已处理

## 技术约束

1. **金额计算**: 必须使用 `decimal.Decimal`，不能用 `float64`
2. **幂等性**: 所有回调、补偿、对账操作必须幂等
3. **事务边界**: 状态变更必须在同一事务内完成
4. **审计日志**: 所有资金操作必须记录完整审计日志
5. **错误处理**: 所有异常必须记录并告警

## 风险提示

1. **微信API限流**: 查单、关单、下载账单都有频率限制，需要合理控制调用频率
2. **对账时间窗口**: 微信账单T+1可用，需要考虑时间差
3. **并发安全**: 补偿任务和回调可能并发执行，需要加锁保护
4. **数据一致性**: 对账发现差异后的处理流程需要人工介入
