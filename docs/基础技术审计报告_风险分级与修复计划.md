# 后端基础技术审计报告（风险分级 + 修复计划）

> 范围：仅 `server/`（Go + Gin + GORM，PostgreSQL + Redis，Tinode/Tencent IM）。  
> 方法：静态代码/配置/文档审计；不连接真实数据库，不做 explain/analyze，不做压测/线上观测。  
> 依赖清单：基于 `docs/基础技术审计报告.md` 的 inventory 结果，并对照 `docs/技术基础分析报告.md` 补充查漏。

## 0. 分级口径

- **P0（阻断/高风险）**：高概率导致线上故障、数据错误或安全/稳定性事故；建议立即修复。
- **P1（重要）**：在并发/多实例/异常场景下风险显著；建议近期修复并补测试/观测。
- **P2（改进）**：架构一致性、可维护性、可观测性提升项；可排期推进。

## 0.1 实施状态（截至 2026-02-27）

- ✅ F-01：已改为异步 goroutine 不再依赖 `*gin.Context`（只传入拷贝后的字段）。
- ✅ F-02：已在 `InitDB` 与 `InitTinodeDB` 配置 `sql.DB` 连接池（支持 `DATABASE_MAX_OPEN_CONNS` 等环境变量覆盖）。
- ✅ F-03：已为关键 Redis 调用引入超时 context（支持 `REDIS_OPERATION_TIMEOUT_MS` 环境变量覆盖）。
- ✅ F-04：已在 escrow 关键写路径引入行锁（`SELECT ... FOR UPDATE`），降低并发丢更新/重复释放风险。
- ✅ F-05：已将 IP 限流迁移为 Redis 计数（多实例共享），并保留内存回退逻辑。
- ✅ F-06：已将 SMS 发送频控迁移为 Redis（手机号 cooldown + IP 日计数 + TTL 自动回收）。
- ⏳ 其余项：本轮未实现（仅给出风险与计划）。

## 2. 详细问题与修复计划

### F-01（P0）异步 goroutine 使用 `*gin.Context`

**现象/风险**  
`AuditLogger` 在请求结束后启动 `go saveAuditLog(c, ...)`，而 `saveAuditLog` 内部继续读取 `c.Get*`、`c.ClientIP()`、`c.Request.UserAgent()`、`c.Writer.Status()` 等字段。Gin 的 `Context` 通常来自对象池，请求结束后可能被复用；异步访问可能出现数据竞争、读取到错误值或触发不稳定问题。

**证据**  
- `server/internal/middleware/audit_logger.go`

**修复建议（不改业务，仅改实现方式）**  
- 在启动 goroutine 前**拷贝所需字段**（operatorId/operatorType/path/body/ip/ua/status/duration 等），将“纯数据结构”传入 goroutine，避免传 `*gin.Context`。

**验收/验证**  
- `go test -race ./...`（至少覆盖涉及中间件/handler 的测试路径）。  
- 手工验证：同一接口并发压测时审计日志 operatorId、ip、status 等字段不串数据（需要运行环境支持）。

---

### F-02（P0）DB 连接池未显式配置（生产路径）

**现象/风险**  
`gorm.Open` 之后未看到对底层 `*sql.DB` 连接池的设置。默认行为依赖驱动与运行时，可能在高并发下出现连接数失控、数据库拒绝连接、级联超时等问题。

**证据**  
- `server/internal/repository/database.go`

**修复建议**  
- `InitDB` 成功后获取 `sqlDB, err := DB.DB()`，配置：
  - `SetMaxOpenConns` / `SetMaxIdleConns`
  - `SetConnMaxLifetime` / `SetConnMaxIdleTime`
- 值的选择建议从环境变量/配置文件读取（避免硬编码），并在文档中给默认值与说明。

**验收/验证**  
- 压测或并发请求下：DB 连接数稳定，且 API 响应无明显连接耗尽错误。  
- 增加运行日志/指标：启动时打印 pool 配置（不打印敏感信息）。

---

### F-03（P1）缺少统一 request timeout/cancel

**现象/风险**  
静态检索未发现 `context.WithTimeout/WithCancel` 的系统性使用；在本轮修复前，Redis 调用普遍使用 `context.Background()`。当下游（DB/Redis/第三方）抖动或阻塞时，容易造成请求堆积、goroutine 堆积，且缺乏统一的“取消/超时”策略。

**证据**  
- `server/internal/repository/redis.go`（`RedisContext()` + `redis.operation_timeout_ms`）
- `server/internal/service/token_service.go`、`server/internal/service/sms_verification.go`、`server/internal/service/identity_service.go`
- `server/internal/handler/admin_auth_handler.go`

**本轮已实施**  
- 为 Redis 调用统一改为使用 `repository.RedisContext()`（默认 3s，可用 `REDIS_OPERATION_TIMEOUT_MS` 覆盖），避免无限期等待。

**修复建议（分阶段）**  
1) P1：继续为关键外部调用（HTTP/第三方 SDK）引入超时与取消，并尽量从 handler 向下传递 request context。  
2) P2：形成统一封装（例如 `ctxutil.WithTimeoutFromGin(c, ...)`），避免各处自写不一致；DB/GORM 查询逐步使用 `WithContext`。

**验收/验证**  
- 模拟 Redis 不可用/超时：请求能在可控时间内返回错误，而非长时间卡住。  

---

### F-04（P1）高并发写缺少显式隔离/锁/乐观锁策略

**现象/风险**  
未发现显式事务隔离级别设置，也未看到“版本号乐观锁”的通用策略。资金/余额类（例如 escrow）在高并发下可能出现幻读、丢更新、余额不一致等并发异常（是否会触发取决于真实调用并发与写模式）。

**证据**  
- `server/internal/service/escrow_service.go`（资金相关事务存在；关键读改写路径已补齐 `FOR UPDATE` 行锁）  
- `server/internal/repository/database.go`（未设置隔离级别）

**本轮已实施**  
- `server/internal/service/escrow_service.go`：在 `Deposit`/`ReleaseFunds` 事务内对 `escrow_accounts` 与 `milestones` 做 `SELECT ... FOR UPDATE`（GORM `clause.Locking{Strength:"UPDATE"}`），避免并发读改写导致的丢更新/重复释放。

**修复建议**  
- 选型其一（需结合业务并发与性能目标）：
  - 悲观锁：更新前 `SELECT ... FOR UPDATE` 锁定行；或用 GORM locking clause。  
  - 乐观锁：为关键表增加 `version` 字段并用 `WHERE version = ?` 更新。  
  - 高一致隔离：对少数关键事务使用更高隔离级别（注意吞吐与冲突重试）。

**验收/验证**  
- 编写并发测试/脚本：模拟并发充值/释放资金，确保最终余额与交易流水一致。  

---

### F-05（P1）限流为单实例内存实现（多实例不一致）

**现象/风险**  
修复前：`rate_limit.go` 将请求时间戳保存在进程内 map；多实例部署时每个节点独立计数，整体限流将被“分片放大”。同时 map 容量与每个 key 的 slice 增长存在压力，需要依赖 cleanup 周期与流量模式。

**证据**  
- `server/internal/middleware/rate_limit.go`

**本轮已实施**  
- `server/internal/middleware/rate_limit.go`：IP 限流改为优先使用 Redis（`INCR` + `PEXPIRE`），多实例共享计数；Redis 不可用时回退到原内存滑动窗口实现。

**修复建议**  
- 若需要更精确的滑动窗口/更平滑的限流：可考虑 Redis ZSET 滑动窗口或令牌桶/漏桶实现（Lua 原子化），并补压测验证开销。  
- 若已有网关层：可将粗粒度限流前置到 Nginx/Ingress，应用层保留细粒度保护。

**验收/验证**  
- 多实例压测：限流阈值在集群维度按预期生效。  

---

### F-06（P1）SMS 频控为内存实现且清理未调度

**现象/风险**  
修复前：`SMSService` 使用内存 map 记录手机号与 IP 的发送记录，并提供 `CleanupOldRecords`，但静态检索未发现调用点。多实例下限流不一致；单实例下长期运行可能累积 map。

**证据**  
- `server/internal/service/sms_service.go`

**本轮已实施**  
- `server/internal/service/sms_service.go`：SMS 频控改为优先使用 Redis（手机号 60 秒 cooldown + IP 当日计数，使用 TTL 自动回收）；Redis 不可用时回退内存实现。
- `server/internal/service/sms_verification.go`：发送验证码成功后调用 `smsService.RecordSent(...)` 触发 Redis 计数更新。

**修复建议**  
- 若需要更严格一致性：可将“检查+占用名额”做成 Redis Lua 原子操作，避免并发场景下短暂超发。  
- 若希望区分“发送成功 vs 发送失败”的计数策略：需要明确业务口径并调整记录时机（当前逻辑在发送成功后计数，与历史一致）。

**验收/验证**  
- 跑一天的压测/模拟流量：map 大小稳定；或 Redis key TTL 正常回收。

### F-07（P2）会话撤销逻辑疑似不完整（`session:{sid}:*` + `KEYS`）

**现象/风险**  
`RevokeSession` 通过 `KEYS session:{sid}:*` 删除会话 token；但静态检索未发现写入这些 keys 的位置，导致撤销效果不确定。`KEYS` 在生产 Redis 上也可能造成阻塞风险（key 数量大时）。

**证据**  
- `server/internal/service/token_service.go`

**修复建议**  
- 明确 session 存储策略：  
  - 方案 A：在签发 token 时写入 `session:{sid}:{jti}` 集合/集合结构，并用 `SCAN`/集合成员删除；  
  - 方案 B：不存 session token，仅靠 refresh `jti` 与 token 过期窗口，撤销改为写入“session revoked”标记并在中间件校验。  

**验收/验证**  
- 撤销会话后：access token 是否立即失效（若期望立即失效）。  

---

### F-08（P2）Tinode 直连 DB 操作消息

**现象/风险**  
消息删除通过 SQL 直接 `DELETE FROM messages ...`，Tinode 官方更推荐通过其 API 进行消息操作，以确保相关数据一致性与副作用处理完整。

**证据**  
- `server/internal/tinode/grpc_client.go`（文件注释已标注 TODO）

**修复建议**  
- 评估并迁移到 Tinode REST/WebSocket API（以 Tinode 官方 del/相关接口为准）。

---

### F-09（P2）分层规范未完全贯彻

**现象/风险**  
仓库规范明确 `handler -> service -> repository`，但部分 handler 直接访问 `repository.DB` 并开启事务。长期会导致事务边界、错误处理、重试策略在多处散落，难以统一治理。

**证据**  
- `docs/Claude开发参考手册.md`
- `server/internal/handler/region_handler.go`

**修复建议**  
- 将“跨多步写操作”的事务边界逐步下沉到 service；handler 保持 IO/校验/响应。

---

### F-10（P2）可观测性不足（metrics/pprof/tracing）

**现象/风险**  
仅看到局部 in-memory snapshot 统计（publicId rollout/rollback/health），未见统一 metrics、pprof、trace 等体系。出现性能劣化/泄漏问题时定位成本高。

**证据**  
- `server/internal/monitor/public_id_rollout.go`
- `server/internal/monitor/public_id_rollback.go`
- `server/internal/monitor/public_id_health.go`

**修复建议**  
- 先引入最小可用的观测面：pprof、关键接口耗时/错误率指标、慢查询日志/采样。
