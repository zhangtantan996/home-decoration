## 二、任务1：设计确认→工长确认桥接优化

### 2.1 问题诊断

**核心问题**：
1. 设计确认后直接进入工长选择，没有过渡期
2. 工长选择与项目创建耦合，两个成交点没有清晰分离
3. 业务流程状态定义混乱

**具体表现**：
- `proposal_service.go` 的 `confirmProposalForBooking()` 在设计确认后立即推进到 `BusinessFlowStageConstructionPartyPending`
- `project_service.go` 中 CrewID 是可选参数，但实际创建项目时必须指定
- `BusinessFlowStageConstructionPartyPending` 混合了"待选择工长"和"待工长确认"两个概念

### 2.2 解决方案

#### 2.2.1 数据模型层调整

**文件**: `server/internal/model/business_closure.go`

**修改内容**:
```go
// 在 L17-19 添加新状态
const (
    // ... 现有状态 ...
    BusinessFlowStageDesignConfirmed        = "design_confirmed"        // 设计确认完成，待选择工长
    BusinessFlowStageConstructionPartyPending = "construction_party_pending" // 待工长确认
    BusinessFlowStageConstructorPending     = "constructor_pending"     // 工长已确认，待项目创建
)
```

**工作量**: 0.5天

---

#### 2.2.2 业务逻辑层重构

**文件1**: `server/internal/service/proposal_service.go`

**修改位置**: L301-401 `confirmProposalForBooking()` 函数

**修改内容**:
```go
// 设计确认后，状态改为 BusinessFlowStageDesignConfirmed
// 不再直接推进到工长选择
func (s *ProposalService) confirmProposalForBooking(ctx context.Context, bookingID uint, proposalID uint) error {
    // ... 现有逻辑 ...
    
    // 修改状态更新
    booking.BusinessFlowStage = model.BusinessFlowStageDesignConfirmed
    
    // 不再自动推进到工长选择
    // 需要用户主动进入工长选择流程
    
    return nil
}
```

**工作量**: 1天

```go
    // 1. 检查 booking 状态是否为 BusinessFlowStageConstructionPartyPending

    // 3. 通知用户
    return nil
}

// 创建项目（只有工长确认后才能调用）
func (s *ProjectService) CreateProject(ctx context.Context, bookingID uint) error {
    // 1. 检查 booking 状态是否为 BusinessFlowStageConstructorPending
    // 2. 创建项目
    // 3. 更新状态为 BusinessFlowStageProjectCreated
    return nil
}
```
