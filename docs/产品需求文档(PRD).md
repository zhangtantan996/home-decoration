## 0. 文档元信息与使用说明

### 0.1 文档定位

本文档是**装修设计一体化平台的唯一开发基线 PRD**。

- 本文档定义的"唯一基线业务流"是后续所有开发、测试、排期、验收的唯一裁剪标准。
- 任何未落在主链路中的功能、页面、接口、状态、报表或平台能力，统一视为**增项**，不得混入主线需求。
- 本文档优先级高于历史设计文档、口头需求、想象性功能描述。

### 0.2 基线声明

**已确认主业务流是唯一裁剪标准**：

1. 商家入驻与能力配置
2. 用户需求进入线索/预约
3. 设计师与用户进行对接，量房谈单，确认预算和设计方案
4. 设计师提交方案/报价，用户确认或拒绝
5. 选择要施工的工长，用户确认或拒绝
6. 确认后生成订单/业务闭环并创建项目
7. 项目按阶段/里程碑执行，伴随托管或账款流转
8. 用户验收，进入放款、退款或关闭
9. 支持分阶段验收放款，也支持一次性整体验收放款

**三条最高优先级规则**：

- **规则 1**：设计确认与工长确认是两个独立成交点，不能合并。
- **规则 2**：验收/放款必须天然支持按阶段，也必须支持整体验收一次性放款。
- **规则 3**：退款、关闭、审计属于主链路异常收口，不是附属售后能力。

### 0.3 判定规则

判定某功能是否属于基线主链路，按以下标准：

1. 是否推动主链路成交/履约/收款/退款/关闭？
2. 是否服务于已确认角色（用户/设计师/工长/装修公司/主材商/管理员）？
3. 是否改变主状态机？

**优先级解释规则**：

- 基线主链路 > 主链路异常收口 > 支撑性能力 > 增项

### 0.4 当前阶段补充说明（2026-04）

在不改变本 PRD 交易主链的前提下，当前阶段补充判断如下：

- 当前阶段是**平台型产品中前期**，重点不是横向铺更多功能，而是把成交、履约、治理闭环压实。
- 当前正式端面战略固定为：
  - **小程序 = 主成交端**
  - **Merchant Web = 主履约端**
  - **Admin Web = 主治理端**
  - **Web/H5 = 承接页 / 支付结果页 / 辅助浏览端**
- 当前需求评估默认先回答 3 个问题：
  1. 这是不是当前阶段刚需？
  2. 它是否真正提升平台增长、成交、履约或治理？
  3. 是否存在更短、更干净、更符合当前阶段的实现路径？

本补充说明用于约束产品优先级判断，不改变本 PRD 已冻结的 9 个主链路节点。

## 1. 业务目标与范围边界

### 1.1 产品本期目标

把商家入驻、线索转化、设计成交、施工成交、项目执行、分阶段验收放款闭环跑通。

### 1.2 核心价值

- **去黑箱化**：通过资金托管、阶段验收、审计留痕，建立工业级的信任标准。
- **双成交点**：设计确认与工长确认分离，用户决策更清晰，商家责任更明确。
- **分阶段放款**：支持按里程碑验收放款，也支持整体验收一次性放款，适配不同项目规模。

### 1.3 In Scope（本期范围）

- 商家入驻与能力配置
- 线索/预约进入
- 设计方案与报价确认
- 工长选择与确认
- 订单生成与项目创建
- 项目阶段执行
- 分阶段验收/放款
- 整体验收/一次性放款
- 异常收口：退款/关闭/审计
- 完工后的评价、案例沉淀与转介绍入口（支撑性增长能力，不新增主链节点）

### 1.4 Out of Scope / 增项池（非本期范围）

以下能力不属于当前基线，归入增项池或未来规划：

- AI 免费设计
- BIM 协同查看器
- AR 试衣间
- LBS 排序算法细节
- 工地地图、直播、复杂可视化能力
- 与当前基线无直接关系的"生态型"想象功能

### 1.5 当前阶段战略与主商业模型

**阶段判断**：

- 当前阶段不是“功能展示期”，而是“可信成交 + 可控履约 + 可追责异常”的收敛期。
- 当前最大风险不是页面不够，而是：
  - 成交前抓手不够强
  - 设计确认到工长确认之间桥接不足
  - 商业模型不够收敛
  - 多端复杂度开始压主线

**主商业模型**：

- 主：设计成交 + 施工成交撮合佣金
- 辅：托管 / 验收 / 风控服务费

**当前不作为主模型的方向**：

- 广告位 / 曝光位收费
- 主材商城主线
- 重 SaaS 收费
- 会员积分体系

**平台护城河定义**：

- 不是“商家展示更多”
- 而是“可信成交 + 可控履约 + 可追责异常”

## 2. 角色定义与职责边界

### 2.1 核心角色

| 角色 | 进入条件 | 在主链路中的动作 | 决策权 | 可见状态 |
|------|---------|----------------|-------|---------|
| 用户/业主 | 注册登录 | 发起需求、确认设计、确认施工主体/施工报价、验收、退款 | 设计确认、工长确认、验收确认 | 全流程可见 |
| 设计师 | 入驻审核通过 | 承接线索、量房谈单、提交沟通确认、设计交付、正式方案、报价基线 | 方案提交、报价调整 | 线索→设计确认 |
| 工长 | 入驻审核通过 | 承接施工、提交施工报价、执行项目 | 施工报价提交、阶段提交 | 工长确认→项目完工 |
| 装修公司 | 入驻审核通过 | 承接设计或施工（可映射到设计师或施工主体能力） | 同设计师或工长 | 同设计师或工长 |
| 监理专员 / project_supervisor | 内部授权 | 协调待开工协同、记录施工日志、上报风险、整理验收证据 | 风险上报、日志留痕 | ready_to_start→完工归档 |
| 主材商 | 入驻审核通过 | 展示商品、接受咨询（暂不纳入当前主链路成交闭环） | 商品上下架 | 旁路能力 |
| 平台管理员 | 内部授权 | 审核入驻、审核案例、审计异常、仲裁退款 | 审核通过/拒绝、仲裁决策 | 全流程可见 |

### 2.2 角色能力维度

- **设计师**：提供设计方案、预算、报价，不承担施工责任。
- **工长**：提供施工执行、排期、节点交付，不承担设计责任。
- **装修公司**：可同时提供设计与施工能力，按实际承接内容映射到对应角色。

## 4. 基线成交点定义

### 4.1 成交点 A：设计确认成交

**输入**：
- 设计方案
- 预算
- 报价

**用户决策**：
- 确认：进入施工方选择
- 拒绝：设计师可重新提交（最多 3 次），超过 3 次进入退款/关闭

**输出**：
- 进入施工方选择，**不等于项目已开工**
- 设计师获得设计费支付承诺（若有）

**关键约束**：
- 设计确认成交**不等于**施工成交
- 设计确认后，用户仍可选择不施工或更换施工方
- 设计确认与工长确认是两个独立成交点，不能合并

### 4.2 成交点 B：工长确认成交

**输入**：
- 拟施工工长/施工团队
- 施工报价/职责边界

**用户决策**：
- 确认：生成订单并创建项目
- 拒绝：重新选择工长或终止项目

**输出**：
- 生成订单
- 创建项目
- 进入项目执行阶段

**关键约束**：
- 工长确认成交是项目创建的**唯一触发条件**
- 工长确认前，所有状态均为"商机阶段"，不是"履约阶段"
- 工长确认后，项目进入正式履约，资金流转开始绑定阶段/里程碑

## 6. 逐阶段业务需求详述

### 6.1 商家入驻与能力配置

**业务目标**：
- 建立供给侧，形成可接单的商家/材料商/服务配置。

**入驻对象**：
- 设计师（个人/公司）
- 工长（个人/公司）
- 装修公司
- 主材商

**审核前后状态**：
- 提交入驻申请 → 审核中 → 审核通过/审核拒绝
- 审核拒绝后可重新提交（最多 3 次）

**可配置能力**：
- 设计能力：设计风格、报价方式、案例展示
- 施工能力：工种、施工年限、亮点标签
- 服务区域：省市区范围
- 报价方式：按平米/按套/按项

**与后续线索分发/可选商家范围的关系**：
- 入驻审核通过后，商家进入可接单池
- 线索分发时，按服务区域、能力匹配度分发
- 用户选择工长时，按服务区域、工种、评价筛选

**关键接口**：
- `POST /api/v1/merchant/apply` - 商家入驻申请
- `POST /api/v1/merchant/apply/:id/resubmit` - 驳回重提
- `POST /api/v1/material-shop/apply` - 主材商入驻申请
- `GET /api/v1/merchant/apply/:phone/status` - 查询入驻状态

**关键页面**：
- 商家端：入驻申请页、入驻进度页、能力配置页
- 管理端：入驻审核列表、入驻审核详情

### 6.2 线索/预约进入

**业务目标**：
- 把用户需求从预约、商机、线索推进到可提交的方案/报价。

**用户如何发起**：
- 用户端：浏览设计师列表 → 点击"预约设计" → 填写需求 → 提交

**进入的是线索还是预约**：
- 预约：用户明确选择设计师，支付意向金
- 线索：用户未明确选择设计师，平台分发

**分发/承接规则**：
- 预约：直接分发给指定设计师
- 线索：按服务区域、能力匹配度分发给多个设计师，先承接先得

**从线索转沟通、从预约转设计阶段的条件**：
- 设计师承接线索/预约 → 进入沟通中
- 设计师与用户沟通确认需求 → 进入设计中

**关键接口**：
- `POST /api/v1/bookings` - 创建预约
- `POST /api/v1/bookings/:id/confirm` - 设计师承接预约
- `GET /api/v1/merchant/leads` - 商家查看线索列表

**关键页面**：
- 用户端：设计师列表、预约页、预约详情
- 商家端：线索列表、预约列表、预约详情

### 6.3 设计沟通、量房、预算确认

**业务目标**：
- 用户与设计师对接，完成量房资料沉淀，并进入正式的预算/设计意向确认节点。

**用户与设计师对接**：
- 设计师承接预约后，与用户电话/IM 沟通
- 确认量房时间、预算范围、设计风格偏好

**量房资料上传**：
- 设计师/商家上传量房照片、尺寸记录、备注
- 用户侧只读查看量房资料，不作为正式确认节点

**预算/设计意向确认**：
- 设计师提交预算范围、设计方向、空间需求、工期预期、特殊要求
- 用户确认可接受，或驳回并填写原因
- 被驳回后允许商家基于同一条记录重提
- 累计驳回达到后台阈值后，才进入退款/关闭

**设计意向确认**：
- 设计师提供设计方向建议
- 用户确认设计意向

**进入"设计费报价/设计交付"前的前提条件**：
- 量房资料已上传
- 预算范围确认
- 设计意向确认

**关键接口**：
- `POST /api/v1/bookings/:id/site-survey` - 上传量房记录
- `POST /api/v1/bookings/:id/budget-confirm` - 预算确认

**关键页面**：
- 商家端：量房记录页、预算确认页
- 用户端：量房进度页、预算确认页

### 6.4 设计方案/报价提交与确认

**业务目标**：
- 设计阶段固定收口为：
  `沟通确认完成 → 设计费报价 → 用户确认并支付设计费 → 设计师提交正式方案 → 用户确认方案 → 进入施工桥接`
- 其中“设计费确认并支付”是成交动作，“正式方案确认”是进入施工桥接的动作。

**设计师报价与正式方案提交**：
- 设计师先提交设计费报价（设计费、说明、定金抵扣）
- 用户确认并支付设计费后，设计师再提交正式方案（效果图、平面图、材料清单）

**用户动作**：
- 设计费阶段：用户确认并支付设计费，进入正式交付阶段
- 正式方案阶段：用户确认后进入工长选择；若拒绝，设计师可重新提交（最多 3 次）

**拒绝后的重提、终止、关闭条件**：
- 拒绝次数 < 3：设计师可重新提交
- 拒绝次数 ≥ 3：自动触发退款，项目关闭

**这里必须强调：确认后只代表"设计成交"，不是施工成交**：
- 设计费支付后，只代表进入正式交付，不代表施工成交
- 正式方案确认后，用户仍可选择不施工或更换施工方
- 正式方案确认与工长确认是两个独立成交点

**关键接口**：
- `POST /api/v1/design-quotes/:id/confirm` - 用户确认设计费报价并生成订单
- `POST /api/v1/orders/:id/pay` - 用户支付设计费
- `POST /api/v1/proposals` - 提交正式设计方案
- `POST /api/v1/proposals/:id/confirm` - 用户确认方案
- `POST /api/v1/proposals/:id/reject` - 用户拒绝方案
- `POST /api/v1/merchant/proposals/resubmit` - 设计师重新提交

**关键页面**：
- 商家端：设计费报价页、正式方案提交页、方案详情页、重提页
- 用户端：预约工作台、设计费报价页、正式方案页

---

### 6.5 施工主体选择与确认

**业务目标**：
- 用户在正式方案确认后进入施工桥接，先看到报价基线，再选择施工主体并确认施工报价，这是**成交点 B：工长确认**。
- “工长确认”保留为业务术语，但实际确认对象允许是 `装修公司施工主体` 或 `独立工长`。
- 这一阶段同时承担“把设计成交桥接到施工成交”的职责，不能退化为单纯信息展示页。

**施工桥接子步骤**：
- 正式方案确认后，设计师先提交报价基线数据（工程量基础表）
- 用户看到基线说明后，再进入施工主体选择
- 施工主体可为：
  - 装修公司主体（可展示其内部工长/班组，但不强制先落到独立工长）
  - 独立工长主体
- 施工主体确定后，进入施工报价确认

**施工主体展示要素**：
- 主体名称、头像、施工年限、工种/能力标签
- 案例展示、用户评价、报价范围
- 主体类型：装修公司 / 独立工长

**桥接要求**：
- 必须支持工长对比，而不是只给单一工长详情
- 必须让用户看懂工期、预算、责任边界
- 必须明确增项、整改、验收、退款、争议规则
- 必须提供真实案例、评价与履约背书，降低用户从设计确认继续推进到施工确认的决策阻力

**用户确认/拒绝**：
- 用户确认：生成订单、创建项目，并进入待开工协同
- 用户拒绝：重新选择施工主体或终止项目

**拒绝后的回退逻辑**：
- 用户可重新选择施工主体
- 用户可终止项目并申请退款

**确认后触发订单生成和项目创建**：
- 生成订单：锁定设计方案、施工主体、施工报价
- 创建项目：进入 `ready_to_start`，由监理协调进场时间后再开工

**关键接口**：
- `GET /api/v1/foremen` - 查询工长列表
- `POST /api/v1/quote-tasks` - 创建施工报价任务
- `POST /api/v1/quote-tasks/:id/submit` - 工长提交施工报价
- `POST /api/v1/quote-tasks/:id/confirm` - 用户确认施工报价

**关键页面**：
- 用户端：施工主体选择/对比页、施工报价确认页
- 商家端：报价基线提交页、施工报价任务列表、施工报价提交页

---

### 6.6 订单生成、业务闭环与项目创建

**业务目标**：
- 把已确认的施工报价落成正式项目，并进入“待开工协同 -> 开工 -> 履约”的生命周期。

**何时生成订单**：
- 用户确认施工报价后，自动生成订单并创建项目

**订单与项目关系**：
- 订单：记录设计方案、施工主体、施工报价、支付计划
- 项目：记录待开工协同、施工执行、阶段验收、资金流转

**合同/支付计划/托管计划的绑定关系**：
- 合同：设计方案 + 施工报价 + 验收标准
- 支付计划：按阶段/里程碑划分支付节点
- 托管计划：资金托管到平台，按验收节点放款

**待开工协同定义**：
- 施工报价确认后，项目先进入 `ready_to_start`
- 当前阶段默认由 `project_supervisor` 负责协调计划进场时间、维护项目进展说明、记录施工日志与风险
- 首版不新增用户点击式“进场确认”，由监理记录协商结果并通知用户/商家

**"业务闭环"定义：从前端商机阶段进入履约阶段**：
- 商机阶段：线索/预约/设计方案/施工主体选择
- 履约阶段：订单生成/项目创建/待开工协同/施工执行/验收放款

**关键接口**：
- `POST /api/v1/orders` - 生成订单
- `POST /api/v1/projects` - 创建项目
- `GET /api/v1/business-flows/:id` - 查询业务闭环状态

**关键页面**：
- 用户端：订单详情、项目详情
- 商家端：订单列表、项目列表
- 管理端：业务闭环监控、订单审核

---

### 6.7 项目阶段执行与资金流转

**业务目标**：
- 项目进入履约，业务和资金都按阶段推进。

**阶段/里程碑定义**：
- 阶段：拆除、水电、泥瓦、木工、油漆、安装、验收
- 里程碑：每个阶段的关键节点（如水电验收、泥瓦验收）

**每阶段的执行记录、验收材料、付款条件**：
- 执行记录：工长上传施工照片、进度说明
- 验收材料：用户上传验收照片、验收意见
- 付款条件：阶段验收通过后，触发该阶段款项放款

**托管或账款流转规则**：
- 用户支付订单款项后，资金托管到平台
- 每个阶段验收通过后，平台放款给工长
- 最后一个阶段验收通过后，放款尾款

**阶段中异常如何冻结后续款项**：
- 阶段验收不通过：冻结该阶段及后续款项
- 工长整改完成后，用户重新验收
- 验收通过后，解冻款项并放款

**关键接口**：
- `POST /api/v1/projects/:id/start` - 项目开工
- `POST /api/v1/projects/:id/milestones/:milestoneId/submit` - 工长提交阶段
- `POST /api/v1/projects/:id/milestones/:milestoneId/approve` - 用户验收通过
- `POST /api/v1/projects/:id/milestones/:milestoneId/reject` - 用户验收不通过

**关键页面**：
- 用户端：项目详情、阶段进度、验收页
- 商家端：项目执行页、阶段提交页、收入明细
- 管理端：项目监控、资金流转监控

---

### 6.8 验收、放款、退款、关闭

**业务目标**：
- 完成项目闭环，把项目、资金、审计状态统一收口。

**分阶段验收/分阶段放款**：
- 每个阶段验收通过后，放款该阶段款项
- 支持多次验收、多次放款

**一次性整体验收/一次性放款**：
- 所有阶段完成后，用户一次性整体验收
- 验收通过后，一次性放款全部款项

**退款触发场景**：
- 设计方案拒绝次数 ≥ 3
- 工长选择失败
- 施工中途暂停/换人
- 阶段验收不通过且无法整改

**关闭场景**：
- 项目正常完成，全部款项放款完毕
- 项目异常终止，退款完成

**审计介入场景**：
- 用户与商家争议无法协商
- 资金流转异常
- 验收标准争议

**明确"异常收口属于主链路的一部分，不是售后附录"**：
- 退款、关闭、审计是主链路的合法分支
- 每种异常必须有触发条件、状态流转、责任人、资金处理、审计留痕

**完工后资产沉淀要求**：
- 项目关闭后，平台应支持用户评价、完工案例沉淀、商家履约标签生成、转介绍入口
- 该能力属于本期支撑性增长能力，不新增主链节点，也不改变主状态机
- 评价和案例沉淀的目标不是内容堆积，而是为后续预约、设计确认、工长确认提供可信背书

**关键接口**：
- `POST /api/v1/projects/:id/complete` - 项目完工
- `POST /api/v1/projects/:id/refund` - 申请退款
- `POST /api/v1/admin/projects/:id/audit` - 审计介入
- `POST /api/v1/admin/projects/:id/close` - 项目关闭

**关键页面**：
- 用户端：验收页、退款申请页
- 商家端：项目完工页、退款处理页
- 管理端：审计列表、仲裁页、项目关闭页

## 7. 主链路异常收口

### 7.1 设计阶段拒绝/终止

**触发场景**：
- 用户拒绝设计方案

**处理流程**：
- 拒绝次数 < 3：设计师可重新提交
- 拒绝次数 ≥ 3：自动触发退款，项目关闭

**状态流转**：
- 待用户确认 → 已拒绝 → 待用户确认（重提）→ 已拒绝（第3次）→ 退款中 → 已关闭

**责任人**：
- 用户：提供拒绝原因
- 设计师：根据拒绝原因调整方案
- 系统：自动触发退款

**资金处理**：
- 意向金全额退款

**审计留痕**：
- 拒绝原因、拒绝时间、重提次数、退款时间

---

### 7.2 工长确认失败/重新选择

**触发场景**：
- 用户拒绝工长施工报价
- 工长无法承接项目

**处理流程**：
- 用户可重新选择工长
- 用户可终止项目并申请退款

**状态流转**：
- 待用户确认 → 已拒绝 → 待选择（重新选择）→ 退款中（终止）→ 已关闭

**责任人**：
- 用户：选择重新选择或终止
- 工长：提供无法承接原因（如有）
- 系统：自动触发退款（如终止）

**资金处理**：
- 意向金全额退款（如终止）

**审计留痕**：
- 拒绝原因、拒绝时间、重新选择次数、退款时间

---

### 7.3 执行中暂停/争议

**触发场景**：
- 用户要求暂停施工
- 工长要求暂停施工
- 用户与工长发生争议

**处理流程**：
- 暂停施工：冻结后续款项，等待恢复
- 争议：平台介入仲裁

**状态流转**：
- 执行中 → 暂停中 → 执行中（恢复）/ 争议中 → 仲裁中 → 已关闭

**责任人**：
- 用户/工长：提供暂停/争议原因
- 管理员：仲裁争议

**资金处理**：
- 暂停期间：冻结后续款项
- 争议期间：冻结所有款项
- 仲裁后：按仲裁结论处理

**审计留痕**：
- 暂停原因、暂停时间、恢复时间、争议内容、仲裁结论

---

### 7.4 分阶段验收不通过

**触发场景**：
- 用户验收某阶段不通过

**处理流程**：
- 工长整改
- 用户重新验收
- 验收通过后，解冻该阶段款项并放款

**状态流转**：
- 阶段验收中 → 阶段拒绝 → 执行中（整改）→ 阶段验收中 → 阶段已验收

**责任人**：
- 用户：提供验收不通过原因
- 工长：整改并重新提交

**资金处理**：
- 冻结该阶段及后续款项
- 验收通过后，解冻并放款

**审计留痕**：
- 验收不通过原因、整改记录、重新验收时间

---

### 7.5 退款

**触发场景**：
- 设计方案拒绝次数 ≥ 3
- 工长选择失败且用户终止
- 施工中途暂停且无法恢复
- 阶段验收不通过且无法整改
- 用户主动申请退款

**处理流程**：
- 用户/系统提交退款申请
- 管理员审核退款申请
- 审核通过后，系统自动退款

**状态流转**：
- 正常流程 → 退款审核中 → 退款通过 → 已退款 → 已关闭

**责任人**：
- 用户/系统：提交退款申请
- 管理员：审核退款申请
- 系统：执行退款

**资金处理**：
- 按退款规则计算退款金额
- 已放款部分不退款
- 未放款部分全额退款

**审计留痕**：
- 退款原因、退款金额、审核意见、退款时间

---

### 7.6 项目关闭

**触发场景**：
- 项目正常完成，全部款项放款完毕
- 项目异常终止，退款完成

**处理流程**：
- 系统自动关闭项目

**状态流转**：
- 已完成 / 已退款 → 已关闭

**责任人**：
- 系统：自动关闭

**资金处理**：
- 无

**审计留痕**：
- 关闭原因、关闭时间

---

### 7.7 审计介入与审计结论

**触发场景**：
- 用户与商家争议无法协商
- 资金流转异常
- 验收标准争议

**处理流程**：
- 用户/商家提交审计申请
- 管理员介入审计
- 管理员给出仲裁结论
- 系统执行仲裁结论

**状态流转**：
- 争议中 → 审计中 → 审计结论 → 已关闭 / 继续执行

**责任人**：
- 用户/商家：提交审计申请
- 管理员：审计并给出仲裁结论
- 系统：执行仲裁结论

## 10. 权限、审计与资金安全要求

### 10.1 托管/账款流转的事务性与审计

**事务性要求**：
- 所有资金流转操作必须在数据库事务中完成
- 资金流转失败时，必须回滚所有相关状态
- 资金流转成功后，必须记录审计日志

**审计要求**：
- 所有资金流转操作必须记录审计日志
- 审计日志包含：操作人、操作时间、操作类型、操作金额、操作前后状态
- 审计日志不可删除、不可修改

---

### 10.2 关键决策节点要留痕

**关键决策节点**：
- 入驻审核
- 方案确认
- 工长确认

- 放款
- 退款
- 关闭

**留痕要求**：
- 决策人、决策时间、决策结果、决策原因
- 留痕数据不可删除、不可修改

---

### 10.3 管理员的可见与可操作边界

**可见范围**：
- 全流程可见

**可操作范围**：
- 审核入驻申请

**禁止操作**：

## 12. 增项判定与需求管理机制

### 12.1 增项定义

**增项**：任何不落在主链路 9 个节点中的功能、页面、接口、状态、报表或平台能力。

### 12.2 判定口径

判定某功能是否属于增项，按以下标准：

1. 是否推动主链路成交/履约/收款/退款/关闭？
2. 是否服务于已确认角色（用户/设计师/工长/装修公司/主材商/管理员）？
3. 是否改变主状态机？

**如果以上任一条答案为"否"，则该功能属于增项。**

### 12.3 常见增项例子

- AI 免费设计
- BIM 协同查看器
- AR 试衣间
- LBS 排序算法细节
- 工地地图、直播、复杂可视化能力
- 与当前基线无直接关系的"生态型"想象功能
- 主材商成交闭环（当前主链路未包含）
- 工人打卡（当前主链路未包含）
- IM 聊天（支撑性能力，非主链路）

### 12.4 增项进入 backlog 的方式

1. 产品提出增项需求
2. 评审增项是否符合产品方向
3. 评审通过后，进入增项池
4. 增项池按优先级排序
5. 增项不作为当前开发、测试、排期和验收依据

### 12.5 未经评审不得写入基线 PRD

**严格规则**：
- 任何增项需求，未经产品评审，不得写入基线 PRD
- 任何增项需求，未经产品评审，不得进入开发排期
- 任何增项需求，未经产品评审，不得进入测试验收

### 12.6 若增项影响状态机、成交点、资金流，则必须重新评审 PRD 主链路

**触发条件**：
- 增项影响主状态机
- 增项影响成交点定义
- 增项影响资金流转规则

**处理流程**：
- 暂停增项开发
- 重新评审 PRD 主链路
- 更新 PRD 版本
- 重新评审增项是否仍为增项

- AI 免费设计
- BIM 协同查看器
- AR 试衣间
- LBS 排序算法细节
- 工地地图、直播、复杂可视化能力

**原因**：

- 不改变主状态机
- 属于增强性能力，非核心闭环

### 13.3 API/页面/流程变更如何回溯到 PRD 版本

**变更流程**：
1. 提出变更需求
2. 评审变更是否影响主链路
3. 如影响主链路，更新 PRD 版本
4. 如不影响主链路，记录为增项
5. 变更完成后，更新 API_CHANGES.md

**回溯规则**：
- 所有 API 变更必须记录在 API_CHANGES.md
- 所有 API 变更必须标注对应 PRD 版本
- 所有 API 变更必须标注是否影响主链路

### 13.4 哪些历史内容保留为背景，不再作为开发约束

**保留为背景**：
- 产品愿景
- 核心隐喻
- 目标用户
- 平台架构图

**不再作为开发约束**：
- 历史版本的功能描述
- 历史版本的接口定义
- 历史版本的状态机

## 14. 验收标准

### 14.1 业务验收标准

**主链路完整性**：
- 是否完整覆盖主链路 9 个节点？
- 是否支持两个独立成交点（设计确认、工长确认）？
- 是否支持分阶段验收放款？
- 是否支持整体验收放款？
- 是否支持退款/关闭/审计收口？

**非基线隔离**：
- 是否能把非基线需求挡在 PRD 外？
- 是否能据此判定需求是否属于增项？

**阶段策略一致性**：
- 是否明确以小程序作为当前主成交端？
- 是否在设计确认到工长确认之间提供了可解释、可对比、可背书的桥接能力？
- 是否在项目关闭后具备最小的评价、案例沉淀和转介绍入口？

---

### 14.2 技术验收标准

**接口完整性**：
- 是否所有主链路节点都有对应接口？
- 是否所有接口都有对应文档？
- 是否所有接口都有对应测试用例？

**状态机完整性**：
- 是否所有状态流转都有对应接口？
- 是否所有状态流转都有对应审计日志？
- 是否所有状态流转都有对应测试用例？

**资金安全**：
- 是否所有资金流转都在事务中完成？
- 是否所有资金流转都有审计日志？
- 是否所有资金流转都有对应测试用例？

## 15. 附录

### 15.1 名词表

| 名词 | 定义 |
|------|------|
| 基线 | 唯一开发基线，所有开发、测试、排期、验收的唯一裁剪标准 |
| 增项 | 任何不落在主链路中的功能、页面、接口、状态、报表或平台能力 |
| 成交点 A | 设计确认成交，用户确认设计方案/报价 |
| 成交点 B | 工长确认成交，用户确认工长/施工报价 |
| 主链路 | 9 个节点的业务流程 |
| 异常收口 | 退款、关闭、审计等异常处理流程 |
| 分阶段验收 | 按阶段/里程碑验收，每个阶段验收通过后放款 |
| 整体验收 | 所有阶段完成后，一次性整体验收，验收通过后一次性放款 |

---

### 15.2 历史流程对照

**v1.0 PRD（平台全景型）**：
- 包含大量非基线能力
- 未明确成交点定义
- 未明确异常收口机制

**v2.0 PRD（基线驱动型）**：
- 仅包含主链路 9 个节点
- 明确两个独立成交点
- 明确异常收口机制
- 明确增项判定规则
