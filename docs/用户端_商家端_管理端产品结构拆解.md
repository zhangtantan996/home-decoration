# 用户端 / 商家端 / 管理端产品结构拆解

> 基于当前仓库已有页面、路由、状态与文档定位整理。
> 目标不是画视觉信息架构图，而是明确三端各自承担什么业务责任，以及它们如何共同组成当前产品。

## 1. 一句话定义三端分工

- `用户端 mini`：主成交端，负责用户进入、预约转化、设计确认、施工确认、项目跟进与售后发起。
- `商家端 merchant`：主履约端，负责接单、量房、设计输出、施工承接、项目执行、纠纷协同与收入提现。
- `管理端 admin`：主治理端，负责审核、监管、资金、风控、争议仲裁、过程审计与平台级配置治理。

这三端不是三个彼此独立的产品，而是同一条家装交易与履约链路上的三个角色界面。

## 2. 整体产品结构

可以把当前产品理解为 6 层：

1. 流量与内容承接层
2. 预约与设计交易层
3. 施工桥接与正式成交层
4. 项目履约执行层
5. 售后、退款、争议治理层
6. 平台供给、资金、规则治理层

三端分别落位如下：

- `用户端`重点覆盖第 1-5 层。
- `商家端`重点覆盖第 2-5 层。
- `管理端`重点覆盖第 5-6 层，并对前面几层提供监管与控制。

## 3. 用户端产品结构

## 3.1 用户端的核心目标

用户端不是简单浏览器，而是要完成下面几件事：

- 被内容与案例吸引进入平台；
- 找到合适的设计/装修服务提供方；
- 发起预约与量房；
- 确认设计方案与设计费用；
- 进入施工方确认与订单阶段；
- 在项目履约期内查看进度、账单、合同、争议与售后入口；
- 在异常情况下发起退款、投诉或仲裁协助。

## 3.3 用户端在整个产品中的定位

用户端承担的是：

- 流量承接
- 成交触发
- 决策确认
- 项目监督感知
- 售后发起

所以用户端不是“看图平台”，而是`交易前台 + 履约查看台 + 售后入口台`。

## 4. 商家端产品结构

## 4.1 商家端的核心目标

商家端的目标不是展示商家信息，而是帮助服务提供方完成整条执行链：

- 入驻并通过审核；
- 接线索和预约；
- 完成量房与预算确认；
- 提交设计方案与报价；
- 推进施工报价与项目承接；
- 管理在建项目；
- 处理投诉、纠纷与回款；
- 沉淀案例与经营数据。

## 5. 管理端产品结构

## 5.1 管理端的核心目标

管理端不是传统“平台设置中心”，而是平台的治理操作系统。它要解决：

- 谁可以进入平台；
- 哪些交易可以推进；
- 项目是否合规；
- 资金怎么监管；
- 异常怎么处理；
- 风险怎么提前发现；
- 全局经营状态如何被看见。

- 争议裁决
- 平台规则配置

所以管理端本质上是`治理操作系统`，不是普通后台。

## 6. 三端如何协同形成完整产品

可以把这三端理解成同一个交易系统的三种视角：

- `用户端`负责提出需求、做关键确认、感知项目状态、发起异常反馈。
- `商家端`负责执行需求、提供方案、承接项目、完成交付、配合治理。
- `管理端`负责在关键节点进行审核、监管、风控、资金和争议处理。

如果画成链路，就是：

1. 用户端把流量和需求送入系统；
2. 商家端把需求变成方案、报价和履约动作；
3. 管理端保证交易可信、过程可控、异常可治理；
4. 再把案例、评价、信用、治理结果反馈回下一轮交易。

## 7. 当前产品结构所体现出的真实产品形态

从三端结构来看，你当前开发的不是：

- 单纯灵感/案例浏览产品；
- 单纯预约 CRM；
- 单纯商家获客工具；
- 单纯工程管理系统。

你实际上在做的是一个：

- 前台可成交，
- 中台可履约，
- 后台可治理，
- 资金与争议可控，
- 能把设计交易和施工履约串成一条闭环链路的家装平台。

## 8. 产品结构层面的重点提醒

如果后续继续做产品梳理或需求决策，建议始终围绕这三条主线来判断：

1. 用户端是不是在提高预约转化、设计确认率、施工确认率和项目透明度。
2. 商家端是不是在提高接单效率、提案效率、履约效率和回款效率。
3. 管理端是不是在提高审核效率、异常治理效率、资金安全和平台信任度。

只要一个需求不能明显加强这三端中的至少一端主职责，它大概率就不是当前阶段的核心需求。
