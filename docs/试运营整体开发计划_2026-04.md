## 1. 目标与判断

本阶段目标不是把所有功能做完，而是把平台收口到：

- **小程序主成交**
- **Merchant Web 主履约**
- **Admin Web 主治理**
- **Web/H5 辅助承接**

试运营完成标准不是“页面齐全”，而是以下 3 件事同时成立：

1. 用户能在小程序完成预约、设计确认、工长确认到项目创建
2. 商家能在 Merchant Web 完成提案、施工报价、项目执行与提现申请
3. Admin 能完成审核、退款、提现、争议与最小经营判断

## 2. 端面职责冻结

### 2.1 当前端面定位

- **小程序**：主获客、主预约、主设计成交、施工桥接前台、微信支付主入口
- **Merchant Web**：线索承接、量房预算、提案与重提、施工报价、项目执行、提现申请、通知中心
- **Admin Web**：审核、订单/项目监控、退款/争议/关闭、提现审核与打款确认、系统阈值配置、最小经营看板
- **Web/H5**：活动落地、分享回流、支付结果页、辅助浏览页

### 2.2 当前不做的事项

本阶段明确不做：

- 支付双通道并行
- 主材商城主线升级
- 商家 package 级拆分
- 大而全运营系统
- 社区、积分、会员、AI 设计、BIM、工地直播

## 3. 分阶段实施计划

### Phase 1：基线冻结与 P0 硬阻断清理（Week 1-2）

#### 目标
统一当前产品真相，清掉试运营前的基础风险。

#### 核心事项
- 统一状态命名、成交点定义、项目创建触发条件，禁止继续混用旧枚举与展示文案状态
- 真实短信验证码替换 mock
- 提现并发安全加锁
- 商家端数据权限复查
- 图片上传接入真实 OSS
- 核心 PII 字段加密链路复查

#### Done when
- 短信、上传、权限、提现并发、PII 五项阻断全部有明确实现与验证
- 端面职责和文档基线在开发侧无冲突

### Phase 2：小程序主成交链补齐（Week 2-5）

#### 目标
把小程序补到“可完成主成交链”的程度。

#### 核心事项
- 补齐 P0 页面与行为：
  - 服务商列表/详情
  - 预约创建/列表/详情
  - 量房/预算确认结果查看
  - 设计方案确认/拒绝
  - 工长对比/施工报价确认
  - 订单详情、项目详情、进度查看
  - 验收入口、退款入口、支付结果页
- 首页强化价值表达和预约导流
- 详情页强化信任转换
- 预约页收敛到最小必要留资
- 设计确认后必须进入施工桥接漏斗，不能停在静态确认页
- Web/H5 收口为承接页、分享页、支付结果页与辅助浏览页

#### Done when
- 用户可以在小程序完整走到“工长确认前”
- 设计确认后的页面路径不再断在静态详情

### Phase 3：服务端交易闭环硬化（Week 3-7）

#### 目标
把已有交易主链补成稳定闭环，而不是继续堆旁路能力。

#### 核心事项
- 统一收口 `Booking / Demand -> Proposal -> Foreman Confirm -> Order -> Project`
- 方案版本管理与拒绝重提
- 工长确认对象与施工桥接读模型
- 项目开工闸门
- 里程碑与支付计划/结算强绑定
- 完工后的评价、案例沉淀、履约标签写入
- 补齐主链自动化：
  - 商家响应超时处理
  - 用户确认超时处理
  - 主链关键节点通知

#### Done when
- 主链关键状态推进不再依赖前端拼装
- 设计拒绝、工长重选、超时、退款、关闭都有服务端收口

### Phase 4：微信小程序支付 + Merchant/Admin 主工作面（Week 4-8）

#### 目标
让试运营能真实收款、商家能真实履约、平台能真实治理。

#### 核心事项
- 只做微信小程序支付：
  - 下单
  - 回调
  - 支付查询
  - 关闭超时订单
  - 退款与退款查询
- 支付覆盖范围只限主链金额节点：
  - 预约/意向金
  - 设计或施工订单付款
  - 里程碑相关支付与退款
- Merchant Web 补齐主工作面：
  - 预约/线索承接
  - 量房与预算记录
  - 方案提交、版本重提、拒绝原因回看
  - 工长施工报价提交
  - 项目开工、阶段提交、收入查看
  - 提现申请与通知中心
- Admin Web 补齐主治理面：
  - 商家审核
  - 订单/项目/阶段监控
  - 提现人工审核与打款确认
  - 退款/争议/关闭处理
  - 系统阈值配置

#### Done when
- 微信支付主链可联调
- 商家和 Admin 在各自工作面内能完成核心动作

### Phase 5：运营可见性与试运营验收（Week 6-12）

#### 目标
形成最小经营可见性，并完成试运营 Go/No-Go 验收。

#### 核心事项
- 只做最小经营数据底座：
  - 原始事件
  - 每日聚合指标
  - Admin 最小看板
- 看板只覆盖 4 类经营判断：
  - 拉新与预约
  - 设计成交与施工桥接
  - 商家供给质量
  - 售后治理与异常积压
- 完成试运营验收：
  - 小程序核心链路联调
  - Merchant/Admin SOP 演练
  - 微信支付联调证据归档
  - 退款、提现、争议、超时任务回归
  - Go/No-Go 清单

#### Done when
- 运营能按日看到预约、设计确认、工长确认、完工、退款/争议
- 试运营联调证据和验收清单完整

## 4. 公开接口与类型边界

### 4.1 允许新增或重构的公开接口

只允许围绕主链闭环新增或重构接口：

- 小程序主链读模型接口：
  - 服务商详情信任字段
  - 方案详情/历史
  - 工长对比
  - 项目进度摘要
- 主链动作接口：
  - 方案重提
  - 工长确认
  - 项目开工
  - 阶段提交/验收
  - 评价提交
  - 案例分享入口
- 支付接口：
  - 微信小程序下单
  - 支付状态查询
  - 回调
  - 关闭订单
  - 退款与退款查询
- 通知接口：
  - 列表
  - 未读数
  - 已读 / 全部已读
- Analytics 最小接口：
  - 事件上报
  - Admin 概览/转化/供给/治理读取

### 4.2 类型和状态要求

- 继续沿用 PRD / BUSINESS_FLOW 现有业务对象命名
- 不新增新的主链实体概念
- 统一使用以下状态口径：
  - 双成交点分离
  - 工长确认才创建项目
  - 主材商不进入主交易闭环
- 评价、案例沉淀、履约标签属于支撑性增长能力，不新增主链节点

## 5. 测试与验收

### 5.1 主链业务回归

- 用户从预约到设计确认到工长确认到项目创建完整跑通
- 设计拒绝重提、工长拒绝重选、退款、关闭、争议路径可收口
- 项目开工、阶段提交、阶段验收、放款/退款状态一致

### 5.2 支付联调

- 微信小程序支付下单、回调、查询、关闭、退款全链路通过
- 重复回调、超时订单、退款中间态具备幂等和补偿能力

### 5.3 多端一致性

- 小程序是主成交端
- Merchant Web 能承接提案与履约
- Admin Web 能承接治理与审核
- Web/H5 不再承担主成交职责

### 5.4 运营与治理

- 提现审核不再“审核即成功”
- 通知中心覆盖主链关键节点
- 最小经营看板能看预约、设计确认、工长确认、完工、退款/争议

### 5.5 安全与稳定性

- 短信、权限、上传、提现并发、PII 加密、支付回调验签都有专项验证
