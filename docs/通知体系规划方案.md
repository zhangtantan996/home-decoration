# 通知体系规划方案（业务流全景版）

> 文档目的：将当前家装平台业务流涉及的通知类能力一次性梳理清楚，作为后续 AI/研发实施的统一输入。
> 
> 适用对象：产品、后端、前端、测试、运营、其他接手执行的 AI。
> 
> 当前结论：项目**已有站内通知基础设施**，但**缺少完整通知产品定义与全链路通知矩阵**。后续工作不应从零重做通知系统，而应基于现有能力补齐“事件 → 模板 → 渠道 → 回执 → 升级”体系。

---

## 1. 目标与范围

### 1.1 目标

围绕当前唯一业务主链，形成完整通知体系，明确：
- 哪些业务事件需要通知
- 谁接收通知
- 通知目标是什么
- 走什么渠道
- 是否需要确认/回执
- 超时后是否催办/升级
- 哪些可以复用现有站内通知能力
- 哪些能力必须新增

### 1.2 业务主链范围

本方案只围绕以下主链，不扩展增项功能：

1. 商家入驻与能力配置  
2. 用户需求进入线索/预约  
3. 量房、预算确认、设计沟通  
4. 设计费报价、支付、正式方案确认（成交点 A）  
5. 工长选择、施工报价确认（成交点 B）  
6. 订单生成与项目创建  
7. 项目按阶段/里程碑执行  
8. 验收、放款、退款、关闭  
9. 争议、审计、仲裁治理  

参考文档：
- `docs/BUSINESS_FLOW.md`
- `docs/产品需求文档(PRD).md`

## 2. 现状判断

### 2.1 已有能力（不要推倒重来）

当前仓库已经有站内通知体系骨架：

#### 后端
- `server/internal/service/notification_service.go`
  - 通知创建入口
  - Admin 广播通知
  - 用户侧通知分类过滤（system / project / payment）
- `server/internal/service/notification_dispatcher.go`
  - 已覆盖 booking / proposal / quote / milestone / completion / payment / change_order 等分发逻辑
- `server/internal/service/notification_publisher.go`
  - 已支持实时推送：new / read / delete / all_read / unread_count
- `server/internal/handler/notification_handler.go`
  - 已支持通知列表、未读数、单条已读、全部已读、删除
- `server/internal/model/model.go`
  - 已有 Notification 模型
  - 已有 UserSettings 中的通知偏好字段

#### 前端
- `admin/src/components/NotificationDropdown.tsx`
  - Admin 通知下拉
  - 类型标签映射
  - WebSocket + polling fallback
- `merchant/src/pages/merchant/MerchantNotifications.tsx`
  - Merchant 通知中心
- `admin/src/services/api.ts`
  - Admin 通知 API 封装
- `merchant/src/services/merchantApi.ts`
  - Merchant 通知 API 封装

### 2.2 当前缺口

当前最大缺口不是“通知入口没有”，而是：

1. 没有完整的业务通知矩阵
2. 没有统一模板/策略层
3. 没有把“已读”和“已处理”分开
4. 没有超时催办与升级机制
5. 用户偏好有了，但 merchant/provider/admin 没统一
6. 目前主要是站内 + 实时，短信/小程序订阅消息/Push/邮件没有体系化

## 3. 推荐通知产品模型

通知统一分成 5 层：

### 3.1 业务通知
用于告知状态变化。

示例：
- 预约已承接
- 方案已提交
- 项目已开工
- 已进入施工桥接阶段

### 3.2 待办通知
接收方必须动作。

示例：
- 预算待确认
- 设计费待支付
- 正式方案待确认
- 施工报价待确认
- 阶段验收待处理
- 退款待审核

### 3.3 风险 / 催办通知
用于提醒即将超时或流程阻塞。

示例：
- 补件超时
- 预算确认超时
- 付款即将过期
- 付款已过期
- 通知待处理超时

### 3.4 结果通知
用于告知结果已经明确。

示例：
- 支付成功 / 失败
- 验收通过 / 驳回
- 退款成功
- 仲裁结论已出

### 3.5 治理通知
给 Admin 的待介入消息，不属于普通消息流。

示例：
- 新入驻待审核
- 退款申请待审核
- 争议待审计
- 资金异常待处理
- 超时升级待干预

---

## 4. 角色视角

### 4.1 用户 / 业主
目标：推动成交、确认关键节点、完成支付与验收。

### 4.2 商家 / 设计师 / 工长 / 装修公司
目标：接单、报价、交付、整改、跟进回款。

### 4.3 管理员 / Admin
目标：审核、治理、仲裁、处理异常与高风险事件。

## 5. 通知矩阵（按业务流分解）

以下矩阵是本方案核心。

---

## 5.1 商家入驻与能力配置

### 事件：商家提交入驻申请
- 接收角色：Admin
- 通知目标：提醒有新入驻待审核
- 建议渠道：站内 + 实时
- 是否需要确认/回执：需要（审核处理）
- 优先级：高
- 超时升级：可选，超过 SLA 自动提醒管理员

### 事件：审核通过
- 接收角色：Merchant
- 通知目标：告知可进入接单池 / 继续能力配置
- 建议渠道：站内 + 实时
- 是否需要确认/回执：不强制
- 优先级：高

### 事件：审核拒绝
- 接收角色：Merchant
- 通知目标：提示拒绝原因与补件入口
- 建议渠道：站内 + 实时 + 短信
- 是否需要确认/回执：需要（是否补件/重提）
- 优先级：高

### 事件：补件 / 重提超时
- 接收角色：Merchant + Admin
- 通知目标：催办 / 治理跟进
- 建议渠道：站内 + 实时
- 是否需要确认/回执：Admin 可处理
- 优先级：中高

---

## 5.2 线索 / 预约阶段

### 事件：预约创建 / 线索分发

- 通知目标：提醒新单待承接

### 事件：商家拒绝预约

### 事件：量房费支付成功 / 失败 / 超时

- 通知目标：推动进入后续流程 / 提示风险
- 建议渠道：站内 + 实时；失败/超时建议补短信

## 6. 哪些通知必须区分“已读��和“已处理”

这是后续实现必须明确的原则。

### 6.1 只需要“已读”
以下通知看过即可，不要求业务动作：
- 商家已接单
- 项目已创建
- 已开工
- 支付成功
- 退款成功
- 已结算
- 项目已关闭

### 6.2 必须“已处理”
以下通知不是点开就结束，必须有业务回执：
- 预算确认
- 设计费报价确认
- 正式方案确认 / 拒绝
- 施工报价确认 / 拒绝
- 阶段验收通过 / 驳回
- 整体验收通过 / 驳回
- 退款审核
- 审计介入处理
- 补件提交

## 7. 渠道策略建议

### 7.1 当前阶段主通道
建议先以以下两类为主：
- 站内通知
- 实时通知（WebSocket / Realtime）

原因：
- 当前代码已有基础设施
- 风险最小
- 实施成本低
- 能最快覆盖主链路

### 7.2 第二阶段扩展通道
只给高优先级事件补外部通道：
- 短信
- 小程序订阅消息
- APP Push
- 邮件（可选）

### 7.3 建议必须补外部通道的事件
- 入驻审核拒绝 / 补件
- 支付失败 / 超时
- 验收待确认
- 退款审核结果
- 争议 / 审计 / 仲裁结论

- 内容模板
- 类别
- 优先级
- 渠道策略
- 是否必须回执
- 超时升级规则

### B. 回执语义
新增“已处理”概念，不再把“已读”当业务完成。

### C. 超时催办与升级
- T+N 未处理时自动催办
- 严重场景升级给 Admin
- 升级过程留痕

### D. 角色级偏好中心
不能只让 user 有通知偏好，merchant/provider/admin 也要纳入。

### E. 治理通知视图
Admin 侧区分：
- 普通通知
- 待处理治理通知

## 11. 给后续 AI / 研发的执行提示

后续接手方不要直接“再造通知系统”，应遵守以下原则：

1. **优先复用现有通知 Service / Dispatcher / Publisher / Handler**
2. **先固化通知字典，再改代码**
3. **先打通站内 + 实时，再扩短信 / 订阅消息 / Push**
4. **把“已读”和“已处理”彻底分开**
5. **治理通知要和普通消息流分层**
6. **所有升级/催办动作都要审计留痕**

## 12. 核心结论

一句话总结：

> 当前项目不缺“通知入口”，缺的是“完整通知产品定义”。

真正要补的是：
- 全业务通知矩阵
- 模板 / 策略层
- 回执语义
- 超时催办 / 升级
- 多角色偏好
- 治理通知分层

只有这些补齐，通知系统才会真正服务成交、履约和治理，而不是继续停留在“有个通知列表”的层面。
