# 版本管理与发布回滚指南 (Release & Rollback Guide)

本文档旨在为非运维专家提供一套简单、安全、可执行的代码更新与现网回滚流程。借鉴了大厂的核心思想，但针对本项目进行了简化，适合中小型项目的单机 Docker 部署。

---

## 1. 核心原则 (Core Principles)

1.  **永远不要覆盖**：发布的每一个版本都应有独立的标记（版本号），而不是简单地覆盖旧文件。
2.  **永远留后路**：在执行更新命令前，必须确信自己能在 1 分钟内恢复到上一个状态。
3.  **环境隔离**：开发环境（本地）随意折腾，生产环境（服务器）必须谨慎。

---

## 2. 版本管理 (Version Control)

我们使用 **Git Tag** 和 **Docker Tag** 来标记版本。

### 2.1 命名规范
采用 `v主版本.次版本.补丁` 的格式，例如：
*   `v1.0.0`: 初始上线
*   `v1.0.1`: 修复了一个小 Bug
*   `v1.1.0`: 增加了一个新功能

### 2.2 发布流程 (Standard Operating Procedure)

假设你现在修完了一个 Bug，准备发布 `v1.0.1`。

#### 第一步：本地打标 (Local)
1.  确认代码已提交并推送到远程仓库。
    ```bash
    git add .
    git commit -m "fix: 修复主材页面刷新问题"
    git push
    ```
2.  给当前代码打上标签（相当于按下快照）。
    ```bash
    git tag v1.0.1
    git push origin v1.0.1
    ```

#### 第二步：服务器更新 (Server)
登录到你的 Linux 服务器，执行以下操作：

1.  **更新代码目录**
    ```bash
    cd home_decoration
    git fetch --tags
    git checkout v1.0.1  # 切换到刚才打的标签，确保代码绝对一致
    ```

2.  **构建并启动新版本 (关键)**
    不要直接覆盖 `latest` 标签，而是显式指定版本构建镜像。
    *修改 `docker-compose.prod.yml` (可选，或者直接用环境变量)*，或者简单地使用 Docker 的构建命令：
    
    ```bash
    # 进入部署目录
    cd deploy
    
    # 停止旧服务 (为了安全，简单粗暴先停后起)
    docker-compose -f docker-compose.prod.yml down
    
    # 构建新镜像，这里建议手动 tag (高级做法)，
    # 但为了简单，我们利用 Docker 的缓存机制，
    # 只要代码变了，重新 build 就会生成新层。
    # 重点：Docker 默认虽覆盖 latest，但我们可以手动备份旧镜像。
    
    # 【简易版安全更新法】
    docker-compose -f docker-compose.prod.yml up -d --build
    ```

3.  **验证**
    观察 5-10 分钟日志：
    ```bash
    docker-compose -f docker-compose.prod.yml logs -f --tail=100
    ```

## 3. 现网回滚 (Rollback Strategy)

**场景**：你刚发布的 `v1.0.1` 上线后，用户反馈 App 打不开了，或者报错了。你需要立即回滚到昨天的 `v1.0.0`。

### 方案 A：Git 回滚法 (最简单，适用本项目)

因为我们是基于源代码构建的，回滚代码 = 回滚服务。

1.  **切回旧版本代码**
    ```bash
    # 在服务器上
    cd home_decoration
    git checkout v1.0.0  # 刚才的旧版本 tag
    ```

2.  **重新启动**
    ```bash
    cd deploy
    docker-compose -f docker-compose.prod.yml up -d --build
    ```
    *原理：Docker 发现代码变回去了，会利用之前的缓存快速重建（或者如果你没清空镜像，它甚至不需要重新编译）。*

### 方案 B：Docker 镜像回滚法 (更专业，秒级恢复)

这需要你在每次发布时，都保留旧的 Docker 镜像。

1.  **发布时**：构建带版本号的镜像
    ```bash
    docker build -t my-api:v1.0.1 -f Dockerfile.backend .
    ```
2.  **修改配置**：在 `docker-compose.prod.yml` 里把 `image: my-api:latest` 改成 `image: my-api:v1.0.1`。
3.  **回滚时**：
    *   把 `docker-compose.prod.yml` 里的版本号改回 `v1.0.0`。
    *   执行 `docker-compose up -d`。
    *   *耗时：1-2 秒*。

---

## 4. 总结：你的 "防手抖" 清单

每次更新服务器前，心里默念这份清单：

1.  [ ] **本地 Git 是否干净？** (代码都提交了吗？)
2.  [ ] **打 Tag 了吗？** (v1.x.x)
3.  [ ] **服务器上 Git Checkout 到 Tag 了吗？** (确保服务器代码 = 本地代码)
4.  [ ] **更新后看日志了吗？** (不要跑完命令就去睡觉)
5.  [ ] **如果炸了，知道怎么切回上个 Tag 吗？** (`git checkout v_old` -> `docker-compose up`)

只要遵守这 5 点，你的发布就是安全的。

# 附录：正式上线前检查清单 (Pre-release Checklist)

> ⚠️ **重要提醒**：正式版本上线前，必须执行以下数据清理步骤！

## 1. 测试数据清理

在开发过程中，数据库中可能残留了带有 `[TEST]` 标记的测试数据，必须在正式上线前清理。

### 清理脚本
```bash
# 1. 备份数据库（重要！）
docker exec decorating_db pg_dump -U postgres home_decoration > backup_before_cleanup.sql

# 2. 执行清理脚本
docker exec -i decorating_db psql -U postgres -d home_decoration < server/scripts/cleanup_test_data.sql

# 3. 验证清理结果 (应返回 0)
docker exec decorating_db psql -U postgres -d home_decoration -c "SELECT COUNT(*) FROM users WHERE nickname LIKE '%[TEST]%';"
```

## 2. 环境配置检查
- [ ] **Android Manifest**: 是否开启了 `usesCleartextTraffic` (如果用 HTTP)？权限是否齐全？
- [ ] **API Base URL**: `mobile/src/services/api.ts` 是否已指向生产环境 IP/域名？
- [ ] **Secret Keys**: 生产环境的数据库密码、JWT Secret 是否已修改为强密码（非默认）？
