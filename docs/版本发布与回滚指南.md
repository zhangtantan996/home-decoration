# 版本发布与回滚指南（Prod / Test）

本文档定义当前仓库推荐的**生产与测试发布与回滚标准动作**。

目标是让日常发版满足三件事：

- **方便**：步骤少，优先复用现有脚本和 Compose
- **安全**：上线前有备份、上线后有验证
- **可回滚**：代码可快速切回稳定 Tag，数据库回滚单独决策

> 部署总入口：`deploy/README.md`

---

## 1. 核心原则

1. **production / test 只部署 Git Tag**
2. **发布前必须先备份**
3. **默认按服务最小化更新**
4. **代码回滚和数据库回滚分离处理**
5. **每次发布必须留有发布记录**
6. **test 与 production 必须完全隔离资源与入口**

---

## 2. Tag 规范

### 2.1 推荐命名

采用语义化版本：

- `v1.0.0`
- `v1.0.1`
- `v1.1.0`

含义：

- 主版本：有较大不兼容变化
- 次版本：新增功能
- 补丁版本：问题修复、小变更

### 2.2 打 Tag 要求

发布前应先在本地完成：

```bash
git tag v1.2.3
git push origin v1.2.3
```

要求：

- Tag 必须对应已经审查和验证的代码
- 不要在服务器上临时打未经团队确认的发布 tag
- 每个生产发布都应有唯一 tag

---

## 3. 标准发布流程

### 3.1 本地准备

1. 完成开发与测试
2. 合并到发布基线
3. 打发布 tag
4. 推送 tag 到远程

### 3.2 服务器发布

在服务器仓库根目录：

Production：

```bash
git fetch --tags --prune
bash deploy/scripts/deploy_prod.sh --tag v1.2.3 --service api
```

Test：

```bash
git fetch --tags --prune
bash deploy/scripts/deploy_test.sh --tag v1.2.3 --service api
```

可选服务范围：

- `api`
- `web`
- `all`

### 3.3 为什么不是直接 pull 分支

不推荐：

```bash
git pull origin dev
git pull origin main
```

原因：

- 生产上线内容会随分支移动而漂移
- 无法清晰回答“线上此刻到底是什么版本”
- 回滚时不如 tag 精确

---

## 4. 发布矩阵

| 场景 | 推荐动作 | 备注 |
|---|---|---|
| 仅后端改动 | 发布 `api` | 风险最小 |
| 仅前端 / website / admin / nginx 改动 | 发布 `web` | 需额外验证 website + admin |
| 前后端都改 | 发布 `all` | 做完整验证 |
| 环境变量仅运行时变化 | restart / recreate 对应服务 | 不一定需要重新 build |
| 环境变量影响构建结果 | rebuild 对应服务 | 例如前端构建注入 |
| 含数据库 schema 变更 | 先执行迁移，再发布服务 | 数据库步骤单独记录 |

---

## 5. 发布前备份规则

### 5.1 必做

每次生产发布前至少执行数据库备份。

### 5.2 推荐默认动作

为了降低漏项概率，建议每次发布同时执行：

- 数据库备份
- uploads 目录备份

当前可复用脚本：

- `deploy/scripts/backup_postgres.sh`
- `deploy/scripts/backup_uploads.sh`
- `deploy/scripts/backup_and_sync_oss.sh`
- `deploy/scripts/oss_sync_backups.sh`

### 5.3 备份记录建议

建议在发布记录中至少记录：

- 备份时间
- 备份文件名
- 备份执行人
- 是否已同步 OSS

---

## 6. 回滚策略

### 6.1 代码回滚

代码回滚标准动作：

1. 切回上一个稳定 tag
2. 只重建受影响服务
3. 执行最小验证

Production 示例：

```bash
bash deploy/scripts/rollback_prod.sh --tag v1.2.2 --service api
```

Test 示例：

```bash
bash deploy/scripts/rollback_test.sh --tag v1.2.2 --service api
```

### 6.2 数据库回滚

数据库回滚不建议自动化执行。必须单独判断：

- 是否已经执行迁移
- 是否已有新数据写入新结构
- 执行 `*_down.sql` 是否会导致数据丢失
- 是否更适合从备份恢复

### 6.3 分离策略

| 问题类型 | 建议动作 |
|---|---|
| 代码逻辑 bug | 只回滚代码 |
| 页面资源 / Nginx 配置错误 | 只回滚 `web` |
| 新后端逻辑与旧 schema 不兼容 | 先回滚代码，再评估数据库 |
| 数据迁移本身有问题 | 先止损，再人工决定是否执行 down SQL 或备份恢复 |

---

## 7. 回滚决策树

### 7.1 快速判断

1. **服务是否启动失败？**
   - 是：先看日志，优先回滚对应服务代码
2. **API 正常但页面异常？**
   - 优先回滚 `web`
3. **数据库迁移已执行吗？**
   - 否：直接代码回滚
   - 是：继续判断迁移是否向前兼容
4. **迁移回滚是否会造成数据丢失？**
   - 是：不要自动 down，先评估备份恢复或热修

### 7.2 最简单的判断原则

- **先回滚代码，后判断数据库**
- **先恢复服务，再处理数据结构**
- **能不动数据库就先不动数据库**

## 8. 发布记录模板

建议每次发布都保存以下记录：

```markdown
## Release Record

- Release tag: v1.2.3
- Previous stable tag: v1.2.2
- Service scope: api / web / all
- Operator: <name>
- Release time: YYYY-MM-DD HH:mm

### Backups
- Database: <backup file>
- Uploads: <backup file>
- OSS sync: yes / no

### Database migrations
- Applied: yes / no
- Migration files:
  - server/migrations/vX.Y.Z_description.sql
- Historical/helper scripts noted separately: yes / no
- Verify SQL executed: yes / no
- Rollback SQL prepared: yes / no

### Post-release verification
- docker compose ps: pass / fail
- api health: pass / fail
- website check: pass / fail
- admin check: pass / fail

### Rollback plan
- Rollback tag: v1.2.2
- DB rollback needed: unknown / no / yes
- Notes: <notes>
```

## 9. 最小发布命令与最小回滚命令

### 9.1 发布

```bash
bash deploy/scripts/deploy_prod.sh --tag v1.2.3 --service api
```

### 9.2 回滚

```bash
bash deploy/scripts/rollback_prod.sh --tag v1.2.2 --service api
```

### 9.3 发布后检查

```bash
docker compose -f deploy/docker-compose.prod.yml ps
docker compose -f deploy/docker-compose.prod.yml logs --tail=100 api
docker compose -f deploy/docker-compose.prod.yml logs --tail=100 web
curl -fsS http://127.0.0.1:8888/api/v1/health
```

## 10. 高风险场景提醒

以下场景发布后必须扩大验证：

- 修改 `deploy/nginx/nginx.conf`
- 修改 website 静态资源构建逻辑
- 修改 admin 构建逻辑
- 修改上传资源访问路径
- 修改数据库 schema
- 修改生产环境变量

特别注意：

- `web` 发布不仅影响后台，也可能影响 website 和统一入口代理
- 数据库回滚不应由脚本无条件自动执行
- 如遇 schema 变更失败，优先人工止损而不是连续重试
