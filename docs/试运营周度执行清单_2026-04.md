## 1. 使用方式

本清单是试运营整体计划的执行拆解版，按周定义：

- 该周必须完成什么
- 该周由哪个端面主负责
- 该周结束时要交付什么
- 该周的验收口径是什么

默认以 8-12 周窗口执行；如果资源紧张，优先保证每周的 `Must` 项，其次才是 `Should` 项。

## 2. Week 1-2：基线冻结与 P0 阻断清理

### Must
- 服务端统一主链状态命名、成交点定义、项目创建触发条件
- 短信验证码替换 mock
- 提现并发安全加锁
- 商家端数据权限复查
- 图片上传接真实 OSS
- 核心 PII 字段加密链路复查

### Should
- 补齐对应安全/权限/上传回归用例
- 梳理各端仍在使用的旧状态字段和兼容入口

### 主要责任面
- `server`
- `merchant`
- `admin`

### 周末交付物
- 可验证的短信发送与校验链
- 提现并发安全实现与测试
- OSS 上传全链路可用
- 数据权限复查清单与修正结果
- PII 加密字段清单与验证结论

### 验收口径
- 试运营 5 个基础阻断项全部关闭
- 开发与测试对“当前主链状态口径”无歧义

---

## 3. Week 2-3：小程序预约与服务商信任链

### Must
- 小程序服务商列表/详情可用
- 首页价值表达与预约导流收口
- 详情页补齐案例、标签、预算/风格等信任字段
- 预约创建/列表/详情可用
- 预约页收敛为最小必要留资

### Should
- 小程序分享回流页与 H5 承接页打通
- 小程序端关键空态、错误态、加载态补齐

### 主要责任面
- `mini`
- `server`
- `web`

### 周末交付物
- 小程序可从首页进入服务商详情并完成预约提交
- 服务商详情具备基础信任转换能力
- 预约相关接口和页面回归通过

### 验收口径
- 用户能在小程序完成“进入 -> 详情 -> 预约”闭环
- 预约前不再依赖 Web 主站路径

---

## 4. Week 3-4：设计成交链与方案版本管理

### Must
- 小程序支持量房/预算确认结果查看
- 小程序支持方案详情、确认、拒绝
- 服务端支持方案版本管理与拒绝重提
- 拒绝原因回看、版本历史查询打通
- 主链关键节点通知第一版可用

### Should
- 用户侧补充方案价值说明与下一步引导文案
- 商家端补充重提后的时间线展示

### 主要责任面
- `server`
- `mini`
- `merchant`

### 周末交付物
- 方案确认/拒绝/重提闭环
- 版本历史接口与页面消费
- 提案提交、拒绝、重提通知可用

### 验收口径
- 用户可完成“预约 -> 方案确认/拒绝”完整动作
- 方案被拒后商家能按版本重新提交，不丢历史记录

---

## 5. Week 4-5：施工桥接漏斗与工长确认

### Must
- 服务端补齐工长确认对象与施工桥接读模型
- 小程序支持工长列表、工长对比、施工报价确认
- 工长详情页明确工期、预算、责任边界与风险规则
- 设计确认后页面流必须进入施工桥接，不允许停在静态详情

### Should
- 加入真实案例与评价背书的桥接模块
- 加入工长筛选与推荐标签

### 主要责任面
- `server`
- `mini`
- `merchant`

### 周末交付物
- 工长对比页
- 施工报价确认页
- 桥接读模型接口

### 验收口径
- 用户可完成“设计确认 -> 工长确认 -> 项目创建前准备”闭环
- 施工桥接页承担成交作用，而不是纯展示作用

## 6. Week 5-6：项目创建、开工闸门与 Merchant 主工作面一期

### Must
- 服务端补齐项目开工闸门
- 统一 `Booking / Demand -> Proposal -> Foreman Confirm -> Order -> Project` 收口
- Merchant Web 可承接：
  - 预约/线索处理
  - 量房与预算记录
  - 方案提交/重提
  - 工长施工报价提交
- 小程序支持订单详情、项目详情、进度摘要

### Should
- Merchant 工作台补齐主链待办视图
- Admin 增加项目创建与开工监控摘要

### 主要责任面
- `server`
- `merchant`
- `mini`
- `admin`

### 周末交付物
- 项目创建与开工链路
- Merchant 主工作面一期
- 小程序订单/项目详情页

### 验收口径
- 工长确认后才创建项目
- 项目创建后必须显式开工，不能再默认进入执行态

## 7. Week 6-8：微信小程序支付与退款闭环

### Must

- 支付回调验签与幂等
- 支付状态查询
- 超时订单关闭
- 退款与退款查询
- 支付结果页与小程序主链对齐

### Should
- 支付失败、确认中、关闭态文案与引导统一
- 支付与业务状态补偿任务落地

### 主要责任面
- `server`
- `mini`
- `web`

### 周末交付物
- 微信小程序支付主链联调通过
- 退款链路可用
- 支付结果页/H5 承接可用

### 验收口径
- 小程序支付下单、回调、查询、关闭、退款都能跑通
- 重复回调与中间态不造成重复入账或重复退款

### Should
- 阶段验收结果通知补齐
- 争议与关闭流程联动项目/资金状态

### 主要责任面
- `server`
- `merchant`
- `mini`
- `admin`

### 周末交付物
- 阶段提交/验收闭环
- 提现人工审核闭环
- 退款/争议对项目状态影响可见

### 验收口径
- 项目执行、阶段验收、放款/退款状态一致
- 提现审核、打款确认、收入状态同步正确

## 11. 排期使用规则

### 11.1 默认优先级
- 先 `Must`
- 再 `Should`
- 若资源冲突，优先保交易闭环、支付闭环、治理闭环，不保美化项或旁路项

### 11.2 需求插入规则
新增需求插入本清单前，必须先回答：

1. 是不是当前阶段刚需
2. 是否真正帮助增长、成交、履约或治理
3. 是否存在更短、更干净的实现路径

### 11.3 不允许的偏移
本阶段不允许把资源转去：

- 主材商城主线
- 大而全 BI
- 社区/会员/积分
- 非主链大改版 UI
- 多支付通道并行
