# 管理后台会话与登录安全审计建议

## 当前结论

- 当前后台会话仍以 `admin_token + localStorage` 为主。
- admin / merchant / user 三套 token 已基于路由前缀做隔离，但仍需依赖前端代码纪律，不能视为强安全边界。
- 本轮不迁移 `HttpOnly Cookie`，优先完成：统一 401/403 处理、状态清理、权限收口、风险说明。

## 当前剩余风险

- `localStorage` 方案仍受 XSS 影响，token 被读取后可被复用。
- `admin_user / admin_permissions / admin_menus` 继续持久化会放大“共享设备 / 残留会话 / 状态不一致”风险。
- 若组件自行处理 `401/403`，容易再次出现遮罩残留、循环跳转、状态错乱。

## 本轮治理基线

- 后台 `401` 只由统一入口处理：清理后台 store 与持久化态，再跳转登录页。
- 后台 `403` 不登出，只保留当前页并提示无权限。
- 后台组件不得自行处理登录失效，不得自行跳转 `/admin/login`。
- admin / merchant / user token 仅读取各自作用域的存储键。
- 退出登录必须清空：token、用户信息、权限、菜单、轮询态、通知态、页面级鉴权缓存态。

## 后续建议（不在本轮实施）

### 会话介质
- 迁移到 `HttpOnly + Secure + SameSite` Cookie
- 配套服务端 refresh / rotation / revoke 机制
- 为后台单独设置更短会话 TTL 与更严格的空闲超时

### 管理员登录安全
- 增加图形验证码或挑战验证码
- 为高权限账号启用 OTP / 2FA
- 增加异常登录提醒（新设备 / 新地域 / 非常规时间）
- 超级管理员账号数量最小化，并与普通运营账号隔离

### 密码策略
- 最少长度、复杂度、历史密码限制
- 首次登录强制改密
- 长期未改密提醒与高权限账号定期轮换
