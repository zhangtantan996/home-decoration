# 报价 ERP 与装修主链路整合方案（v2）

> 文档版本：v2.0  
> 文档状态：对齐稿  
> 适用范围：小程序、Merchant Web、Admin Web、后端主链路与报价 ERP 整合  
> 基线依据：`docs/产品需求文档(PRD).md`、`docs/BUSINESS_FLOW.md`、`AGENTS.md`

---

## 1. 背景

当前仓库中的报价系统已经具备 v1 能力基础，包括标准项库、模板、价格库、报价任务、工长报价、版本留痕与一定程度的业务流联动。  
但现阶段的核心问题已经不是“有没有报价系统”，而是：

1. 报价能力还没有被完整收束进当前装修主链；
2. 设计确认到工长确认之间仍缺少稳定桥接层；
3. 工长确认之后的支付、验收、结算、治理还没有与报价结果强绑定；
4. 一些体现平台特色的能力被拆得过后，容易导致主链上线后仍然“能报价、不能控履约”。

因此，v2 的目标不是把现有报价系统扩成大而全工程 ERP，而是把它重构为当前主链中的施工成交桥接与履约约束底座。

---

## 2. 方案目标

本方案聚焦五个目标：

1. **对齐双成交点主链**  
   明确设计确认与工长确认是两个独立成交点，不能合并。

2. **打通设计到施工的桥接闭环**  
   让设计交付、工程量基础表、工长选择、施工报价、用户确认成为一条稳定链路。

3. **把报价确认真正变成履约起点**  
   用户确认施工报价后，直接进入订单、项目、支付计划与待监理协调开工状态，而不是停留在“报价完成”。

4. **把平台特色能力以前置轻量版接入主方案**  
   监理、变更单、关键审批、尾款机制、后台治理看板不能全部放到远期，但当前只做最小闭环所需版本。

5. **保持当前阶段边界清晰**  
   只做半包，不把主材系统、复杂 BI、重审批中心、完整工程作业平台一次性拉进当前主线。

---

## 3. 核心原则

### 3.1 报价 ERP 服从主链，不反向定义主链

报价 ERP 的职责是服务于：

`设计交付确认 → 报价基线数据 → 施工主体选择(company|foreman) → 施工报价 → 用户确认 → 项目创建 → 待监理协调开工 → 履约验收`

它不是独立业务主线，不应反向把项目做成工程 ERP 先行。

### 3.2 双成交点必须冻结

- **成交点 A：设计确认**
  - 用户确认设计费 / 正式方案
  - 代表设计成交成立
  - 不创建项目
  - 不进入正式履约

- **成交点 B：工长确认**
  - 用户确认施工主体与施工报价
  - 代表施工成交成立
  - 触发订单生成与项目创建
  - 从商机阶段进入履约阶段

### 3.3 项目创建边界不可前移

正式项目只能在工长确认后创建。  
因此：

- 设计方案确认后，不创建项目；
- 报价基线数据提交后，不创建项目；
- 报价任务生成后，不创建项目；
- 工长提交报价后，不创建项目；
- 只有用户确认施工报价后，才创建项目并进入待监理协调开工。

### 3.4 当前只支持半包

当前施工报价链路只覆盖半包范围：

- 施工费
- 人工费
- 工序与辅材相关费用

不纳入：

- 主材商城成交闭环
- 全包套餐
- 独立主材履约主链

因此需要显式支持：

- `pricing_mode = half_package`
- `material_included = false`

### 3.5 三端分工固定

- **小程序**：主成交端  
  负责用户预约、设计确认、工长选择、报价确认、支付推进、验收确认。

- **Merchant Web**：主履约端  
  负责设计资料提交、工程量基础表维护、施工报价、节点材料、履约资料。

- **Admin Web**：主治理端  
  负责监管、异常收口、争议处理、资金状态治理、链路看板。

---

## 4. 总体方案

v2 的总体策略不是推倒重做，而是在保留现有 Quote ERP 主骨架的前提下，完成三类增强：

1. **补齐设计交付到施工报价的源清单层**
2. **让施工报价确认与项目 / 支付 / 验收真正绑定**
3. **把平台特色能力以前置轻量版方式纳入主方案**

也就是：

- 保留已有标准项库、模板、价格库、报价任务、工长报价结构；
- 新增工程量基础表层；
- 增强报价任务与工长报价的来源、偏差、责任解释能力；
- 在确认施工报价后生成支付计划、项目、待监理协调开工状态；
- 为监理、变更、审批、节点验收和治理看板留出首期可用的执行接口。

---

## 5. 三层能力结构

## 5.1 第一层：主链闭环包

这一层必须首期一起做，不做则主链仍然不闭环。

### 5.1.1 预算 / 风格 / 沟通结果确认

定位：量房后的关键确认节点，用于锁定设计沟通边界。  
它不是项目创建节点，也不是施工成交节点。

需要沉淀的信息：

- 预算区间
- 风格方向
- 空间需求
- 工期预期
- 特殊要求

作用：

- 作为设计输出工程量基础表的前置边界；
- 作为工长推荐与报价解释的前置依据；
- 作为后续变更判断的原始参考。

### 5.1.2 工程量基础表（源清单层）

定位：施工报价的设计源数据层，不是报价结果本身。  
提交时机固定为：**正式方案确认后**。

建议新增对象：

- `construction_quantity_bases`
- `construction_quantity_base_items`

建议关键字段：

- 头部：`booking_id`、`proposal_id`、`designer_provider_id`、`version`、`status`、`notes`
- 明细：`room_name`、`work_category`、`standard_item_id`、`item_name`、`quantity`、`unit`、`specification`、`formula_snapshot`、`remark`

作用：

- 为工长报价提供统一输入；
- 为工长推荐提供预算与工种结构依据；
- 为后续偏差解释、轻量变更、验收结算提供源头参照。

### 5.1.3 报价任务强绑定工程量基础表

保留现有：

- `QuoteList`
- `QuoteListItem`
- `QuoteSubmission`
- `QuoteSubmissionItem`

但增强引用关系。

`QuoteList` 需要支持：

- `quantity_base_id`
- `quantity_base_version`
- `pricing_mode`
- `material_included`
- `payment_plan_generated_flag`

`QuoteListItem` 需要支持：

- `quantity_base_item_id`
- `baseline_quantity`
- `quoted_quantity`
- `quantity_adjustable_flag`
- `quantity_change_reason`
- `source_stage`

`QuoteSubmissionItem` 需要支持：

- 是否偏离基础量
- 偏离原因
- 是否需要平台标记
- 是否需要用户再次确认

### 5.1.4 施工主体推荐、选择与施工报价桥接

这一层是报价 ERP 在主链中的核心职责。

需要支持：

- 系统推荐施工主体
- 设计师辅助推荐
- 用户自主选择
- 施工主体类型为 `company | foreman`
- 装修公司主体可直接被确认；如有内部工长/班组信息，可展示但不强制先切到独立工长

推荐维度至少包括：

- 预算匹配度
- 地区匹配度
- 工种匹配度
- 价格库覆盖率
- 历史履约能力摘要

报价确认页至少需要展示：

- 空间
- 项目
- 数量
- 单位
- 单价
- 小计
- 总价
- 偏差说明
- 工期与责任边界

### 5.1.5 报价确认后自动生成支付计划

用户确认施工报价后，必须直接生成最小支付计划。  
第一阶段至少覆盖：

1. 首付款
2. 节点进度款
3. 尾款

每笔付款需要关联：

- `quote_list_id`
- `project_id`
- `milestone_id`（如适用）
- `payment_plan_type`
- `amount`
- `due_at`
- `paid_at`
- `status`

这一步是 v2 的关键：  
**报价确认后不只是“可支付”，而是“已进入带资金约束的履约准备态”。**

### 5.1.6 尾款机制（首期轻量版）

尾款不能再被视为纯远期能力。  
首期至少要明确：

- 尾款属于支付计划的一部分；
- 与最终验收绑定；
- 未完成最终验收前不能释放；
- 后续可扩展为延迟释放、质保期扣留等更复杂策略。

也就是说：  
**首期先把尾款位置立住，复杂规则后扩。**

### 5.1.7 最小验收结算骨架

首期不要求把所有施工治理做重，但必须具备最小骨架：

- 项目里程碑
- 节点状态
- 用户验收状态
- 对应款项释放状态

至少支持两种模式：

1. 分阶段验收放款
2. 整体验收后一次性完成尾款收口

### 5.1.8 Admin 最小链路治理看板

这不是 BI 大屏，而是当前阶段治理刚需。

建议首期至少可见：

- 当前主链阶段
- 当前负责人
- 是否待用户确认
- 是否待商家处理
- 是否待支付
- 是否待监理提交
- 是否待验收
- 是否超时
- 当前累计应付 / 已付 / 待付
- 当前节点
- 是否争议中

## 5.2 第二层：履约护栏包

这一层不应再被放到真正远期，而应与首期并行补齐，避免主链上线后履约失控。

### 5.2.1 监理轻角色接入

监理在 v2 中的定位是：  
**履约证据与现场记录责任人，不是成交前主角色。**

首期轻量接入能力包括：

- 待监理协调开工项目承接
- 计划进场时间维护
- 节点图片上传
- 文字说明
- 风险备注
- 验收材料整理
- 完工资料提交

职责边界：

- 设计师：前期设计与工程量基础表
- 工长：施工执行
- 监理：待协调开工、节点图文与验收材料
- 用户：关键验收与最终确认

### 5.2.2 轻量变更单机制

变更单不应该再完全后置。  
因为装修过程中一定会发生：

- 增项
- 减项
- 数量变化
- 工期变化

首期轻量版只做：

- 变更发起
- 变更原因
- 对金额 / 工期影响说明
- 用户确认后生效

暂不做：

- 复杂财务拆账
- 多轮审批编排
- 全量自动重算引擎

### 5.2.3 关键审批点

审批不做成重型中心，但必须前置最小治理点。

建议首期纳入：

- 报价偏差超阈值标记 / 审批
- 变更单确认
- 退款 / 争议审核

原则：

- 只管异常与高风险；
- 不覆盖所有正常报价流程；
- 不损伤成交效率。

### 5.2.4 节点验收与结算自动联动

主链一旦进入施工执行，就需要闭合：

`节点提交 → 用户验收 → 款项释放 / 冻结 → 下一节点`

至少需要支持：

- 节点提交
- 用户通过 / 驳回
- 驳回后整改重提
- 对应款项释放或冻结

### 5.2.5 轻量异常治理与审计留痕

当前阶段必须预留以下异常收口：

- 阶段验收不通过
- 暂停施工
- 争议升级
- 退款申请
- 项目关闭

关键要求：

- 动作留痕
- 状态可追溯
- Admin 可介入

## 5.3 第三层：后置重化能力

这一层才是真正应放到后面的内容。

### 5.3.1 复杂工程量公式引擎

包括：

- 高级公式
- 自动推导
- 复杂校验
- 多模板联动

### 5.3.2 完整监理作业平台

包括：

- 监理排班
- 巡检计划
- 标准质检模板
- 监理绩效统计

### 5.3.3 重型多级审批中心

包括：

- 按角色 / 城市 / 金额 / 风险级别分流
- 全链路审批流编排
- 复杂规则引擎

### 5.3.4 BI 驾驶舱与重报表

包括：

- 复杂经营分析
- 转化漏斗钻取
- 城市 / 品类 / 工种分层
- 长周期履约质量分析

### 5.3.5 完整售后与长期服务体系

包括：

- 售后工单
- 延保
- 长期回访
- 长周期服务订单

## 6. 关键业务边界

### 6.1 量房不是用户成交节点

量房资料只做留档和查看，不做用户确认成交。  
当前确认节点是：

- 预算
- 风格
- 沟通结果
- 正式方案
- 工长与施工报价

### 6.2 工程量基础表不是独立产品

它是报价 ERP 的组成部分，是施工报价源清单层。  
它不独立成交易线，不单独定义项目。

### 6.3 工长确认是项目创建唯一触发条件

施工报价确认后：

- 生成订单
- 创建项目
- 生成支付计划
- 进入待监理协调开工

在此之前，全部属于商机阶段。

### 6.4 监理不替代工长，也不替代用户

监理负责记录与证据，也负责 `ready_to_start` 阶段的待开工协同；但不负责：

- 设计成交
- 施工成交
- 用户验收决策
- 最终争议裁决

### 6.5 审批只覆盖关键风险场景

审批不应默认覆盖全部正常路径，避免损伤成交效率。

### 6.6 后台看板当前以治理为目标

首期后台看板只解决：

- 能否看到主链堵点
- 能否看到风险项目
- 能否看到支付与验收卡点

不追求一步到位变成经营驾驶舱。

## 8. 分阶段实施建议

## 8.1 第一阶段：完成主链闭环包

目标：打通

`设计确认 → 工程量基础表 → 工长选择 → 施工报价 → 用户确认 → 项目创建 → 支付计划 → 待监理协调开工`

建议范围：

1. 新增预算 / 风格 / 沟通结果确认单
2. 新增工程量基础表
3. 工程量基础表生成报价任务
4. 工长推荐与选择
5. 用户确认施工报价
6. 自动生成首付款 / 节点款 / 尾款结构
7. 建立最小验收结算骨架
8. 建立 Admin 最小治理看板

## 8.2 第二阶段：并行补齐履约护栏包

目标：让项目创建后的履约过程可见、可验、可控。

建议范围：

1. 监理轻角色接入
2. 节点上传
3. 节点验收
4. 轻量变更单
5. 关键审批点
6. 节点验收与结算自动联动
7. 异常冻结、争议、退款、关闭留痕

## 8.3 第三阶段：引入重化扩展能力

建议范围：

1. 复杂公式引擎
2. 完整监理平台
3. 重型审批中心
4. BI 驾驶舱
5. 长周期售后体系

## 9. 最终结论

v2 的核心结论只有三点：

### 9.1 报价 ERP 必须成为装修主链的桥接底座

它服务于：

`设计交付确认 → 报价基线数据 → 施工主体选择 → 施工报价 → 工长确认成交 → 项目创建 → 待监理协调开工 → 履约验收`

而不是独立扩张成先行工程 ERP。

### 9.2 平台特色能力不能全部后置

以下能力应以前置轻量版进入主方案：

- 监理
- 变更单
- 关键审批
- 尾款机制
- 后台治理看板

否则系统即使能成交，也难以控履约。

### 9.3 真正后置的，只应是重型平台化能力

后置的应该是：

- 复杂公式
- 完整监理平台
- 重型审批中心
- BI 驾驶舱
- 长周期售后

而不是把主链护栏能力一并推迟。

基于以上边界，v2 可以作为后续产品、研发、测试、运营统一对齐的正式底稿。
