## 1. 概述

> 说明：本文档中的 `quote_pk / quote_tasks / quote_pk_submissions` 结构已退为历史兼容，只读保留；现行报价 ERP 主链以 `QuantityBase / QuoteList / QuoteSubmission / ChangeOrder / Settlement / Payout` 为准。

### 1.1 数据库信息
- **数据库类型**: PostgreSQL 14+
- **字符集**: UTF-8
- **时区**: Asia/Shanghai
- **连接池**: pgBouncer

### 1.2 设计原则
- 遵循第三范式（3NF）
- 使用软删除（deleted_at）
- 统一时间戳字段（created_at, updated_at）
- 使用BIGSERIAL作为主键
- 敏感数据加密存储

## 2. 数据库架构

### 2.1 模块划分

```
家装平台数据库
├── 用户模块（users, providers）
├── 预约模块（bookings, proposals, site_surveys）
├── 项目模块（projects, milestones）
├── 支付模块（orders, transactions, escrow_accounts）
├── 验收模块（inspection_checklists, inspection_templates）
├── 合同模块（contracts）
├── 报价模块（legacy quote_tasks / quote_pk_submissions，只读兼容；现行 QuoteList / QuoteSubmission / ChangeOrder）
├── 通知模块（notifications）
└── 审计模块（audit_logs）
```

### 2.2 ER关系图（文字描述）

**核心实体关系**：
- User (1) ----< (N) Booking：一个用户可以有多个预约
- Booking (1) ----< (N) Proposal：一个预约可以有多个设计方案
- Proposal (1) ---- (1) Project：一个确认的方案对应一个项目
- Project (1) ----< (N) Milestone：一个项目有多个施工节点
- Milestone (1) ----< (N) InspectionChecklist：每个节点有验收清单
- Project (1) ----< (N) Contract：一个项目可以有多个合同
- User (1) ----< (N) Order：一个用户可以有多个订单
- Project (1) ---- (1) EscrowAccount：一个项目对应一个托管账户

## 4. 索引设计

### 4.1 主键索引
所有表都使用 `id BIGSERIAL PRIMARY KEY`

### 4.2 唯一索引
- `users.phone` - 手机号唯一
- `providers.user_id` - 一个用户只能有一个服务商账号
- `orders.order_no` - 订单号唯一
- `contracts.contract_no` - 合同编号唯一
- `inspection_templates.category` - 验收模板类别唯一

### 4.3 普通索引
- **时间索引**: 所有表的 `created_at` 字段
- **状态索引**: 所有表的 `status` 字段
- **关联索引**: 所有外键字段（user_id, provider_id, booking_id, project_id等）
- **业务索引**: 
  - `bookings.business_flow_stage` - 业务流程阶段查询
  - `projects.closure_type` - 项目关闭类型查询
  - `quote_tasks.expired_at` - 报价任务过期查询

### 4.4 复合索引
- `(provider_type, status, verified)` on providers - 工长匹配查询
- `(style, region, min_area, max_area)` on quote_estimate_templates - 智能报价匹配

## 5. 外键关系

### 5.1 用户关系
- `bookings.user_id` → `users.id`
- `bookings.provider_id` → `providers.id`
- `providers.user_id` → `users.id`

### 5.2 预约关系
- `proposals.booking_id` → `bookings.id`
- `proposals.provider_id` → `providers.id`

### 5.3 项目关系
- `projects.booking_id` → `bookings.id`
- `projects.proposal_id` → `proposals.id`
- `projects.owner_id` → `users.id`
- `projects.provider_id` → `providers.id`
- `milestones.project_id` → `projects.id`

### 5.4 支付关系
- `orders.user_id` → `users.id`
- `escrow_accounts.project_id` → `projects.id`

### 5.5 验收关系
- `inspection_checklists.milestone_id` → `milestones.id`
- `inspection_checklists.project_id` → `projects.id`

### 5.6 合同关系
- `contracts.booking_id` → `bookings.id`
- `contracts.project_id` → `projects.id`
- `contracts.user_id` → `users.id`
- `contracts.provider_id` → `providers.id`

### 5.7 报价关系
- 现行 ERP 主链：`quantity_bases -> quote_lists -> quote_submissions -> quote_submission_revisions -> change_orders -> payment_plans / settlement_orders / payout_orders`
- legacy 兼容链：`quote_tasks -> quote_pk_submissions` 仅保留历史查询，不再作为主导航、主通知、主业务写链。
- 用户确认施工报价后，`projects.construction_quote_snapshot` 必须保存成交时的报价行项目、报价版本、责任说明和付款计划摘要，争议/结算回放不得只依赖后续可变读模型。
- 商家价格库现行关键字段：`quote_price_book_items.unit_price_cent / min_charge_cent / remark`，后台巡检按施工主体适用工种统计覆盖率、缺失必填项和异常价。
- `quote_tasks.booking_id` → `bookings.id`
- `quote_tasks.user_id` → `users.id`
- `quote_pk_submissions.quote_task_id` → `quote_tasks.id`
- `quote_pk_submissions.provider_id` → `providers.id`

## 附录

### A. 数据库迁移脚本执行顺序

1. `add_inspection_tables.sql` - 验收清单表
2. `add_project_closure_fields.sql` - 项目关闭字段
3. `add_quote_templates.sql` - 智能报价模板
4. `add_quote_pk_tables.sql` - legacy 多工长报价PK表（历史兼容，只读保留）
5. `add_contract_esign_deposit_fields.sql` - 合同电子签章和定金字段

### B. 数据备份策略

- **全量备份**: 每天凌晨2点
- **增量备份**: 每小时一次
- **保留策略**: 全量备份保留30天，增量备份保留7天

### C. 性能优化建议

1. 定期执行 `VACUUM ANALYZE` 清理和分析表
2. 监控慢查询日志，优化慢SQL
3. 对大表进行分区（如 audit_logs 按月分区）
4. 使用连接池（pgBouncer）管理数据库连接
5. 定期检查索引使用情况，删除无用索引
