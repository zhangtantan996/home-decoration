# 用户侧 Web 开发方案

## 1. 文档目的

本文档用于在 `home-decoration` 项目中明确 **用户侧 Web（user-web）** 的产品定位、页面范围、阶段目标、目录建议、实施顺序与验收标准。
本方案基于现有 MVP 规划文档收敛，不重新发明方向，而是在既有规划上给出一版可执行实施方案。

## 2. 关联文档

本方案默认以以下文档为上位依据：

- `docs/mvp-product-direction.md`
- `docs/mvp-page-map.md`
- `docs/mvp-feature-tiering.md`

三者分工如下：

### 2.1 `mvp-product-direction.md`
用于明确：
- MVP 的产品目标
- 各端角色定位
- 阶段性边界

### 2.2 `mvp-page-map.md`
用于明确：
- 页面归属

## 3. 用户侧 Web 的产品定位

用户侧 Web 是面向普通用户/消费者的公开访问入口，核心职责不是管理后台，也不是商家工作台，而是：

- 展示平台品牌与能力
- 承接公开内容浏览
- 提供服务发现入口
- 支持用户浏览、筛选、比较、了解服务者与案例
- 提供轻咨询 / 留资 / 跳转等前台动作

### 3.1 user-web 当前阶段的角色
在当前阶段，user-web 应被视为：

> 一个独立的前台 Web 入口，优先承接“浏览决策链路”。

### 3.2 当前阶段不应承载的内容
在 MVP 第一阶段，user-web 不应优先承载：

- 商家后台管理能力
- 管理后台能力
- 重交易闭环
- 完整账户中心
- 复杂订单/支付/财务能力
- 与 merchant/admin 高度耦合的操作型页面

---

## 4. 当前推进背景

当前项目同时存在以下背景：

- `merchant / admin` 正在做边界分离
- user-web 与 `merchant / admin` 分离主线不冲突
- user-web 适合作为并行副线推进
- 但 user-web 不应反向拖慢主线分离

因此，当前最优策略不是把 user-web 做成另一个“大而全系统”，而是：

> 轻展示、低耦合、先骨架后功能。

---

## 5. 开发总策略

## 5.1 核心策略
用户侧 Web 当前应采用以下策略：

- 独立目录先落地
- 优先做公开浏览链路
- 优先做低耦合内容
- 优先复用数据能力，而非强行复用页面实现
- 先跑通，再优化抽象
- 不阻塞 merchant/admin 分离主线

## 5.2 一句话原则
> 先把“看、找、比、选”做好，再考虑“登、管、付、深互动”。

## 6. 建议目录方案

建议在仓库中采用以下结构：

```text
home-decoration/
├─ admin/
├─ merchant/
├─ web/
├─ shared/
├─ server/
├─ mobile/
├─ mini/
└─ docs/
```

其中 user-web 建议结构为：

```text
web/
├─ package.json
├─ vite.config.ts
├─ tsconfig.json
├─ index.html
├─ src/
│  ├─ app/
│  ├─ router/
│  ├─ pages/
│  ├─ components/
│  ├─ modules/
│  ├─ services/
│  ├─ hooks/
│  ├─ utils/
│  ├─ types/
│  └─ main.tsx
└─ public/
```

### 6.1 目录职责建议

#### `pages/`
页面级视图组件。

#### `modules/`
按业务域组织，例如：
- `home`
- `cases`
- `providers`
- `merchants`
- `search`

#### `services/`
用户侧 Web API 请求、DTO 适配、请求封装。

#### `types/`
用户侧视图模型、公共类型、adapter 类型。

#### `components/`
仅 user-web 使用的页面组件。

#### `shared/`
只放跨端真正低耦合的内容，如：
- design tokens
- types
- utils
- api adapter
- 极通用 UI

## 7. 页面范围建议

本节结合 `mvp-page-map.md` 与 `mvp-feature-tiering.md`，给出 user-web 推荐页面范围。

## 7.1 P0 页面（第一批必须做）
建议优先落这批页面：

- 首页
- 搜索结果页
- 列表页
- 详情页
- 基础导航（Header / Footer / 分类入口）
- 空态 / 错误态 / 加载态

### 7.1.1 首页
承接平台的第一入口，展示：
- 核心服务入口
- 推荐内容
- 热门商家 / 设计师 / 工长 / 案例
- 搜索入口
- 分类导航

### 7.1.2 列表页
可包括：
- 案例列表
- 商家列表
- 设计师列表
- 工长列表
- 分类服务列表

### 7.1.3 详情页
可包括：
- 案例详情页
- 服务者详情页
- 商家详情页
- 内容详情页

### 7.1.4 搜索 / 筛选页
支持：
- 关键词搜索
- 分类筛选
- 地区筛选
- 条件排序（若后端支持）

---

## 7.2 P1 页面（第二批建议做）
在 P0 跑通后，可继续推进：

- 留资页 / 留资模块
- 咨询入口页
- 收藏页
- 推荐页 / 专题页
- 分享落地页
- 用户轻中心（若已有 MVP 规划）

---

## 7.3 P2 页面（当前暂缓）
当前不建议优先推进：

- 完整登录注册体系
- 重用户中心
- 交易链路
- 支付能力
- 深消息系统
- 复杂互动闭环

---

## 8. 功能优先级建议

## 8.1 P0 功能
必须优先完成：

- 首页框架
- 列表展示
- 详情展示
- 搜索 / 筛选
- 基础数据读取
- 加载态 / 空态 / 错误态
- 基础响应式适配

## 8.2 P1 功能
第二阶段补充：

- 留资
- 咨询入口
- 收藏
- 轻登录占位
- 浏览历史 / 轻个性化入口

## 8.3 P2 功能
后置：

- 完整账户体系
- 订单 / 支付
- 财务相关
- 深度消息互动
- 高耦合业务闭环

---

## 9. 与 merchant / admin 的边界

## 9.1 user-web 不应承载
- 商家后台页面
- 商家管理台逻辑
- 审核后台能力
- admin 路由与权限体系
- 仅后台可见的内部数据能力

## 9.2 user-web 可以依赖
- 公开 API
- 展示型数据接口
- shared tokens / utils / types
- 与前台展示相关的公共能力

## 9.3 当前阶段边界原则
user-web 当前阶段必须满足：

> 不阻塞 merchant / admin 分离主线。

即：
- 不强依赖 admin 工程
- 不依赖 merchant 工作台逻辑
- 不因共享抽象过度而拖慢主线

---

## 10. 资产复用建议

## 10.1 优先评估可复用资产
建议优先评估以下内容：

- `shared/design-tokens`
- `mobile/src/types/**`
- `mobile/src/services/api.ts` 中可抽象的数据请求模式
- `mini` 中展示型数据结构经验
- `server` 已有公开查询接口

## 10.2 谨慎复用的内容
以下内容不建议直接硬搬：

- React Native 专属组件实现
- 小程序页面结构
- admin 内部组件
- merchant/admin 高耦合服务层

## 10.3 复用原则
优先复用：
- 数据结构
- 类型
- 工具函数
- adapter
- design tokens

谨慎复用：
- 页面结构
- 强交互组件
- 平台相关渲染实现

---

## 11. 分阶段实施方案

## Phase 1：建立独立 web 骨架

### 目标
建立 user-web 独立目录与最小运行骨架。

### 实施内容
- 新建 `web/`
- 初始化最小 Vite + React 工程
- 建立基础路由
- 建立首页占位
- 配置基础 env / API base URL / 构建脚本

### 验收标准
- `web` 可启动
- `web` 可 build
- 至少有首页可访问

---

## Phase 2：完成公开浏览主链路

### 目标
完成用户浏览主链路。

### 建议优先页面
- 首页
- 列表页
- 详情页
- 搜索 / 筛选页

### 目标能力
用户能够完成：
- 进入首页
- 浏览内容
- 搜索内容
- 查看详情
- 初步比较服务对象

### 验收标准
- 浏览主链路可走通
- 页面之间跳转正常
- 公开数据读取正常

---

## Phase 3：补轻交互能力

### 目标
在不做重账户的前提下，补用户前台轻交互。

### 可选内容
- 留资
- 咨询入口
- 收藏
- 分享落地
- 推荐位优化
- 页面空态/错误态完善

### 验收标准
- 用户不只是浏览，还能完成最小互动动作

---

## Phase 4：为后续功能留接口

### 目标
不立即做复杂业务，但为未来扩展留后路。

### 预留内容
- auth 入口占位
- 用户中心入口占位
- 预约 / 下单 / 互动入口占位
- 埋点 / SEO 基础结构（如需要）

### 验收标准
- 未来可继续扩展，而不需要推倒重做

---

## 12. UI 与布局要求

此前已有明确协作偏好：
凡涉及 UI 改动或评审，必须优先保证布局稳定性。

因此 user-web 必须重点验证以下情况：

- 长标题
- 长描述
- 空数据
- 图片缺失
- 极端价格 / 标签长度
- 加载中
- 请求失败
- 列表过长
- 卡片挤压
- 桌面宽度 / 移动宽度切换
- 动态增删内容

### 最低要求
- 不出现明显错位
- 不出现文本溢出失控
- 不出现卡片挤压到不可读
- 不出现错误态 / 空态布局崩坏

---

## 13. 测试建议

## 13.1 最小测试项
- `web` build
- 核心页面路由可打开
- 列表 / 详情请求成功或正确失败展示
- 空态 / 错误态 / 加载态展示正常
- 响应式布局无明显错位

## 13.2 建议验证项
- 首页
- 列表页
- 详情页
- 搜索筛选页
- 长文本 / 空图片 / 空数据场景
- 弱网 / 请求失败场景

---

## 14. 第一轮交付建议

如果当前要快速启动 user-web，建议第一轮只做到：

- 建立 `web/` 独立工程
- 落首页骨架
- 落一个列表页
- 落一个详情页
- 打通最小公开数据读取
- 保证 build 可过
- 不碰重登录态
- 不抢 merchant/admin 分离主线资源

这是当前阶段最有性价比的切入点。

---

## 15. 非目标

当前 user-web 方案明确不包括：

- 完整用户账户体系
- 支付与交易闭环
- 商家后台管理能力
- admin 能力迁移
- server 架构重构
- 大规模 shared 平台化改造

---

## 16. 结论

用户侧 Web 当前的正确推进方式不是“大而全”，而是：

- 轻展示
- 低耦合
- 先骨架
- 先浏览链路
- 先独立入口
- 不阻塞主线分离

在现阶段，user-web 最值得优先做的是：

- 独立目录
- 首页
- 列表
- 详情
- 搜索 / 筛选
- 轻交互入口

而不是：
- 重账户
- 重交易
- 重后台联动
- 过度共享抽象
