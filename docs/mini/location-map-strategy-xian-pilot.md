# 小程序地图定位能力评估：西安试点

> 评估日期：2026-04-25
> 适用范围：`mini` 小程序用户端、服务商浏览、报价/需求/预约、材料店与项目履约相关地址能力。
> 当前结论：西安单城试点阶段，不建议把地图定位作为近期必做能力；建议先保留地址结构和后端封装预案，等多城扩张、材料店到店、服务半径调度或距离排序成为真实需求后再接入。

## 1. 背景

当前业务首先在西安试点，且服务商基本覆盖整个西安市范围接单。

这意味着当前最重要的匹配维度不是“离用户最近”，而是：

- 服务商类型：设计师、装修公司、工长、主材商。
- 服务能力：设计风格、施工能力、履约口碑、案例质量。
- 服务区域：是否覆盖西安市及对应区县。
- 预算与房屋信息：面积、户型、装修类型、预算区间。
- 当前交易阶段：报价、预约、方案、订单、项目。

在这种业务前提下，地图定位的价值会明显低于“地址标准化”和“服务范围匹配”。

## 2. 当前代码现状

### 2.1 已有地址与区域能力

- `mini/src/services/regions.ts` 已有省市区接口消费能力。
- `server/internal/service/provider_service.go` 已有服务商城市/服务区域过滤。
- `docs/TROUBLESHOOTING.md` 记录过服务区域从文本迁移到行政区划码。
- `server/internal/service/quote_inquiry_service.go` 已能从地址文本推断报价城市。

### 2.2 已有经纬度字段，但未形成闭环

服务端模型中已有若干地理字段，例如：

- 服务商/材料店相关模型存在 `latitude / longitude / address`。
- 预约、项目、报价询价等模型存在 `address`，部分模型存在经纬度或加密地址字段。
- `provider_service.go` 中距离计算仍是 TODO，说明当前并未依赖用户坐标做服务商排序。

### 2.3 当前用户输入仍以手填地址为主

例如需求创建页当前仍是：

- 城市
- 区域
- 详细地址

这对西安试点是可用的，但后续可以优化为“默认西安 + 区县选择 + 详细地址输入”，不必一开始接地图定位。

## 3. 是否现在接入地图定位

### 3.1 不建议现在做成核心能力

西安试点阶段，暂不建议现在直接接入完整地图定位能力。

原因：

- 服务商全西安接单，距离对分单和成交影响不大。
- 用户选择设计师/装修公司/工长时，更关注案例、报价、口碑、服务能力，而不是 3 公里还是 8 公里。
- 家装上门服务天然不是即时配送，空间距离不是核心效率指标。
- 一进首页就申请定位会带来隐私压力，反而影响首屏体验。
- 当前服务商筛选、报价、需求、预约都可以先用城市/区县/地址文本完成。
- 接入地图服务会带来 key 管理、配额、缓存、坐标系、合规说明、隐私权限、异常兜底等额外复杂度。

### 3.2 当前真正需要的是地址标准化，不是实时定位

近期更有价值的是：

- 报价、需求、预约地址字段统一。
- 西安作为默认城市。
- 区县用行政区划码选择，不再完全手填。
- 详细地址保留手输。
- 后端保存 `cityCode / cityName / districtCode / districtName / address`。
- 用户家庭地址不在列表页暴露完整门牌。

这比“地图选点 + 经纬度 + 距离排序”更符合当前阶段。

## 4. 开源还是付费

### 4.1 不建议纯开源地图

不建议纯开源地图作为主方案。

原因：

- 微信小程序内地图底图、POI、地址解析更适合使用平台能力或国内 LBS 服务。
- 国内地址存在 GCJ-02 坐标体系问题，纯开源方案容易出现偏移和数据不完整。
- 开源地图通常不能直接解决小区/楼盘/POI 搜索和逆地址解析问题。
- 后续商用稳定性、合规和服务 SLA 不如成熟 LBS 平台。

### 4.2 不建议现在购买付费能力

当前也不建议马上购买付费地图能力。

原因：

- 试点城市单一，调用量和业务价值还不足以证明付费必要性。
- 当前没有必须依赖地图服务的主链路。
- 服务商不是按距离派单，用户也不是按地图找服务商。
- 可先用行政区划 + 手填地址完成试点闭环。

### 4.3 推荐路线

推荐路线是：

1. 当前阶段不接完整地图服务。
2. 代码和数据结构保留地图能力预案。
3. 后续需要时优先接微信原生位置能力 + 腾讯位置服务。
4. 付费只作为调用量超过免费额度或业务依赖增强后的升级项。

## 5. 分阶段方案

### 阶段 0：西安试点当前推荐方案

目标：不接地图定位，先把地址和服务区域做稳。

建议动作：

- 小程序地址表单默认城市为“西安”。
- 城市固定或弱化，只让用户选择区县。
- 区县使用 `regions` 行政区划数据，不完全手填。
- 详细地址继续手动输入。
- 报价询价、需求、预约统一保存城市和区县字段。
- 服务商列表继续按服务区域/城市过滤。
- 不展示“定位授权”“附近服务商”“距离我 xx km”等能力。
- 隐私设置中不保留未消费的定位开关。

适用页面：

- `pages/quote-inquiry/create`
- `pages/demands/create`
- `pages/booking/create`
- 后续服务商入驻/资料编辑页

### 阶段 1：轻量地图选点

触发条件：

- 用户手填地址错误率明显偏高。
- 平台需要更准确地识别小区/楼盘。
- 预约地址需要更标准化。
- 材料店或服务商详情需要“打开地图”。

建议能力：

- 在地址输入旁增加“选择地址”。
- 用户主动点击后调用小程序原生选点能力。
- 不在首页自动申请定位。
- 用户取消或拒绝授权时静默返回或提示“可手动填写地址”。
- 保存 `address / latitude / longitude / cityCode / districtCode / geoSource`。
- 详情页支持“打开地图”。

注意：

- 经纬度只用于地址辅助，不用于强制定位用户。
- 用户家庭地址需要按权限展示。
- 不做持续定位。

### 阶段 2：腾讯位置服务后端封装

触发条件：

- 地址解析、逆地址解析、POI 搜索成为高频需求。
- 多城扩张后需要更准确地识别城市和区县。
- 服务商/材料店需要距离排序。

建议能力：

- 后端封装腾讯位置服务 WebService。
- 小程序不直接暴露地图 key。
- 增加地址解析缓存，避免同一地址重复请求。
- 支持 POI 输入提示、地址解析、逆地址解析、距离计算。
- 配置调用量监控和限流。

### 阶段 3：付费地图能力

触发条件：

- 免费额度不够。
- 服务商距离排序直接影响转化。
- 材料店到店导航成为重点能力。
- 施工现场打卡、项目巡检、路线规划进入业务主链。

到这个阶段再评估购买额度或更高 SLA。

## 6. 是否需要地图能力的判断标准

满足以下任意两项，再进入阶段 1 或阶段 2：

- 手填地址导致大量客服纠错。
- 多个服务商只服务部分区县，不再全西安接单。
- 用户明确需要看“离我最近”的材料店或服务商。
- 平台开始按距离或服务半径分单。
- 材料店线下到店成为重要成交入口。
- 项目巡检、工人打卡、现场定位成为履约管理要求。
- 多城试点开始，城市识别和跨城服务范围出现误匹配。

不满足这些条件时，不建议提前做地图定位。

## 7. 推荐结论

当前西安试点阶段：

- 不做首页定位。
- 不做附近服务商。
- 不做实时定位。
- 不做纯开源地图。
- 不立即购买付费地图服务。
- 先做城市/区县/地址标准化。
- 后续如需地图，优先微信原生能力和腾讯位置服务。

短期更应该优先做：

1. 地址字段统一。
2. 西安默认城市。
3. 区县选择器。
4. 服务商服务区域准确展示。
5. 预约/订单/项目详情地址权限收口。

地图定位能力可以作为后续增强项，而不是当前主链路前置项。
