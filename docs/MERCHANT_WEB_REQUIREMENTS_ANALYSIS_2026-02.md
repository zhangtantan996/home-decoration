# 商家中心（服务商）Web 页面全面需求分析与迭代计划（2026-02-09）

## 1. 目标与范围

### 1.1 目标
- 以**入驻转化优先**为原则，系统梳理 `/merchant/*` 全链路页面需求。
- 解决“角色定义存在但页面体验不一致/闭环不完整”的问题。
- 形成可直接进入研发排期的**两阶段迭代计划**（先闭环、再体验重构）。

### 1.2 范围
- 前端：`admin/src/pages/merchant/*`、`admin/src/layouts/MerchantLayout.tsx`、`admin/src/services/merchantApi.ts`
- 后端：`server/internal/handler/*merchant*.go`、`server/internal/router/router.go`
- 文档：PRD、MVP 状态、API 文档与变更清单
- 终端：**桌面 Web 优先**（不含 mobile/mini 交互重构）

### 1.3 当前已确认角色
- 独立设计师（personal）
- 设计工作室（studio）
- 装修公司（company）
- 工长/项目经理（foreman）

## 2. 现状基线盘点（As-Is）

### 2.1 路由与页面矩阵（已上线）
- 入驻前链路：`/merchant/`、`/merchant/login`、`/merchant/register`、`/merchant/apply-status`
- 入驻后链路：`/merchant/dashboard`、`/bookings`、`/proposals`、`/orders`、`/chat`、`/income`、`/withdraw`、`/bank-accounts`、`/cases`、`/settings`
- 路由定义：`admin/src/merchant-router.tsx`

### 2.2 已完成能力（可用）
- 入口页已补齐工长卡片，支持 `?type=foreman`：`admin/src/pages/merchant/MerchantEntry.tsx`
- 入驻申请 API 已支持 foreman + `workTypes`：`server/internal/handler/merchant_apply_handler.go`
- 审核通过已映射 `foreman -> provider_type=3`：`server/internal/handler/merchant_apply_handler.go`
- 商家基础业务模块齐全（预约/方案/订单/财务/聊天）

### 2.3 关键断层（面向迭代）
1. **接口契约不统一（高风险）**
   - 统一响应是 `{code,message,data}`，但前端页面解析方式混杂（有的取 `data`，有的直接当平铺对象）。
   - 结果：页面间偶发“有接口无数据”的静默失败风险。

2. **工作台数据结构前后端不一致（高风险）**
   - 前端 `MerchantDashboard` 期待平铺字段（如 `todayBookings`）。
   - 后端 `/merchant/dashboard` 返回分组字段（`bookings/proposals/orders`）。
   - 结果：工作台卡片数值不稳定或长期为 0。

3. **入驻上传仍有占位 URL（高风险）**
   - 注册页上传回调仍写入 `https://placeholder.com/...`。
   - 结果：提交后资料不可追溯，审核真实性不足。

4. **提现/银行卡二次验证流程前后不一致（高风险）**
   - 后端提现与新增银行卡要求 `verificationCode`。
   - 前端表单未提供该字段，存在调用失败风险。

5. **角色信息登录后被“扁平化”**
   - `personal/studio` 登录后统一映射为 `designer`，角色细分能力丢失。
   - 结果：无法做工作室与独立设计师的差异化页面策略。

6. **工长入驻后资料维护闭环仍不完整**
   - 设置页以“设计风格/公司简介”为主，缺少工长核心字段（工种、施工服务项）可编辑闭环。

7. **服务设置模型有库表，无 API 闭环**
   - `MerchantServiceSetting` 模型存在，但商家端缺 GET/PUT 对应能力。

## 3. 服务商角色任务模型（To-Be）

### 3.1 独立设计师
- 主目标：接单、出方案、展示作品、维护个人品牌
- 关键页面：预约、方案、作品集、聊天、收入
- 差异化需求：风格标签、案例质量、设计费结构、响应速度

### 3.2 设计工作室
- 主目标：团队化接单、标准化交付、品牌化展示
- 关键页面：工作台、方案、作品集、设置、财务
- 差异化需求：团队规模、服务能力边界、品牌物料统一

### 3.3 装修公司
- 主目标：资质背书、施工交付、资金流与账户管理
- 关键页面：预约、订单、财务、设置
- 差异化需求：执照信息、施工能力、服务区域与履约稳定性

### 3.4 工长/项目经理
- 主目标：施工接单、现场交付、施工案例沉淀
- 关键页面：预约、订单、聊天、设置（施工维度）
- 差异化需求：工种矩阵、施工经验、施工案例、可承接范围

## 4. 页面级需求分析（按业务链路）

### 4.1 入口页 `/merchant/`
**现状**
- 4 角色入口已存在，但信息密度不足（仅标题+一句描述）。

**需求**
- 每个角色卡片增加：适用对象、必备材料、预计审核时长。
- 明确“双主路径”：`已有账号登录` 与 `立即入驻`。
- 增加“选择角色后可修改”的心理预期文案，降低犹豫成本。

### 4.2 登录页 `/merchant/login`
**现状**
- 仅手机号+验证码，快捷入口存在但不完整。

**需求**
- 快捷入口覆盖四角色（含工作室入口文案）。
- 登录成功响应中补充 `applicantType`（原始角色）供前端 IA 决策。
- 显示最近一次申请状态入口（减少用户来回跳转）。

### 4.3 注册页 `/merchant/register`
**现状**
- 已支持 foreman 动态字段；但上传仍有 placeholder；步骤通用化较重。

**需求**
- 按角色分流步骤：
  - 设计类（personal/studio/company）：资质→作品→服务设置
  - 施工类（foreman）：资质→施工经验/工种→施工案例→服务设置
- 接入真实上传：身份证/执照/案例全部走 `merchantUploadApi.uploadImage`。
- 增加草稿保存（localStorage）与断点恢复。
- 增加“提交前材料校验清单”。

### 4.4 申请状态页 `/merchant/apply-status`
**现状**
- 已支持驳回后按原类型重提。

**需求**
- 对驳回原因结构化展示（问题项 + 修复建议 + 跳转锚点）。
- 支持“保留上次资料后继续编辑”而非全量重填。

### 4.5 工作台 `/merchant/dashboard`
**现状**
- 有卡片与快捷入口，但字段契约与后端不一致。

**需求**
- 统一契约（推荐后端返回前端直用字段，或前端做明确 mapper）。
- 角色化工作台卡片：
  - 设计类突出方案待确认、案例审核
  - 工长突出施工订单、施工进度、响应时效
- 快捷入口按角色动态展示。

### 4.6 预约/方案/订单 (`/bookings` `/proposals` `/orders`)
**现状**
- 预约页对所有角色都提供“录入设计方案”，与工长角色定位冲突。

**需求**
- 将“方案”抽象为“报价/施工方案”，按角色展示字段：
  - 设计类：设计费 + 设计说明
  - 施工类：施工报价 + 工期 + 工种分项
- 工长在 `bookings` 中不应出现纯设计语义按钮文案。
- 订单页增加角色相关筛选（设计费/施工费等）。

### 4.7 作品/案例 `/cases`
**现状**
- 侧边菜单对 foreman 隐藏 `/cases`，但入驻又要求提交施工案例。

**需求**
- 明确策略二选一：
  1) foreman 保留案例中心（改名“施工案例”）
  2) foreman 不暴露案例中心，则在设置页提供施工案例维护入口
- 前后端字段统一 `caseType`（design/construction）以支持后续检索。

### 4.8 设置页 `/settings`
**现状**
- 表单以设计风格为中心；服务设置卡片未与后端打通。

**需求**
- 基本资料分角色：
  - 设计类：擅长风格、设计简介
  - 工长：工种、施工简介、施工年限
- 新增服务设置 API：
  - `GET /api/v1/merchant/service-settings`
  - `PUT /api/v1/merchant/service-settings`
- 服务设置项：接单状态、自动确认时长、响应时效、价格区间、服务包。

### 4.9 财务链路 `/income` `/withdraw` `/bank-accounts`
**现状**
- 可用但前端解析与后端响应存在不一致；二次验证码交互缺失。

**需求**
- 全部统一解析 `ApiEnvelope<T>`。
- 新增二次验证码输入与发送流程（提现/银行卡新增共用）。
- 提现前增加风控提示：到账时效、失败回退机制、支持银行说明。

### 4.10 消息页 `/chat` + 通知
**现状**
- 功能较完整，支持多附件，但与业务动作联动不足。

**需求**
- 增加“业务快捷动作”：从会话直达预约详情/方案详情/订单。
- 通知中心增加可执行 CTA（例如“去处理预约”）。

## 5. 公共接口与数据契约需求

## 5.1 响应结构统一（必须）
- 前端统一消费：`{ code, message, data }`
- 建议在 `admin/src/services/merchantApi.ts` 引入统一 `unwrapResponse<T>()`。

### 5.2 登录态角色字段（必须）
- 登录返回增加：
  - `applicantType`（原始：personal/studio/company/foreman）
  - `providerSubType`（归一化：designer/company/foreman）
- 前端 IA 决策优先使用 `applicantType`，功能权限可用 `providerSubType`。

### 5.3 工长资料闭环（必须）
- `GET/PUT /merchant/info` 支持 `workTypes` 字段。
- 施工类资料可持续编辑，不仅在入驻阶段可填写。

### 5.4 服务设置 API（必须）
- 新增并落库到 `merchant_service_settings`。
- 前端设置页与后端真实数据双向绑定。

### 5.5 上传能力（必须）
- 注册页全部上传字段切换到真实上传。
- 失败重试/进度提示/上传后预览需统一。

## 6. 两阶段迭代计划（开发可执行版）

## 阶段 1：闭环补齐（4-6 天，转化优先）

### 目标
- 不大改视觉，先确保“能注册、能审核、能登录、能维护、能提现”的闭环可靠。

### 任务包
1. **契约与解析统一（前后端）**
   - 修复 dashboard 字段对齐
   - 统一 merchant 页面响应解析
2. **注册上传真实化**
   - 去除 placeholder URL
   - 接入真实上传并做失败反馈
3. **工长资料闭环**
   - `merchant/info` 增加 `workTypes` 读写
   - 设置页新增工长字段
4. **财务二次验证流程补齐**
   - 银行卡新增/提现补充验证码字段与交互
5. **文档同步**
   - API 文档、变更清单、MVP 状态文档统一

### 阶段验收
- foreman 从入口到登录闭环跑通
- dashboard 关键卡片数据正确展示
- 提现/银行卡新增不再因缺字段失败
- 关键页面不再混用响应解析方式

## 阶段 2：体验重构（5-7 天，增长优化）

### 目标
- 构建角色化 IA 与更高转化的入驻体验。

### 任务包
1. **入口与登录 IA 重构**
   - 角色卡信息增强（适用对象/材料/时长）
   - 双主路径强化（登录 vs 入驻）
2. **角色化动态流程**
   - 设计类与施工类分步骤表单
   - 驳回原因结构化回填
3. **角色化工作台与导航**
   - 设计类/施工类差异化模块编排
4. **埋点体系接入**
   - 角色选择率、步骤流失率、提交成功率、审核通过率
5. **可用性细节**
   - 草稿保存、断点续填、提交前预检

### 阶段验收
- 入驻漏斗各步骤可观测
- 角色路径清晰，前后流程无“跳错页/看不懂”
- 主要页面的角色文案一致性达标

## 8. 关键风险与依赖

### 风险
- 历史接口返回格式不一致，改动可能影响现网页面。
- 角色语义（applicantType vs providerSubType）调整会影响菜单与权限判断。
- 上传从 mock 切实链路后，需要联调存储路径与访问域名。

### 依赖
- 后端提供统一契约与 service-settings API。
- 前端统一 response parser 与 role context。
- QA 提供按角色的回归矩阵。
