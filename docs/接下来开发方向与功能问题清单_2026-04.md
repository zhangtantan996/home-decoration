# 接下来开发方向与功能问题清单（2026-04）

> 适用范围：当前仓库 `home-decoration`
> 依据真相源：`docs/产品需求文档(PRD).md`、`docs/BUSINESS_FLOW.md`、`docs/平台阶段判断与主商业模型_2026-04.md`、`docs/小程序优先增长与成交操作系统_2026-04.md`、`docs/经营指标与运营驾驶舱定义_2026-04.md`、`docs/当前实现业务流_vs_文档基线业务流分析.md`

## 1. 结论先行

当前项目接下来最需要解决的，不是继续横向增加功能，而是把主业务链做深、做稳、做成可经营的闭环。

如果按当前阶段判断，后续开发方向必须严格围绕 4 件事展开：

1. 强化小程序主成交端
2. 补设计确认到工长确认的桥接能力
3. 建立商家供给分层与治理规则
4. 统一经营指标与驾驶舱口径

这意味着：

- 不应继续把资源优先投入到主材商城、广告位、积分、泛内容社区等旁路能力
- 不应把“页面存在”误判为“业务已经跑通”
- 不应继续扩大多端复杂度，而应优先补主链路掉点和异常收口

## 2. 当前阶段判断

当前项目已经不是演示型 MVP，也不是单纯的信息展示平台，而是一个正在验证并收敛“可信成交 + 可控履约 + 可治理异常”的交易平台。

所以当前开发方向的判断标准不应该是“还能加什么功能”，而应该是：

- 是否能提升预约、设计确认、工长确认的转化
- 是否能提升项目履约稳定性
- 是否能降低退款、争议、投诉、延期、扯皮
- 是否能让平台形成真实经营和治理抓手

如果一个功能不能直接服务这四件事，就不应进入当前主线。

## 3. 现在最核心的 6 个问题

## 3.1 设计确认到工长确认之间，流程有了，但桥接能力还不够强

这是当前最核心的问题。

现状不是“没有流程”，而是用户在设计确认之后，缺少足够强的继续成交理由。当前最容易发生的问题包括：

- 用户认可设计方案，但不敢继续确认施工主体
- 用户看不懂施工报价差异和责任边界
- 用户不知道设计图能否真正落地
- 用户担心后续增项、延期、扯皮、验收不过
- 用户看不到足够清晰的案例、评价、履约结果支撑

这段链路如果转不动，平台就会停在“做了设计，但没有把施工成交吃下来”的阶段。

### 为什么这是 P0

因为当前主商业模型不是内容变现，也不是广告变现，而是：

- 设计成交佣金
- 施工成交佣金
- 托管 / 验收 / 风控服务费

如果设计确认之后无法稳定进入工长确认，主商业模型就只完成了一半。

### 当前真正缺的不是更多按钮，而是 6 类桥接抓手

1. 同一方案下的施工主体对比
2. 同一方案下的报价基线解释
3. 工期、增项、责任边界说明
4. 阶段验收与整改规则说明
5. 风险提示与平台介入机制说明
6. 已完工案例与真实评价背书

### 开发建议

优先补的是这类功能表达，而不是再加新的外围模块：

- 小程序里的施工方选择对比页
- 报价解释组件和责任边界说明卡片
- 设计确认后的下一步引导页
- 工长确认前的风险与保障说明页
- 真实案例、完工评价、履约标签联动展示

## 3.2 小程序已经是主成交端，但成交抓手仍可能弱于流程覆盖

当前小程序页面已经覆盖首页、案例、预约、方案、报价、项目、退款、争议等核心路径，方向是对的。

但当前更大的风险是：

- 页面有了，但不一定能有效推用户往下走
- 业务状态有了，但不一定足够可解释
- 提交流程有了，但不一定能建立信任

### 当前可能存在的实际问题

- 首页价值表达不够集中，用户看完不知道为什么要在平台内做决策
- 服务商详情页更像展示页，信任转换抓手不足
- 预约页可能仍然采集过多，提交阻力偏大
- 支付前页面更像催单，而不是解释“为什么现在值得确认”
- 设计确认后，缺少清晰的下一步预期说明

### 开发建议

当前小程序应优先围绕成交漏斗做增强：

1. 首页只保留最关键的成交入口和信任抓手
2. 详情页强化案例、履约标签、平台规则和 CTA
3. 预约页收敛表单、降低阻力、解释下一步
4. 支付前页面讲清业务承诺、退款规则、后续路径
5. 设计确认后增加施工桥接引导与对比能力

## 3.3 履约后链路已有基础，但“验收 -> 放款 -> 完工 -> 归档 -> 案例沉淀”还要继续打穿

从当前代码信号看，项目、里程碑、完工、退款、争议、提现、案例管理这些能力都已经有明显基础。

但当前仍有一个关键问题：

**后链路有模块，不等于后链路已经真正闭环。**

### 当前最值得警惕的风险

- 用户端是否真的能逐节点发起验收
- 商家端是否真的能逐节点提交验收材料
- 管理端是否能在异常节点顺滑介入
- 验收通过是否与放款、结算、托管释放严格绑定
- 是否同时支持“里程碑放款”和“整体验收一次性放款”
- 完工之后的归档、案例草稿、评价沉淀是否严格同口径

### 特别需要注意的业务口径风险

项目进入 archived，不一定意味着资金已经完全结清。

如果用户、商家、后台三端对 archived、completion、settlement、case_pending_generation 这些状态的理解不一致，后面会出现：

- 用户误以为项目已经彻底结束
- 商家误以为可以立即提现或结算完成
- 后台在财务、案例、项目关闭口径上出现冲突

### 开发建议

当前应优先做一次后链路专项核查与补强，重点看：

1. 里程碑验收动作是否三端一致
2. 验收结果是否驱动支付/托管/放款动作
3. 完工审批后是否统一进入案例沉淀链
4. archived、settlement、payout、case draft 的业务定义是否统一
5. 页面文案、状态标签、后端状态机是否一致

## 3.4 Merchant 已经在承担主履约角色，但供给治理规则还需要继续收敛

当前 merchant 不是资料后台，而是商家真正的履约工作台。

这意味着下一个问题不是“商家端还要不要继续做功能”，而是：

- 什么样的商家算优质供给
- 什么样的商家应该优先分发预约
- 什么样的商家需要重点扶持
- 什么样的商家要进入风险观察、限流或处罚

如果这套规则不明确，会出现两个后果：

- 前端成交转化上来了，但后端履约质量不稳定
- 平台的好供给和差供给混在一起，用户感知波动很大

### 当前建议优先建设的治理抓手

1. 商家分层标准
2. 响应率、提案率、确认率、完工率等履约标签
3. 风险商家识别与限流机制
4. 优质商家扶持与曝光机制
5. 履约异常、退款、争议与商家评级联动

## 3.5 Admin 已经有治理骨架，但还要防止“治理页面很多，治理机制不够统一”

当前 admin 已经覆盖审核、项目、风控、退款、提现、仲裁、日志等模块，说明方向是对的。

但治理端最常见的问题不是没有页面，而是：

- 相同业务在不同模块里口径不统一
- 争议、退款、项目审计之间职责边界不清楚
- 财务动作、审核动作、项目状态动作没有完全串起来
- 日志有了，但关键动作未形成可追责闭环

### 开发建议

治理端接下来不要盲目继续扩菜单，而要优先做机制收敛：

1. 明确退款、争议、仲裁、项目审计之间的边界
2. 明确哪些动作改变项目状态，哪些只改变财务状态
3. 明确哪些动作必须留痕、复核、复审
4. 把资金、项目、争议、案例沉淀的状态口径统一到一套真相源

## 3.6 指标口径如果不尽快统一，后面的开发会越来越像盲飞

当前阶段不需要一个什么都看的大 BI，而是需要一套能指导经营决策的统一口径。

如果没有统一指标，会出现：

- 产品团队不知道该优化预约还是优化桥接
- 运营团队不知道是商家问题还是用户转化问题
- 开发团队做了很多需求，但无法判断是否真正改善主链路
- 各端各自看自己的“完成数”，却没有全链路真相

### 当前必须尽快落地的统一指标

北极星：

- 完工且无重大争议的项目数

五个核心拆解指标：

1. 有效预约数
2. 设计确认数
3. 工长确认数
4. 完工项目数
5. 争议率 / 退款率

### 开发建议

先不要追求复杂 BI，优先做：

- 服务端业务结果事件的统一沉淀
- 双漏斗核心节点定义
- 管理端一级经营看板
- 周报和复盘统一口径

## 4. 当前不建议优先投入的方向

在当前阶段，下列方向都可以保留，但不应抢主线资源：

- 主材商城升为当前主经营引擎
- 广告位 / 曝光位收费体系
- 积分体系 / 会员体系
- 泛内容社区能力
- 为了“看起来完整”继续横向补边缘模块

原因不是这些方向永远不做，而是它们当前不能优先解决主问题：

- 主成交是否稳定
- 设计到施工是否桥接成功
- 履约是否稳定
- 异常是否能被治理收口

## 5. 建议的开发优先级

如果按接下来 1 到 2 个阶段来排，建议优先级如下：

### P0：必须优先解决

1. 设计确认 -> 工长确认桥接能力
2. 小程序成交页与支付前说明能力
3. 里程碑验收 / 放款 / 完工 / 归档口径统一
4. archived / settlement / case draft / payout 状态定义统一
5. 核心经营指标与双漏斗口径统一

### P1：主链稳定后紧跟推进

1. 商家供给分层和治理规则
2. 优质商家扶持与风险商家限流机制
3. 完工后的评价、案例、履约标签沉淀
4. Admin 治理机制与日志/审计口径统一

### P2：等主链稳定后再考虑

1. 内容分发增强
2. 主材商城深化
3. 广告或其他商业化探索
4. 非主链社区化功能

## 6. 一句话结论

你现在最该做的，不是继续问“还能加什么功能”，而是把这条主链真正打穿：

**预约 -> 设计确认 -> 工长确认 -> 项目履约 -> 验收放款 -> 完工沉淀**

其中最关键的两个开发焦点是：

1. 设计确认到工长确认的桥接能力
2. 履约后链路的验收、放款、归档、案例沉淀一致性

如果这两段打不穿，平台会停留在“流程很多、页面很多、但经营闭环不够深”的状态；如果这两段打穿，平台才真正具备交易基础设施价值。
