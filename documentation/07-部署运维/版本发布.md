# 版本发布

**最后更新**: 2026-01-25

本文档介绍了项目的 Git 工作流、版本发布流程以及在生产环境出现问题时的回滚方案。

## Git 工作流

本项目采用**双分支工作流**，确保开发与生产环境隔离。

### 分支策略
- **dev**: 开发分支。日常功能开发、修复 Bug 和集成测试都在此分支进行。
- **main**: 生产分支。仅包含经过充分测试、准备上线的稳定代码。

### 核心规则
1. 禁止直接在 `main` 分支进行日常开发。
2. 只有在测试环境通过验证的代码才能合并到 `main` 分支。
3. 每次发布到生产环境必须在 `main` 分支上打版本标签（Git Tag）。

---

## 发布流程

### 1. 准备发布
在合并到 `main` 之前，确保 `dev` 分支代码已在测试环境完成回归测试。

```bash
# 1. 切换到 main 分支
git checkout main
# 2. 拉取最新代码
git pull origin main
# 3. 合并 dev 分支
git merge dev
```

### 2. 创建版本标签
版本号遵循 `v主版本.次版本.补丁` (Semantic Versioning) 规范。

```bash
# 创建带注释的标签
git tag -a v1.0.0 -m "Release v1.0.0"

# 推送分支和标签到远程仓库
git push origin main
git push origin v1.0.0
```

### 3. 部署到生产
在生产服务器上拉取指定标签的代码并重新构建。

```bash
# 进入生产服务器代码目录
cd /www/home_decoration_prod

# 获取最新标签
git fetch --tags

# 切换到发布版本
git checkout v1.0.0

# 执行部署脚本（示例：使用 Docker Compose）
cd deploy
docker-compose -f docker-compose.prod.yml up -d --build
```

## 回滚方案

当生产环境出现紧急故障（如严重 Bug 或服务不可用）时，必须执行回滚。

### 快速回滚 (Git Revert)
如果故障是由单次提交引起的，且需要保留后续提交历史，可以使用 `git revert`。

```bash
# 撤销最近一次提交并生成新提交
git revert HEAD
git push origin main
```

### 版本回退 (Git Tag Checkout)
这是最稳健的回滚方式，直接将生产环境切回到上一个稳定的版本标签。

1. **查看历史标签**：
   ```bash
   git tag -l
   ```

2. **切换并部署**：
   ```bash
   # 假设上一个稳定版本是 v0.9.9
   git checkout v0.9.9

   # 重新启动服务
   cd deploy
   docker-compose -f docker-compose.prod.yml up -d --build
   ```

3. **验证回滚**：
   - 检查容器状态：`docker-compose ps`
   - 查看实时日志：`docker-compose logs -f --tail=100`
   - 执行健康检查：`curl http://localhost/api/health`
