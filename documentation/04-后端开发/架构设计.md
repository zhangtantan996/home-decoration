# 后端架构设计

**最后更新**: 2026-01-25

## 架构概览

本项目后端基于 Go 语言开发，采用 **Gin** 作为 Web 框架，**GORM** 作为 ORM 工具。架构设计遵循**严格的分层模式**（Layered Architecture），旨在实现关注点分离、提高代码的可维护性和可测试性。

### 核心技术栈
- **语言**: Go 1.21
- **Web 框架**: Gin v1.9.1
- **ORM**: GORM v1.25.5
- **数据库**: PostgreSQL 15
- **缓存**: Redis 6.2
- **认证**: JWT (golang-jwt/jwt/v5)

---

## 分层架构

后端代码严格遵循 `Handler -> Service -> Repository` 的三层架构模式。

### 1. Handler 层 (传输层)
**位置**: `server/internal/handler/`
**职责**:
- 解析 HTTP 请求参数（Query, Body, Path）。
- 验证请求数据的合法性（使用 Validator）。
- 调用对应的 Service 层函数执行业务逻辑。
- 封装并返回统一格式的 HTTP 响应（使用 `server/pkg/response`）。

**示例代码**:
```go
// server/internal/handler/user_handler.go
func (h *UserHandler) GetUserProfile(c *gin.Context) {
    // 1. 解析/验证参数
    userID, err := strconv.ParseUint(c.Param("id"), 10, 64)
    if err != nil {
        response.BadRequest(c, "无效的用户ID")
        return
    }

    // 2. 调用 Service 层
    user, err := h.userService.GetUserByID(c.Request.Context(), userID)
    if err != nil {
        response.ServerError(c, err.Error())
        return
    }

    // 3. 返回响应
    response.Success(c, user)
}
```

### 2. Service 层 (业务逻辑层)
**位置**: `server/internal/service/`
**职责**:
- 实现核心业务规则和逻辑。
- 协调多个 Repository 或外部服务（如短信、支付）。
- 管理数据库事务（Transaction）。
- 转换数据模型（Model -> DTO）。

**示例代码**:
```go
// server/internal/service/user_service.go
func (s *userService) GetUserByID(ctx context.Context, id uint64) (*model.User, error) {
    // 业务逻辑处理
    user, err := s.userRepo.FindByID(ctx, id)
    if err != nil {
        return nil, fmt.Errorf("查询用户失败: %w", err)
    }
    return user, nil
}
```

### 3. Repository 层 (数据访问层)
**位置**: `server/internal/repository/`
**职责**:
- 封装所有的数据库 CRUD 操作。
- 隐藏 GORM 的具体实现细节，为 Service 层提供抽象接口。
- 处理数据库特定的查询优化或缓存逻辑。

**示例代码**:
```go
// server/internal/repository/user_repository.go
func (r *userRepository) FindByID(ctx context.Context, id uint64) (*model.User, error) {
    var user model.User
    if err := r.db.WithContext(ctx).First(&user, id).Error; err != nil {
        return nil, err
    }
    return &user, nil
}
```

## 目录结构说明

```
server/
├── cmd/api/          # 程序入口，负责初始化配置、数据库连接和启动服务器
├── internal/         # 内部业务代码（外部不可引用）
│   ├── config/       # 配置定义与解析 (Viper)
│   ├── cron/         # 定时任务
│   ├── handler/      # HTTP 处理器
│   ├── middleware/   # Gin 中间件 (CORS, JWT, 日志, 限流)
│   ├── model/        # GORM 数据模型定义
│   ├── repository/   # 数据库访问层
│   ├── router/       # 路由注册与组织
│   ├── service/      # 业务逻辑层
│   └── utils/        # 业务相关的工具包 (如腾讯云 IM 集成)
├── pkg/              # 公共工具包（可被外部项目引用）
│   ├── response/     # 统一响应封装
│   └── utils/        # 通用工具 (加密、验证器)
├── migrations/       # 数据库迁移 SQL 脚本
└── scripts/          # 运维与初始化脚本 (数据初始化、DB 重置)
```

## 代码组织规范

### 1. 命名规范
- **文件命名**: 使用蛇形命名法（snake_case.go），如 `user_handler.go`。
- **接口命名**: Service 和 Repository 接口通常以 `Service` 或 `Repository` 结尾。
- **变量命名**: 使用小驼峰命名法（camelCase）。

### 2. 错误处理
- 禁止在请求处理路径中使用 `panic`。
- 函数应当返回 `error`，并使用 `%w` 包装上下文：`fmt.Errorf("service error: %w", err)`。
- 在 Handler 层统一进行错误分类并转换为 HTTP 状态码。

### 3. 依赖注入
- 采用简单的构造函数注入模式：`NewXXXHandler(service)` -> `NewXXXService(repo)` -> `NewXXXRepository(db)`。
- 依赖关系在 `cmd/api/main.go` 或专门的初始化函数中完成组装。

## 最佳实践

1. **职责单一**: 一个 Handler 对应的函数不应包含复杂的业务计算，一个 Repository 函数只负责简单的数据操作。
2. **模型隔离**: 尽管目前处于 MVP 阶段模型复用较多，但建议在复杂业务中区分 `model`（数据库表结构）和 `DTO`（接口传输对象）。
3. **安全第一**: 绝不硬编码任何密钥（JWT Secret, DB Password）。所有配置应通过 `config.yaml` 或环境变量加载。
4. **日志记录**: 关键业务节点必须记录日志，并包含必要的上下文信息（如 TraceID, UserID）。
5. **事务一致性**: 涉及多表更新的操作必须在 Service 层开启事务。
