# 内容审核 API

**最后更新**: 2026-01-25

## 概述
内容审核模块负责管理平台的内容质量与安全，包括案例审核、评论管理和敏感词管理。管理员可以审核商家提交的案例（新增/修改/删除），管理用户评论的状态，以及维护敏感词库以实现自动内容过滤。

## 业务流程

### 案例审核流程
1. 商家提交案例（新增/修改/删除）后，系统创建审核记录（`CaseAudit`），状态为 `pending`（待审核）。
2. 管理员在后台查看待审核列表，点击查看详情。
3. 管理员审核：
   - **通过**：案例数据写入/更新到 `ProviderCase` 表，审核记录状态变为 `approved`。
   - **拒绝**：填写拒绝原因，审核记录状态变为 `rejected`，商家可查看原因并重新提交。

### 评论管理流程
1. 用户在案例详情页发表评论，默认状态为 `approved`（已通过）。
2. 管理员可以修改评论状态：
   - `approved`：正常显示
   - `pending_review`：待审核（不显示）
   - `hidden`：隐藏（不显示）
   - `deleted`：已删除（软删除）

### 敏感词管理流程
1. 管理员可以单个添加敏感词，或批量导入敏感词库。
2. 系统在用户提交内容（评论、案例描述等）时，自动检测敏感词并执行相应动作（拦截/审核/替换）。
3. 支持正则表达式匹配，可以灵活配置敏感词规则。

## API 列表

### 案例审核管理

#### 1. 获取待审核案例列表
**接口**: `GET /api/v1/admin/audits/cases`

**描述**: 获取待审核的案例列表，支持按状态筛选和分页。

**请求头**:
```
Authorization: Bearer <admin_token>
```

**查询参数**:
| 参数名 | 类型 | 必填 | 说明 |
| :--- | :--- | :--- | :--- |
| page | int | 否 | 页码，默认 1 |
| pageSize | int | 否 | 每页数量，默认 10 |
| status | string | 否 | 审核状态：`0`-待审核（默认），`1`-已通过，`2`-已拒绝，`processed`-所有已处理 |

**响应示例**:
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "list": [
      {
        "id": 123,
        "caseId": 456,
        "providerId": 789,
        "providerName": "张三设计工作室",
        "actionType": "create",
        "title": "现代简约风格客厅设计",
        "status": 0,
        "createdAt": "2026-01-25T10:30:00Z"
      }
    ],
    "total": 25
  }
}
```

**字段说明**:
- `actionType`: 操作类型
  - `create`: 新增案例
  - `update`: 修改案例
  - `delete`: 删除案例
- `status`: 审核状态
  - `0`: pending（待审核）
  - `1`: approved（已通过）
  - `2`: rejected（已拒绝）

#### 3. 通过审核
**接口**: `POST /api/v1/admin/audits/cases/:id/approve`

**描述**: 通过指定的审核申请，根据操作类型执行相应动作（新增/修改/删除案例）。

**请求头**:
```
Authorization: Bearer <admin_token>
```

**路径参数**:
| 参数名 | 类型 | 必填 | 说明 |
| :--- | :--- | :--- | :--- |
| id | uint64 | 是 | 审核记录 ID |

**响应示例**:
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "message": "审核已通过"
  }
}
```

**业务逻辑**:
- `create`: 在 `ProviderCase` 表中创建新案例，`show_in_inspiration` 设为 `true`（审核通过的案例自动展示到灵感库）
- `update`: 更新 `ProviderCase` 表中的对应案例数据
- `delete`: 物理删除 `ProviderCase` 表中的对应案例
- 审核记录状态更新为 `1`（approved），记录审核人和审核时间

**错误响应**:
- `404`: 记录不存在
- `400`: 该记录已审核（不能重复审核）
- `400`: 作品风格不能为空（数据校验失败）
- `500`: 数据库操作失败

| 参数名 | 类型 | 必填 | 说明 |
| :--- | :--- | :--- | :--- |
| reason | string | 是 | 拒绝原因，商家可查看 |

**响应示例**:

**错误响应**:
- `404`: 记录不存在

#### 9. 批量导入敏感词
**接口**: `POST /api/v1/admin/sensitive-words/import`

**描述**: 批量导入敏感词库，支持 TXT 和 CSV 格式。

**请求头**:
```
Authorization: Bearer <admin_token>
Content-Type: multipart/form-data
```

**请求参数**:
| 参数名 | 类型 | 必填 | 说明 |
| :--- | :--- | :--- | :--- |
| file | file | 是 | 敏感词文件（TXT 或 CSV） |
| defaultCategory | string | 否 | 默认分类（CSV 中未指定时使用） |
| defaultLevel | string | 否 | 默认严重程度，默认 `normal` |
| defaultAction | string | 否 | 默认触发动作，默认 `review` |

**文件格式说明**:

**1. TXT 格式**（每行一个词）:
```
违禁词1
违禁词2
re:正则表达式.*匹配
# 这是注释，会被忽略
// 这也是注释
```
- 每行一个敏感词
- 以 `re:` 开头表示该行是正则表达式
- 以 `#` 或 `//` 开头的行为注释，会被忽略
- 空行会被忽略

**2. CSV 格式**（逗号分隔）:
```
word,category,level,action,is_regex
违禁词1,政治,high,block,false
违禁词2,色情,normal,review,false
正则.*匹配,广告,low,replace,true
```
- 第一行可以是标题行（会被当作普通数据处理，建议不加标题行）
- 格式：`word,category,level,action,is_regex`
- `is_regex` 支持：`1`/`true`/`yes`/`y` 表示 true，`0`/`false`/`no`/`n` 表示 false

**响应示例**:
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "message": "导入完成",
    "lines": 1000,
    "parsed": 950,
    "inserted": 920
  }
}
```

**字段说明**:
- `lines`: 文件总行数
- `parsed`: 成功解析的行数（排除空行、注释、格式错误）
- `inserted`: 实际插入数据库的记录数（已存在的词会被跳过，使用 `ON CONFLICT DO NOTHING` 策略）

**导入规则**:
- 使用 `ON CONFLICT DO NOTHING` 策略，已存在的敏感词不会被覆盖（保护管理员手工调整的规则）
- 批量插入，每 1000 条提交一次事务，提高性能
- 支持大文件导入（单行最大 1MB，总文件无限制）

**错误响应**:
- `400`: 请上传文件(file)
- `500`: 读取文件失败
- `500`: 导入失败

## 数据模型

### CaseAudit（案例审核记录）
```go
type CaseAudit struct {
    ID             uint64     // 审核记录 ID
    CaseID         *uint64    // 关联的案例 ID（新增时为空）
    ProviderID     uint64     // 商家 ID
    ActionType     string     // 操作类型：create/update/delete
    
    // 案例数据快照
    Title          string     // 标题
    CoverImage     string     // 封面图
    Style          string     // 风格
    Layout         string     // 户型
    Area           string     // 面积
    Price          float64    // 价格
    QuoteTotalCent int64      // 报价总计（分）
    QuoteCurrency  string     // 币种（默认 CNY）
    QuoteItems     string     // 报价明细（JSONB）
    Year           string     // 年份
    Description    string     // 描述
    Images         string     // 图片数组（JSON）
    
    // 审核状态
    Status         int8       // 0:pending, 1:approved, 2:rejected
    RejectReason   string     // 拒绝原因
    AuditedBy      uint64     // 审核人 ID
    AuditedAt      *time.Time // 审核时间
    
    CreatedAt      time.Time
    UpdatedAt      time.Time
}
```

### CaseComment（案例评论）
```go
type CaseComment struct {
    ID            uint64     // 评论 ID
    CaseID        uint64     // 案例 ID
    UserID        uint64     // 用户 ID
    Content       string     // 评论内容
    Status        string     // 状态：approved/pending_review/hidden/deleted
    RejectReason  string     // 拒绝原因
    ModeratedBy   *uint64    // 审核人 ID
    ModeratedAt   *time.Time // 审核时间
    ReportCount   int        // 被举报次数
    
    CreatedAt     time.Time
    UpdatedAt     time.Time
}
```

### SensitiveWord（敏感词）
```go
type SensitiveWord struct {
    ID        uint64    // 敏感词 ID
    Word      string    // 敏感词内容（唯一索引）
    Category  string    // 分类
    Level     string    // 严重程度：low/normal/high
    Action    string    // 触发动作：block/review/replace
    IsRegex   bool      // 是否为正则表达式
    
    CreatedAt time.Time
    UpdatedAt time.Time
}
```

## 错误码说明

| 错误码 | 说明 | 备注 |
| :--- | :--- | :--- |
| 400 | 参数错误 | 请求参数不合法或缺少必填字段 |
| 401 | 认证失败 | Token 过期或无效 |
| 403 | 权限不足 | 非管理员用户无法访问 |
| 404 | 记录不存在 | 指定的审核记录/评论/敏感词不存在 |
| 500 | 服务器错误 | 数据库操作失败或其他内部错误 |

## 注意事项

1. **权限控制**: 所有接口都需要管理员权限，需在请求头中携带有效的管理员 Token。
2. **审核幂等性**: 已审核的记录不能重复审核，避免数据不一致。
3. **敏感词导入**: 使用 `ON CONFLICT DO NOTHING` 策略，已存在的敏感词不会被覆盖，保护管理员手工调整的规则。
4. **评论状态**: 评论状态修改后立即生效，前端需要实时刷新评论列表。
5. **案例审核**: 审核通过的案例会自动设置 `show_in_inspiration = true`，展示到灵感库。
6. **正则表达式**: 使用正则表达式时需谨慎，避免性能问题和误匹配。建议先在测试环境验证。
7. **批量导入性能**: 批量导入采用分批提交策略（每 1000 条），支持大文件导入，但建议单次导入不超过 10 万条。
