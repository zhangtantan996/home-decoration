# 业务流程图

**最后更新**: 2026-01-25

本文档使用 Mermaid 图表展示装修设计一体化平台的核心业务流程，包括用户认证、预约到订单、项目进度管理以及商家入驻等关键业务场景。每个流程图标注了相关的 API 端点，便于开发人员理解系统交互逻辑。

---

## 1. 用户认证流程

用户认证流程涵盖注册、登录（验证码/密码）、Token 刷新以及微信小程序登录等场景。

### 1.1 注册与登录流程

```mermaid
sequenceDiagram
    participant C as 客户端
    participant API as API服务器
    participant DB as 数据库
    participant SMS as 短信服务

    %% 注册流程
    rect rgb(240, 248, 255)
        Note over C,SMS: 用户注册流程
        C->>+API: POST /api/v1/auth/send-code<br/>{phone: "13800138000"}
        API->>SMS: 发送验证码
        SMS-->>API: 发送成功
        API-->>-C: {code: 0, message: "验证码已发送"}
        
        C->>+API: POST /api/v1/auth/register<br/>{phone, code, password?, nickname?}
        API->>API: 验证验证码
        API->>DB: 检查手机号是否已注册
        DB-->>API: 未注册
        API->>DB: 创建用户记录
        API->>API: 生成 JWT Token + RefreshToken
        API-->>-C: {token, refreshToken, expiresIn, user}
    end

    %% 登录流程
    rect rgb(255, 250, 240)
        Note over C,DB: 用户登录流程（验证码方式）
        C->>+API: POST /api/v1/auth/send-code<br/>{phone}
        API->>SMS: 发送验证码
        API-->>-C: 验证码已发送
        
        C->>+API: POST /api/v1/auth/login<br/>{phone, type: "code", code}
        API->>API: 验证验证码
        API->>DB: 查询用户信息
        DB-->>API: 返回用户数据
        API->>API: 生成 JWT Token + RefreshToken
        API-->>-C: {token, refreshToken, expiresIn, user}
    end

    %% 密码登录
    rect rgb(240, 255, 240)
        Note over C,DB: 用户登录流程（密码方式）
        C->>+API: POST /api/v1/auth/login<br/>{phone, type: "password", password}
        API->>DB: 查询用户并验证密码
        DB-->>API: 验证通过
        API->>API: 生成 JWT Token + RefreshToken
        API-->>-C: {token, refreshToken, expiresIn, user}
    end
```

### 1.2 Token 刷新流程

```mermaid
sequenceDiagram
    participant C as 客户端
    participant API as API服务器
    participant Redis as Redis缓存

    C->>+API: 业务请求（携带过期 Token）
    API-->>-C: {code: 401, message: "Token已过期"}
    
    C->>+API: POST /api/v1/auth/refresh<br/>{refreshToken}
    API->>API: 验证 RefreshToken 签名
    API->>Redis: 检查 RefreshToken 是否在黑名单
    Redis-->>API: 未被撤销
    API->>API: 生成新的 Token + RefreshToken
    API->>Redis: 将旧 RefreshToken 加入黑名单
    API-->>-C: {token, refreshToken, expiresIn}
    
    C->>API: 重新发起业务请求（携带新 Token）
    API-->>C: 业务数据
```

### 1.3 微信小程序登录流程

```mermaid
sequenceDiagram
    participant Mini as 小程序
    participant API as API服务器
    participant WX as 微信服务器
    participant DB as 数据库

    Mini->>Mini: wx.login() 获取 code
    Mini->>+API: POST /api/v1/auth/wechat/mini/login<br/>{code}
    API->>+WX: 请求 openid 和 session_key
    WX-->>-API: {openid, session_key}
    
    API->>DB: 根据 openid 查询用户
    
    alt 用户已绑定手机号
        DB-->>API: 返回用户信息
        API->>API: 生成 JWT Token
        API-->>-Mini: {token, user}
    else 用户未绑定手机号
        DB-->>API: 用户不存在
        API-->>-Mini: {needBindPhone: true, tempToken}
        
        Mini->>Mini: 引导用户授权手机号
        Mini->>+API: POST /api/v1/auth/wechat/mini/bind-phone<br/>{tempToken, encryptedData, iv}
        API->>WX: 解密手机号
        WX-->>API: 手机号明文
        API->>DB: 创建/绑定用户
        API->>API: 生成正式 Token
        API-->>-Mini: {token, user}
    end
```

## 2. 预约到订单流程

从用户发起预约、支付意向金，到服务商提交方案、用户确认方案并生成订单的完整流程。

```mermaid
sequenceDiagram
    participant U as 用户端
    participant API as API服务器
    participant M as 商家端
    participant Pay as 支付系统
    participant DB as 数据库

    %% 创建预约
    rect rgb(240, 248, 255)
        Note over U,DB: 阶段1: 创建预约并支付意向金
        U->>+API: POST /api/v1/bookings<br/>{providerId, serviceType, requirements}
        API->>DB: 创建预约记录（status: pending）
        DB-->>API: bookingId
        API-->>-U: {bookingId, intentFee: 99}
        
        U->>+API: POST /api/v1/bookings/:id/pay-intent<br/>支付意向金
        API->>Pay: 调用支付接口
        Pay-->>API: 支付成功
        API->>DB: 更新预约状态（status: pending）
        API->>M: 推送预约通知
        API-->>-U: 支付成功
    end

    %% 服务商接单
    rect rgb(255, 250, 240)
        Note over M,DB: 阶段2: 服务商接单
        M->>+API: GET /api/v1/merchant/bookings
        API-->>-M: 预约列表
        
        M->>+API: PUT /api/v1/merchant/bookings/:id/handle<br/>{action: "accept"}
        API->>DB: 更新预约状态（status: confirmed）
        API->>U: 推送接单通知
        API-->>-M: 接单成功
    end

    %% 提交方案
    rect rgb(240, 255, 240)
        Note over M,DB: 阶段3: 服务商提交方案
        M->>+API: POST /api/v1/merchant/proposals<br/>{bookingId, designFee, description, attachments}
        API->>DB: 创建方案记录（status: pending）
        DB-->>API: proposalId
        API->>U: 推送方案通知
        API-->>-M: 方案已提交
    end

    %% 用户确认方案
    rect rgb(255, 248, 240)
        Note over U,DB: 阶段4: 用户确认方案
        U->>+API: GET /api/v1/proposals
        API-->>-U: 待确认方案列表
        
        U->>+API: GET /api/v1/proposals/:id
        API-->>-U: 方案详情
        
        alt 用户接受方案
            U->>+API: POST /api/v1/proposals/:id/confirm
            API->>DB: 更新方案状态（status: confirmed）
            API->>DB: 生成订单（Order）
            API->>DB: 更新预约状态（status: in_progress）
            API->>M: 推送方案确认通知
            API-->>-U: {orderId, totalAmount}
        else 用户拒绝方案
            U->>+API: POST /api/v1/proposals/:id/reject<br/>{reason}
            API->>DB: 更新方案状态（status: rejected）
            API->>M: 推送拒绝通知（含原因）
            API-->>-U: 拒绝成功
            Note over M: 商家可重新提交方案
        end
    end

    %% 支付订单
    rect rgb(255, 240, 240)
        Note over U,Pay: 阶段5: 支付订单
        U->>+API: GET /api/v1/orders/pending-payments
        API-->>-U: 待支付订单列表
        
        U->>+API: POST /api/v1/orders/:id/pay<br/>{paymentMethod}
        API->>Pay: 调用支付接口
        Pay-->>API: 支付成功
        API->>DB: 更新订单状态（status: paid）
        API->>DB: 创建托管账户记录
        API->>M: 推送支付成功通知
        API-->>-U: 支付成功
    end
```

**关键状态转换**:
- 预约状态: `pending` → `confirmed` → `in_progress` → `completed` / `cancelled`
- 方案状态: `pending` → `confirmed` / `rejected`
- 订单状态: `pending_payment` → `paid` → `settled` / `refunded`

## 3. 项目进度流程

项目从启动、施工、节点验收到资金释放的完整生命周期管理。

```mermaid
sequenceDiagram
    participant U as 用户端
    participant API as API服务器
    participant M as 商家端
    participant Escrow as 托管账户
    participant DB as 数据库

    %% 创建项目
    rect rgb(240, 248, 255)
        Note over U,DB: 阶段1: 创建项目
        U->>+API: POST /api/v1/projects<br/>{orderId, startDate, phases}
        API->>DB: 创建项目记录（status: pending）
        API->>DB: 创建项目阶段（phases）
        API-->>-U: {projectId}
    end

    %% 首期款充值
    rect rgb(255, 250, 240)
        Note over U,Escrow: 阶段2: 充值到托管账户
        U->>+API: POST /api/v1/projects/:id/deposit<br/>{amount, phase: "开工"}
        API->>Escrow: 创建托管账户
        API->>DB: 记录充值交易
        API->>DB: 更新项目状态（status: in_progress）
        API->>M: 推送开工通知
        API-->>-U: 充值成功
    end

    %% 施工与日志
    rect rgb(240, 255, 240)
        Note over M,DB: 阶段3: 施工进度汇报
        loop 每个施工阶段
            M->>+API: POST /api/v1/projects/:id/logs<br/>{phaseId, content, images}
            API->>DB: 创建施工日志
            API->>U: 推送进度更新通知
            API-->>-M: 日志已记录
        end
    end

    %% 节点验收
    rect rgb(255, 248, 240)
        Note over M,U: 阶段4: 节点验收
        M->>+API: PUT /api/v1/phases/:phaseId<br/>{status: "pending_acceptance"}
        API->>DB: 更新阶段状态
        API->>U: 推送验收申请通知
        API-->>-M: 验收申请已提交
        
        U->>+API: GET /api/v1/projects/:id/milestones
        API-->>-U: 待验收节点列表
        
        alt 验收通过
            U->>+API: POST /api/v1/projects/:id/accept<br/>{phaseId, rating, comment}
            API->>DB: 更新阶段状态（status: completed）
            API->>Escrow: 释放该阶段资金
            API->>DB: 记录资金释放交易
            API->>M: 推送验收通过 + 资金到账通知
            API-->>-U: 验收成功
        else 验收不通过
            U->>+API: POST /api/v1/projects/:id/accept<br/>{phaseId, passed: false, reason}
            API->>DB: 更新阶段状态（status: rework）
            API->>M: 推送整改通知（含原因）
            API-->>-U: 已要求整改
            Note over M: 商家整改后重新申请验收
        end
    end

    %% 项目完工
    rect rgb(240, 255, 255)
        Note over U,Escrow: 阶段5: 项目完工与尾款结算
        M->>+API: PUT /api/v1/phases/:phaseId<br/>{status: "pending_acceptance"}<br/>（竣工阶段）
        API->>U: 推送竣工验收通知
        API-->>-M: 竣工验收申请已提交
        
        U->>+API: POST /api/v1/projects/:id/accept<br/>{phaseId: "竣工", rating}
        API->>DB: 更新项目状态（status: completed）
        API->>Escrow: 释放尾款（扣除质保金）
        API->>DB: 记录交易
        API->>M: 推送项目完工通知
        API-->>-U: 项目已完工
        
        Note over Escrow: 质保金在质保期满后释放
    end
```

**关键 API 端点**:
- `POST /api/v1/projects` - 创建项目
- `POST /api/v1/projects/:id/deposit` - 充值到托管账户
- `POST /api/v1/projects/:id/logs` - 创建施工日志
- `GET /api/v1/projects/:id/milestones` - 获取项目里程碑
- `POST /api/v1/projects/:id/accept` - 节点验收
- `POST /api/v1/projects/:id/release` - 释放资金

## 4. 商家入驻流程

服务商（设计师/装修公司/工长）申请入驻平台的审核流程。

```mermaid
sequenceDiagram
    participant M as 商家
    participant API as API服务器
    participant Admin as 管理后台
    participant DB as 数据库
    participant SMS as 短信服务

    %% 提交申请
    rect rgb(240, 248, 255)
        Note over M,DB: 阶段1: 提交入驻申请
        M->>+API: POST /api/v1/merchant/apply<br/>{phone, name, type, businessLicense, idCard}
        API->>DB: 创建申请记录（status: pending）
        DB-->>API: applicationId
        API->>Admin: 推送审核通知
        API-->>-M: {applicationId, message: "申请已提交"}
    end

    %% 查询申请状态
    rect rgb(255, 250, 240)
        Note over M,API: 阶段2: 查询申请状态
        M->>+API: GET /api/v1/merchant/apply/:phone/status
        API->>DB: 查询申请记录
        DB-->>API: 申请详情
        API-->>-M: {status: "pending", submitTime, reviewTime?}
    end

    %% 管理员审核
    rect rgb(240, 255, 240)
        Note over Admin,DB: 阶段3: 管理员审核
        Admin->>+API: GET /api/v1/admin/merchant-applications
        API-->>-Admin: 待审核申请列表
        
        Admin->>+API: GET /api/v1/admin/merchant-applications/:id
        API-->>-Admin: 申请详情（含资质文件）
        
        alt 审核通过
            Admin->>+API: POST /api/v1/admin/merchant-applications/:id/approve<br/>{initialPassword}
            API->>DB: 更新申请状态（status: approved）
            API->>DB: 创建商家账号（Provider）
            API->>SMS: 发送开通通知（含初始密码）
            API->>M: 推送审核通过通知
            API-->>-Admin: 审核成功
        else 审核拒绝
            Admin->>+API: POST /api/v1/admin/merchant-applications/:id/reject<br/>{reason}
            API->>DB: 更新申请状态（status: rejected）
            API->>SMS: 发送拒绝通知（含原因）
            API->>M: 推送审核拒绝通知
            API-->>-Admin: 已拒绝
        end
    end

    %% 商家登录
    rect rgb(255, 248, 240)
        Note over M,API: 阶段4: 商家首次登录
        M->>+API: POST /api/v1/merchant/login<br/>{phone, password: "初始密码"}
        API->>DB: 验证商家账号
        DB-->>API: 验证通过
        API->>API: 生成 JWT Token（type: merchant）
        API-->>-M: {token, merchantInfo}
        
        M->>+API: PUT /api/v1/merchant/info<br/>{newPassword, ...}
        API->>DB: 更新商家信息
        API-->>-M: 信息更新成功
    end

    %% 补充资料
    rect rgb(240, 255, 255)
        Note over M,DB: 阶段5: 完善商家资料
        M->>+API: POST /api/v1/merchant/avatar<br/>上传头像
        API-->>-M: 头像URL
        
        M->>+API: POST /api/v1/merchant/cases<br/>上传作品集
        API->>DB: 创建作品记录（status: pending_audit）
        API->>Admin: 推送作品审核通知
        API-->>-M: 作品已提交审核
        
        Note over M: 商家可正常接单
    end
```

**关键状态转换**:
- 申请状态: `pending` → `approved` / `rejected`
- 商家状态: `inactive` → `active` / `suspended`
- 作品审核: `pending_audit` → `approved` / `rejected`

**关键 API 端点**:
- `POST /api/v1/merchant/apply` - 提交入驻申请
- `GET /api/v1/merchant/apply/:phone/status` - 查询申请状态
- `POST /api/v1/merchant/apply/:id/resubmit` - 重新提交申请（被拒后）
- `GET /api/v1/admin/merchant-applications` - 管理员查看申请列表
- `POST /api/v1/admin/merchant-applications/:id/approve` - 审核通过
- `POST /api/v1/admin/merchant-applications/:id/reject` - 审核拒绝
- `POST /api/v1/merchant/login` - 商家登录

## 流程图使用说明

### Mermaid 语法
本文档使用 Mermaid 的 `sequenceDiagram` 语法绘制时序图，支持在 GitHub、GitLab、VS Code（需安装插件）等平台直接渲染。

### 图例说明
- **矩形框（rect）**: 表示业务阶段或逻辑分组
- **实线箭头（->>）**: 同步请求
- **虚线箭头（-->>）**: 响应返回
- **alt/else**: 条件分支（如审核通过/拒绝）
- **loop**: 循环操作（如多次施工日志）

### 在线预览
如需在线预览或编辑流程图，可使用以下工具：
- [Mermaid Live Editor](https://mermaid.live/)
- VS Code 插件: `Markdown Preview Mermaid Support`
