# 方案模块 API（用户端）

**最后更新**: 2026-01-25

## 概述
方案模块负责处理用户端的设计方案管理，包括查看方案列表、获取方案详情、确认方案、拒绝方案以及查看版本历史。用户确认方案后会自动创建设计费订单，需在 48 小时内完成支付。拒绝方案后，商家可重新提交新版本方案（版本号递增）。

## 方案状态流转
```
pending（待处理）
    ↓
    ├─→ confirmed（已确认）→ 创建设计费订单 → 48小时支付窗口
    └─→ rejected（已拒绝）→ 商家可重新提交 → 版本号递增（v1 → v2 → v3）
```

## 版本管理说明
- 每次商家提交方案时，系统会分配版本号（v1, v2, v3...）
- 用户拒绝方案后，商家可重新提交，生成新版本
- 系统记录每个版本的拒绝原因和拒绝时间
- 同一预约下的所有方案版本可通过版本历史接口查询
- 累计拒绝 3 次后，预约进入争议处理流程

| confirmedAt | string | 确认时间（ISO 8601 格式） |
| submittedAt | string | 提交时间（ISO 8601 格式） |

### 2. 获取待处理方案数量
**接口**: `GET /api/v1/proposals/pending-count`

**描述**: 获取当前用户待处理（status=1）的方案数量，用于首页红点提示。

**认证**: 需要 Bearer Token（用户身份）

**请求参数**: 无

**响应示例**:
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "count": 3
  }
}
```

### 5. 确认方案
**接口**: `POST /api/v1/proposals/:id/confirm`

**描述**: 用户确认设计方案，系统自动创建设计费订单，用户需在 48 小时内完成支付。确认后方案状态变更为 `confirmed`（2），同一预约下的其他待处理方案自动失效。

**认证**: 需要 Bearer Token（用户身份）

**路径参数**:
| 参数名 | 类型 | 必填 | 说明 |
| :--- | :--- | :--- | :--- |
| id | uint64 | 是 | 方案ID |

**请求参数**: 无

**响应示例**:
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "order": {
      "id": 50,
      "proposalId": 1,
      "bookingId": 10,
      "orderNo": "ORD202601251030001",
      "orderType": "design",
      "totalAmount": 5000.00,
      "paidAmount": 0.00,
      "discount": 99.00,
      "status": 0,
      "expireAt": "2026-01-27T10:30:00Z",
      "createdAt": "2026-01-25T10:30:00Z"
    },
    "message": "请在48小时内完成设计费支付"
  }
}
```

**业务规则**:
1. 只能确认 `status=1`（待确认）的方案
2. 确认后自动创建设计费订单，订单金额为 `designFee`
3. 如果预约时支付了意向金，会自动抵扣到设计费订单中（`discount` 字段）
4. 订单有效期为 48 小时（`expireAt`），超时未支付订单自动取消
5. 确认后同一预约下的其他待处理方案自动失效（`status=4`）

**错误码**:
| 错误码 | 说明 |
| :--- | :--- |
| 400 | 方案不存在 / 方案状态不正确 / 无权操作此方案 |
| 404 | 方案不存在 |

### 6. 拒绝方案
**接口**: `POST /api/v1/proposals/:id/reject`

**描述**: 用户拒绝设计方案并提供拒绝原因，商家可根据反馈重新提交新版本方案。拒绝后方案状态变更为 `rejected`（3）。

**认证**: 需要 Bearer Token（用户身份）

**路径参数**:
| 参数名 | 类型 | 必填 | 说明 |
| :--- | :--- | :--- | :--- |
| id | uint64 | 是 | 方案ID |

**请求参数**:
| 参数名 | 类型 | 必填 | 说明 |
| :--- | :--- | :--- | :--- |
| reason | string | 是 | 拒绝原因（5-500 字符） |

**请求示例**:
```json
{
  "reason": "预算超出预期，希望能优化材料成本，将总费用控制在 18 万以内。另外施工周期希望能缩短到 50 天左右。"
}
```

**响应示例**:
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "message": "方案已拒绝，商家可重新提交"
  }
}
```

**业务规则**:
1. 只能拒绝 `status=1`（待确认）的方案
2. 拒绝原因必填，长度 5-500 字符
3. 拒绝后方案状态变更为 `rejected`（3），记录拒绝原因和拒绝时间
4. 系统累计该预约的拒绝次数（`rejectionCount`）
5. 商家可查看拒绝原因并重新提交新版本方案（版本号递增）
6. 累计拒绝 3 次后，预约进入争议处理流程，不再允许重新提交

**错误码**:
| 错误码 | 说明 |
| :--- | :--- |
| 400 | 参数错误 / 方案不存在 / 方案状态不正确 / 无权操作此方案 / 拒绝原因长度不符合要求 |
| 404 | 方案不存在 |

---

## 错误码说明

| 错误码 | 说明 | 备注 |
| :--- | :--- | :--- |
| 400 | 参数错误 | 请求参数不合法或业务规则校验失败 |
| 401 | 认证失败 | Token 过期或无效 |
| 403 | 无权操作 | 尝试操作非本人的方案 |
| 404 | 资源不存在 | 方案或预约不存在 |
| 500 | 服务器错误 | 系统内部错误 |

## 使用示例

### 获取方案列表 (Curl)
```bash
curl -X GET http://localhost:8080/api/v1/proposals \
  -H "Authorization: Bearer <your_token_here>"
```

### 确认方案 (Curl)
```bash
curl -X POST http://localhost:8080/api/v1/proposals/1/confirm \
  -H "Authorization: Bearer <your_token_here>"
```

### 拒绝方案 (Curl)
```bash
curl -X POST http://localhost:8080/api/v1/proposals/1/reject \
  -H "Authorization: Bearer <your_token_here>" \
  -H "Content-Type: application/json" \
  -d '{
    "reason": "预算超出预期，希望能优化材料成本"
  }'
```

### 获取版本历史 (Curl)
```bash
curl -X GET http://localhost:8080/api/v1/proposals/booking/10/history \
  -H "Authorization: Bearer <your_token_here>"
```

## 注意事项

1. **支付时效性**: 确认方案后必须在 48 小时内完成设计费支付，否则订单自动取消，需重新确认方案。
2. **版本管理**: 每次拒绝后商家重新提交的方案会生成新版本号，建议通过版本历史接口查看完整沟通记录。
3. **争议处理**: 累计拒绝 3 次后，预约进入争议处理流程，需要平台客服介入协调。
4. **意向金抵扣**: 如果预约时支付了意向金，确认方案后会自动抵扣到设计费订单中。
5. **权限校验**: 所有接口都会校验方案归属，只能操作本人预约关联的方案。
