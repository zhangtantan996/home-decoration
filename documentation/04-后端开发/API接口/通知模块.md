# 通知模块 API

**最后更新**: 2026-01-25

## 概述
通知模块负责处理站内消息推送，支持用户、商家、管理员三种角色。系统在关键业务节点（预约、方案、订单、支付、审核等）自动触发通知，用户可查看、标记已读、删除通知。

## 路径前缀说明
三种角色共用相同的 handler 实现，但路径前缀不同：

| 角色 | 路径前缀 | 认证中间件 | 说明 |
| :--- | :--- | :--- | :--- |
| 用户 | `/api/v1/notifications` | JWT | 普通用户（业主） |
| 商家 | `/api/v1/merchant/notifications` | MerchantJWT | 服务商（设计师/装修公司/工长） |
| 管理员 | `/api/v1/admin/notifications` | AdminJWT | 后台管理员 |

**实现机制**：Handler 通过检查 `admin_id`、`providerId`、`userId` 上下文字段自动识别角色类型（`userType`），查询对应的通知数据。

## 通知类型枚举

| 类型常量 | 值 | 说明 | 接收角色 |
| :--- | :--- | :--- | :--- |
| `booking.intent_paid` | 意向金已支付 | 用户支付意向金后通知商家 | 商家、管理员 |
| `booking.confirmed` | 预约已确认 | 商家接单后通知用户 | 用户 |
| `booking.cancelled` | 预约已取消 | 预约取消通知 | 用户、商家 |
| `booking.intent_refunded` | 意向金已退款 | 意向金退款成功通知用户 | 用户 |
| `proposal.submitted` | 方案已提交 | 商家提交设计方案通知用户 | 用户 |
| `proposal.confirmed` | 方案已确认 | 用户确认方案通知商家 | 商家 |
| `proposal.rejected` | 方案被拒绝 | 用户拒绝方案通知商家（含拒绝原因） | 商家 |
| `order.created` | 订单已生成 | 设计费订单生成通知用户 | 用户 |
| `order.paid` | 订单已支付 | 用户支付订单通知商家 | 商家 |
| `order.expiring` | 订单即将过期 | 订单支付期限临近提醒 | 用户 |
| `order.expired` | 订单已过期 | 订单超时未支付自动取消 | 用户 |
| `withdraw.approved` | 提现审核通过 | 商家提现申请审核通过 | 商家 |
| `withdraw.rejected` | 提现审核拒绝 | 商家提现申请被拒绝（含原因） | 商家 |
| `withdraw.completed` | 提现已完成 | 提现到账通知 | 商家 |
| `audit.approved` | 审核通过 | 服务商/门店审核通过 | 商家 |
| `audit.rejected` | 审核拒绝 | 服务商/门店审核拒绝（含原因） | 商家 |
| `case_audit.approved` | 作品审核通过 | 作品审核通过可展示 | 商家 |
| `case_audit.rejected` | 作品审核拒绝 | 作品审核拒绝（含原因） | 商家 |

## API 列表

### 1. 获取通知列表
**接口**: `GET /notifications`

**描述**: 分页获取当前用户的通知列表，按创建时间倒序排列。

**认证**: 需要 JWT Token（根据角色使用对应的认证中间件）

**请求参数**:
| 参数名 | 类型 | 必填 | 默认值 | 说明 |
| :--- | :--- | :--- | :--- | :--- |
| page | int | 否 | 1 | 页码 |
| pageSize | int | 否 | 20 | 每页数量 |

**响应示例**:
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "list": [
      {
        "id": 123,
        "userId": 456,
        "userType": "user",
        "title": "商家已接单",
        "content": "商家已确认接单，请等待方案提交",
        "type": "booking.confirmed",
        "relatedId": 789,
        "relatedType": "booking",
        "isRead": false,
        "readAt": null,
        "actionUrl": "/bookings/789",
        "extra": "{\"bookingId\":789,\"providerId\":101}",
        "createdAt": "2026-01-25T10:30:00Z",
        "updatedAt": "2026-01-25T10:30:00Z"
      },
      {
        "id": 122,
        "userId": 456,
        "userType": "user",
        "title": "设计方案已提交",
        "content": "商家已提交设计方案，请查看并确认",
        "type": "proposal.submitted",
        "relatedId": 88,
        "relatedType": "proposal",
        "isRead": true,
        "readAt": "2026-01-25T11:00:00Z",
        "actionUrl": "/proposals/88",
        "extra": "{\"proposalId\":88,\"bookingId\":789}",
        "createdAt": "2026-01-24T15:20:00Z",
        "updatedAt": "2026-01-25T11:00:00Z"
      }
    ],
    "total": 45,
    "page": 1,
    "pageSize": 20
  }
}
```

**字段说明**:
- `userId`: 通知接收者 ID
- `userType`: 接收者类型（`user`/`provider`/`admin`）
- `type`: 通知类型（见通知类型枚举表）
- `relatedId`: 关联业务对象 ID（如预约 ID、订单 ID）
- `relatedType`: 关联对象类型（`booking`/`proposal`/`order`/`withdraw`）
- `isRead`: 是否已读
- `readAt`: 已读时间（未读时为 `null`）
- `actionUrl`: 跳转链接（前端路由路径）
- `extra`: 扩展数据（JSON 字符串，包含业务相关字段）

### 2. 获取未读数量
**接口**: `GET /notifications/unread-count`

**描述**: 获取当前用户的未读通知数量，用于显示红点角标。

**认证**: 需要 JWT Token

**响应示例**:
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "count": 3
  }
}
```

---

### 3. 标记单个通知为已读
**接口**: `PUT /notifications/:id/read`

**描述**: 将指定通知标记为已读状态。

**认证**: 需要 JWT Token

**路径参数**:
| 参数名 | 类型 | 必填 | 说明 |
| :--- | :--- | :--- | :--- |
| id | uint64 | 是 | 通知 ID |

**响应示例**:
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "message": "标记成功"
  }
}
```

**错误响应**:
```json
{
  "code": 400,
  "message": "无效的通知ID"
}
```

```json
{
  "code": 500,
  "message": "通知不存在"
}
```

**业务规则**:
- 只能标记属于当前用户的通知
- 重复标记已读的通知不会报错
- 标记成功后更新 `isRead` 为 `true`，`readAt` 为当前时间

---

### 4. 标记全部通知为已读
**接口**: `PUT /notifications/read-all`

**描述**: 将当前用户的所有未读通知批量标记为已读。

**认证**: 需要 JWT Token

**响应示例**:
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "message": "全部已读"
  }
}
```

**业务规则**:
- 只更新 `isRead = false` 的通知
- 批量更新 `isRead` 为 `true`，`readAt` 为当前时间
- 如果没有未读通知，操作仍然成功返回

---

### 5. 删除通知
**接口**: `DELETE /notifications/:id`

**描述**: 删除指定通知（软删除或物理删除，取决于实现）。

**认证**: 需要 JWT Token

**路径参数**:
| 参数名 | 类型 | 必填 | 说明 |
| :--- | :--- | :--- | :--- |
| id | uint64 | 是 | 通知 ID |

**响应示例**:
```json
{
  "code": 0,
  "message": "success",
  "data": {
    "message": "删除成功"
  }
}
```

**错误响应**:
```json
{
  "code": 400,
  "message": "无效的通知ID"
}
```

```json
{
  "code": 500,
  "message": "通知不存在"
}
```

**业务规则**:
- 只能删除属于当前用户的通知
- 删除后通知不可恢复
- 删除不存在的通知返回错误

---

## 错误码说明

| 错误码 | 说明 | 备注 |
| :--- | :--- | :--- |
| 400 | 参数错误 | 通知 ID 格式不正确 |
| 401 | 认证失败 | Token 过期或无效 |
| 403 | 权限不足 | 尝试操作其他用户的通知 |
| 500 | 服务器错误 | 通知不存在或数据库操作失败 |

## 使用示例

### 获取通知列表（用户端）
```bash
curl -X GET "http://localhost:8080/api/v1/notifications?page=1&pageSize=10" \
  -H "Authorization: Bearer <user_token>"
```

### 获取通知列表（商家端）
```bash
curl -X GET "http://localhost:8080/api/v1/merchant/notifications?page=1&pageSize=10" \
  -H "Authorization: Bearer <merchant_token>"
```

### 获取通知列表（管理员）
```bash
curl -X GET "http://localhost:8080/api/v1/admin/notifications?page=1&pageSize=10" \
  -H "Authorization: Bearer <admin_token>"
```

### 获取未读数量
```bash
curl -X GET "http://localhost:8080/api/v1/notifications/unread-count" \
  -H "Authorization: Bearer <your_token>"
```

### 标记单个通知为已读
```bash
curl -X PUT "http://localhost:8080/api/v1/notifications/123/read" \
  -H "Authorization: Bearer <your_token>"
```

### 标记全部通知为已读
```bash
curl -X PUT "http://localhost:8080/api/v1/notifications/read-all" \
  -H "Authorization: Bearer <your_token>"
```

### 删除通知
```bash
curl -X DELETE "http://localhost:8080/api/v1/notifications/123" \
  -H "Authorization: Bearer <your_token>"
```

## 注意事项

### 1. 角色识别机制
Handler 通过以下优先级识别用户角色：
1. 检查 `admin_id` 上下文 → 识别为管理员（`userType = "admin"`）
2. 检查 `providerId` 上下文 → 识别为商家（`userType = "provider"`）
3. 检查 `userId` 上下文 → 识别为普通用户（`userType = "user"`）

### 2. 通知触发时机
通知由后端业务逻辑自动触发，前端无需手动创建：
- **预约流程**：支付意向金 → 商家接单 → 意向金退款
- **方案流程**：提交方案 → 确认方案 → 拒绝方案
- **订单流程**：生成订单 → 支付订单 → 订单过期
- **提现流程**：审核通过 → 审核拒绝 → 提现完成
- **审核流程**：服务商审核 → 作品审核

### 3. Extra 字段解析
`extra` 字段为 JSON 字符串，包含业务相关的扩展数据：
```json
// 预约通知
{
  "bookingId": 789,
  "address": "北京市朝阳区xxx",
  "intentFee": 500.00
}

// 方案通知
{
  "proposalId": 88,
  "bookingId": 789,
  "version": 2,
  "reason": "风格不符合预期"
}

// 订单通知
{
  "orderId": 456,
  "amount": 5000.00
}

// 提现通知
{
  "withdrawId": 123,
  "amount": 10000.00,
  "orderNo": "WD202601250001",
  "reason": "银行账户信息有误"
}
```

### 4. 前端集成建议
- **轮询未读数量**：每 30-60 秒调用 `/unread-count` 更新角标
- **WebSocket 推送**：生产环境建议接入 WebSocket 实时推送通知
- **本地缓存**：已读通知可缓存到本地，减少重复请求
- **跳转处理**：根据 `actionUrl` 和 `relatedType` 跳转到对应业务页面

### 5. 性能优化
- 通知列表已建立索引：`user_id + user_type`、`is_read`、`type`
- 未读数量查询使用 `COUNT` 聚合，性能较好
- 建议前端分页加载，避免一次性加载过多通知

### 6. 数据清理策略
- 建议定期清理 90 天以上的已读通知
- 重要通知（订单、支付相关）可保留更长时间
- 删除操作为物理删除，不可恢复
