# 数据库设计

**最后更新**: 2026-01-25

## 概述

本项目数据库采用 **PostgreSQL 15**，后端使用 **GORM v1.30.0** 作为 ORM 框架进行数据持久化操作。系统设计遵循高内聚低耦合原则，通过逻辑外关联（代码层面维护关系）而非物理外键约束，以提高扩展性和迁移灵活性。

## 核心表结构

### 1. 用户与权限模块

#### users - 用户表
存储平台所有类型的终端用户（业主、服务商、工人、管理员）。

| 字段 | 类型 | 说明 | 约束 |
|------|------|------|------|
| id | bigint | 自增主键 | PK |
| phone | varchar(20) | 手机号 | UNIQUE, INDEX |
| nickname | varchar(50) | 昵称 | |
| avatar | varchar(500) | 头像地址 | |
| password | varchar(255) | 密码哈希 | 不对外暴露 |
| user_type | int8 | 用户类型: 1业主, 2服务商, 3工人, 4管理员 | |
| status | int8 | 状态: 1正常, 0封禁 | DEFAULT: 1 |

#### sys_admins - 系统管理员表
专门用于管理后台的账号。

| 字段 | 类型 | 说明 | 约束 |
|------|------|------|------|
| id | bigint | 主键 | PK |
| username | varchar(50) | 登录名 | UNIQUE, NOT NULL |
| is_super_admin | boolean | 是否超级管理员 | DEFAULT: false |

## 重要字段说明

- **Base.ID**: 采用 `uint64` 类型，映射到 PostgreSQL 的 `bigint`，支持海量数据。
- **Status 字段**: 统一使用 `int8` 类型，通过代码中的常量（如 `OrderStatusPending`）定义枚举含义。
- **金额字段**: 系统内部计算使用 `float64` 处理元，涉及财务结算时建议在应用层或数据库快照（如 `QuoteTotalCent`）中使用“分”作为单位以避免精度问题。
- **JSONB 扩展**: `Notification.extra`、`Provider.certifications` 等字段采用 PostgreSQL 的 `JSONB` 类型，支持灵活的非结构化数据存储和高效查询。

## 索引设计

1.  **唯一索引**:
    - `users.phone`: 保证用户账号唯一。
    - `orders.order_no`: 保证业务订单号唯一。
    - `sys_admins.username`: 保证后台管理员账号唯一。
2.  **复合索引**:
    - `user_follows(user_id, target_id, target_type)`: 优化关注状态查询。
    - `notifications(user_id, user_type, is_read)`: 优化用户未读消息查询。
3.  **普通索引**:
    - 所有逻辑外键字段（如 `project_id`, `user_id`, `provider_id`）均建立普通索引以提升 Join 和过滤性能。
    - `created_at`: 常用作排序字段，在日志和列表查询中建立索引。

## 数据库变更管理

所有数据库变更必须通过 `server/migrations/` 下的 SQL 脚本进行，并记录在 `docs/数据库迁移规范.md` 中。严禁在生产环境手动修改表结构。
