# 安全规范

本规范旨在确立项目的安全开发标准，确保系统免受常见安全漏洞的威胁，保护用户数据和系统资产。

## 1. 认证与授权 (Authentication & Authorization)

### 1.1 JWT (JSON Web Token) 使用规范
项目采用 JWT 进行身份验证，遵循双 Token 机制（Access Token + Refresh Token）。

- **Access Token**:
  - 有效期：1 小时。
  - 存储：移动端存储在 `SecureStorage` 或 `Keychain`；Web 端建议存储在 `httpOnly Cookie`。
  - 必须包含：`userId`/`adminId`、`token_type` (user/admin)、`exp`。
- **Refresh Token**:
  - 有效期：7 天。
  - 作用：用于在 Access Token 过期后获取新的 Access Token。
- **密钥管理**:
  - `JWT_SECRET` 必须通过环境变量设置，严禁硬编码。
  - 生产环境必须使用强随机密钥（建议 64 位：`openssl rand -base64 64`）。
- **验证机制**:
  - 管理后台接口必须验证 `token_type` 是否为 `admin`。

### 1.2 密码处理
- **哈希算法**: 使用 `bcrypt` 算法进行密码哈希，工作因子（Cost）建议为 10 或以上。
- **严禁明文**: 数据库中严禁存储明文密码。
- **验证流程**:
  ```go
  // 后端示例
  err := bcrypt.CompareHashAndPassword([]byte(hashedPassword), []byte(plainPassword))
  ```

### 1.3 权限控制 (RBAC)
- 遵循“最小权限原则”。
- 后端必须在中间件层进行权限检查。
- 管理后台操作必须通过 `RequirePermission` 中间件验证权限标识。

## 2. 数据保护 (Data Protection)

### 2.1 敏感数据加密
- **存储加密**: 对于身份证号、银行卡号等极度敏感数据，必须在数据库中加密存储。
- **加密算法**: 使用 `AES-256-GCM`。
- **密钥要求**: `ENCRYPTION_KEY` 必须从环境变量读取，严禁默认值。若未设置，程序应拒绝启动。

### 2.2 日志安全
- **脱敏处理**: 日志中严禁记录密码、支付密码、完整 Token、完整银行卡号等敏感信息。
- **审计日志**: 管理后台的 `POST/PUT/DELETE` 操作必须记录操作审计日志，包括操作人、IP、操作内容及结果。

### 2.3 错误处理
- **生产环境**: 禁止返回详细的系统错误（如数据库错误堆栈）。返回通用的错误提示（如“系统繁忙”或“参数错误”）。
- **详细日志**: 内部记录详细日志，方便排查。

## 3. 常见漏洞防护 (Vulnerability Prevention)

### 3.1 SQL 注入 (SQL Injection)
- **统一使用 ORM**: 使用 GORM 进行数据库操作。
- **参数化查询**: 严禁直接拼接 SQL 字符串。
- **代码示例**:
  ```go
  // ✅ 正确：参数化查询
  db.Where("username = ?", username).First(&user)

  // ❌ 错误：手动拼接
  db.Raw("SELECT * FROM users WHERE username = '" + username + "'").Scan(&user)
  ```

### 3.2 XSS (跨站脚本攻击)
- **输入过滤**: 对用户输入进行长度和格式验证。
- **输出编码**: 前端 React 默认会进行输出编码，但在使用 `dangerouslySetInnerHTML` 时必须严格审查。
- **安全响应头**: 后端必须设置 `X-XSS-Protection: 1; mode=block`。

### 3.3 CSRF (跨站请求伪造)
- **CORS 白名单**: 严格限制 `Access-Control-Allow-Origin`，禁止使用 `*`。
- **SameSite Cookie**: 如果使用 Cookie 存储 Token，必须设置 `SameSite=Lax` 或 `Strict`。

### 3.4 限流与暴力破解
- **登录保护**: 对登录、注册、验证码发送接口实施限流。
- **锁定机制**: 连续登录失败（如 5 次）后，锁定账号或 IP 30 分钟。

### 3.5 文件上传安全
- **类型检查**: 仅允许预定义的白名单后缀（如 `.jpg`, `.png`, `.webp`）。
- **大小限制**: 限制单个文件大小（如 10MB）。
- **随机命名**: 上传后的文件必须重命名为随机字符串，防止目录遍历。

## 4. 安全检查清单 (Security Checklist)

### 4.1 开发阶段
- [ ] 是否存在硬编码的秘密（Key, Token, Password）？
- [ ] 敏感接口是否添加了限流保护？
- [ ] 是否使用了参数化查询防止 SQL 注入？
- [ ] 敏感数据在日志中是否已脱敏？

### 4.2 部署阶段
- [ ] `SERVER_MODE` 是否设置为 `release`？
- [ ] `JWT_SECRET` 和 `ENCRYPTION_KEY` 是否已更换为生产随机密钥？
- [ ] 是否禁用了生产环境的调试端点（`/api/v1/debug/*`）？
- [ ] 数据库和 Redis 是否配置了强密码并限制了访问 IP？
- [ ] 是否启用了安全响应头？

## 5. 代码参考示例

### 5.1 后端安全响应头 (Go)
```go
func SecurityHeaders() gin.HandlerFunc {
    return func(c *gin.Context) {
        c.Header("X-Content-Type-Options", "nosniff")
        c.Header("X-Frame-Options", "DENY")
        c.Header("X-XSS-Protection", "1; mode=block")
        c.Header("Referrer-Policy", "strict-origin-when-cross-origin")
        c.Next()
    }
}
```
