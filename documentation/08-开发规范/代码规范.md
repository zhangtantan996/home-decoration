# 代码规范

本文档定义了“装修设计一体化平台”项目的代码编写标准和规范。所有开发者必须严格遵守，以确保代码库的一致性、可读性和可维护性。

**最后更新: 2026-01-25**

---

## 1. 命名规范

### 1.1 文件命名
- **Go**: 必须使用 `snake_case.go`（例如：`user_handler.go`, `merchant_service.go`）。
- **React 组件**: 必须使用 `PascalCase.tsx`（例如：`UserList.tsx`, `DashboardIndex.tsx`）。
- **TS 工具/服务/Store**: 使用 `camelCase.ts`（例如：`authStore.ts`, `api.ts`）。
- **TS 类/接口**: 使用 `PascalCase.ts`（例如：`SecureStorage.ts`）。

### 1.2 变量与函数命名
- **Go**:
  - 导出变量/函数/结构体: `PascalCase`（例如：`CreateUser`, `UserRepository`）。
  - 私有变量/函数/结构体: `camelCase`（例如：`getUserByID`, `dbInstance`）。
- **TypeScript/JavaScript**:
  - 变量/函数: `camelCase`（例如：`isLoading`, `fetchData`）。
  - 组件: `PascalCase`（例如：`const UserCard = () => { ... }`）。
  - 常量: `SCREAMING_SNAKE_CASE`（例如：`MAX_RETRY_COUNT`）。

## 2. Go 后端开发规范

### 2.1 强制分层架构
后端代码必须严格遵循以下三层架构：
1.  **Handler (Controller) 层**: 负责解析 HTTP 请求、参数验证、调用 Service 并返回标准响应。
2.  **Service 层**: 负责核心业务逻辑。禁止在 Handler 中编写复杂业务逻辑。
3.  **Repository 层**: 负责与数据库（GORM）或缓存（Redis）进行交互。禁止在 Handler 或 Service 中直接操作底层驱动。

### 2.2 错误处理
- **不要直接 panic**: 在 HTTP 请求处理路径中应始终返回错误，而不是导致崩溃。
- **错误包装**: 使用 `fmt.Errorf("...: %w", err)` 包装错误以保留原始错误上下文。
- **标准响应**: 必须使用 `server/pkg/response` 下的辅助函数返回结果。

### 2.3 代码示例 (Go)

```go
// server/internal/handler/user_handler.go
package handler

import (
    "home-decoration-server/pkg/response"
    "github.com/gin-gonic/gin"
)

func (h *UserHandler) GetUser(c *gin.Context) {
    id := c.Param("id")
    // 1. 调用 Service
    user, err := h.userService.GetUserDetail(id)
    if err != nil {
        // 2. 使用包装的响应函数
        response.Error(c, 404, "用户不存在")
        return
    }
    // 3. 返回成功响应
    response.Success(c, user)
}

// server/internal/service/user_service.go
func (s *UserService) GetUserDetail(id string) (*model.User, error) {
    // 业务逻辑验证
    if id == "" {
        return nil, errors.New("id 不能为空")
    }
    // 调用 Repository
    return s.userRepo.FindByID(id)
}
```

## 3. TypeScript / React 开发规范

### 3.1 类型安全
- **禁止使用 `any`**: 尽量显式定义接口（Interface）或类型（Type）。
- **优先使用 `import type`**: 对于仅用于类型检查的导入，使用 `import type` 以减少构建产物体积。
- **严格模式**: 项目已开启 `strict: true`，禁止通过 `@ts-ignore` 逃避类型检查。

### 3.2 状态管理 (Zustand)
- 统一使用 **Zustand** 进行全局状态管理，禁止随意引入 Redux 或 MobX。
- Store 文件应放在各端的 `src/stores/` 目录下。

### 3.3 导入规范
导入语句应按以下顺序分组：
1.  React/React Native 内置库。
2.  第三方库（Ant Design, Axios 等）。
3.  项目内部公共组件/工具。
4.  当前模块的本地组件/样式。

### 3.4 代码示例 (React)

```tsx
// admin/src/pages/users/UserList.tsx
import React, { useEffect, useState } from 'react';
import { Table, message } from 'antd';
import type { ColumnsType } from 'antd/es/table';
import { adminUserApi } from '../../services/api';

// 1. 定义接口
interface User {
  id: number;
  phone: string;
  nickname: string;
}

const UserList: React.FC = () => {
  const [users, setUsers] = useState<User[]>([]);
  const [loading, setLoading] = useState(false);

  // 2. 异步数据加载
  const loadData = async () => {
    setLoading(true);
    try {
      const res = await adminUserApi.list();
      if (res.code === 0) {
        setUsers(res.data.list);
      }
    } catch (error) {
      message.error('加载失败');
    } finally {
      setLoading(false);
    }
  };

  useEffect(() => {
    loadData();
  }, []);

  const columns: ColumnsType<User> = [
    { title: 'ID', dataIndex: 'id' },
    { title: '昵称', dataIndex: 'nickname' },
  ];

  return <Table loading={loading} dataSource={users} columns={columns} rowKey="id" />;
};

export default UserList;
```

## 4. 通用安全与质量要求

- **禁止硬编码密钥**: 严禁将 JWT Secret、数据库密码、API Key 等提交到代码库。应使用 `.env` 文件或配置中心。
- **日志脱敏**: 在日志中记录用户信息时，必须对手机号、身份证号、密码等敏感字段进行脱敏处理。
- **提交规范**:
  - 提交前必须运行代码格式化（Go: `make fmt`, Mini: `npm run format`）。
  - 提交信息必须遵循 Conventional Commits 规范（例如：`feat(user): add login feature`）。
