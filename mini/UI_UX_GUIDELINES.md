# 小程序 UI/UX 设计与实现规范

> 适用范围：`mini/` 全部页面与组件。
>
> 本文档是对 [DEVELOPMENT_PLAN.md](./DEVELOPMENT_PLAN.md) 与 [DEVELOPMENT_TASKS.md](./DEVELOPMENT_TASKS.md) 的补充要求。目标不是“能用就行”，而是做出审美稳定、信息层次清晰、交互自然的小程序端体验。

## 一、设计目标

小程序端的设计方向固定为：

- 轻奢但不过度装饰
- 信息清楚、操作明确、状态可感知
- 不是通用模板感页面，而是有家装行业气质的页面
- 首先保证转化效率，其次再做视觉强化

关键词：

- 温暖
- 专业
- 干净
- 高级
- 克制

## 二、视觉方向

### 2.1 色彩

- 品牌主色：`#D4AF37`
- 页面底色：浅灰白，不使用纯死白铺满整页
- 卡片底色：白色，配柔和边框或轻阴影
- 强提示色只用于状态与关键动作，不大面积滥用

### 2.2 版式

- 页面必须有明确的首屏重点，不允许所有内容同权平铺
- 信息分组优先用卡片、分段标题、留白，而不是靠细线硬切
- 每个页面必须有 1 个主操作，不允许主次操作权重混乱
- 卡片、筛选栏、底部操作条之间要有节奏感，不允许间距混乱

### 2.3 字体层级

- 页面标题：强调、稳重，不可过小
- 卡片标题：清晰可扫读
- 辅助信息：弱化但可读，不允许灰到看不清
- 数字、价格、评分等重点信息要有明显视觉优先级

## 三、交互要求

### 3.1 通用体验

- 所有列表页都必须有：加载态、空态、错误态
- 所有提交动作都必须有：禁用态、进行中反馈、成功/失败反馈
- 所有主流程页面都必须考虑未登录态
- 所有跨页跳转都应让用户知道“为什么会跳”和“跳过去做什么”

### 3.2 页面操作原则

- 主按钮固定使用高识别度样式，不允许每页随意变体
- 次按钮保持克制，避免和主按钮抢权重
- 危险操作必须二次确认
- 长流程页面底部操作区要固定或足够显眼，不允许用户滑到底找不到动作

### 3.3 表单体验

- 必填项要明确标识
- 错误提示要贴近字段，不只弹 toast
- 输入控件高度、圆角、内边距保持统一
- 表单页面提交前必须能快速扫一眼确认关键字段

## 四、页面级审美标准

### 4.1 首页

- 需要一个足够强的首屏视觉区，不做普通九宫格首页
- 推荐“主 Banner + 快速入口 + 推荐内容分层”的结构
- 入口卡片要有图标、标题、解释，不要只是文字堆叠

### 4.2 服务商列表/详情

- 列表卡片必须体现专业感：头像、评分、标签、案例预览、主动作
- 详情页要有明确的信任建立区：资质、评分、经验、服务区域
- “在线咨询”“立即预约”要始终明确且易点

### 4.3 方案/订单/项目

- 这三类页面的核心是信息层级清晰，而不是花哨
- 状态、金额、节点、时间是第一视觉优先级
- 详情页必须做到用户 3 秒内知道当前进度和下一步动作

### 4.4 消息/聊天

- 会话列表强调未读、最后消息、最近时间
- 聊天气泡要简洁，间距舒服，不做过度装饰
- 聊天页输入区与消息区层次必须稳定，避免杂乱

### 4.5 个人中心

- 不能只是功能入口堆叠
- 需要“用户身份 + 核心资产 + 常用动作”的层次结构
- 个人中心首屏必须能表达“我是谁、我能做什么、我最近要处理什么”

## 五、组件实现要求

### 5.1 组件优先级

- 优先复用 `mini/src/components/*`
- NutUI 可以用，但页面层不应直接散落过多第三方样式细节
- 新增组件先考虑是否可沉淀为项目组件，而不是写死在页面里

### 5.2 图标规则

- 禁止 emoji
- 页面图标优先使用统一 `Icon` 门面
- TabBar 图标由 SVG 源统一导出产物

### 5.3 动效规则

- 允许轻量动效，但必须服务于层级与反馈
- 不做廉价弹跳、夸张缩放、无意义过场
- 推荐使用：
  - skeleton 渐进加载
  - 卡片轻量出现动画
  - 状态切换淡入

## 六、实现与验收清单

开发任何一个页面前，先检查：

1. 这个页面的首要目标是什么
2. 主操作是什么
3. 信息是否有明确层级
4. 图标是否全部为矢量来源
5. 是否有加载态、空态、错误态、未登录态

页面完成后，必须再检查：

1. 首屏是否有重点，不像通用后台表单
2. 主按钮是否明显
3. 卡片、标题、间距是否统一
4. 是否存在大段平铺文字、拥挤布局、操作分散问题
5. 是否在微信开发者工具和真机上都保持清晰可读

## 七、执行原则

1. 功能实现与 UI/UX 优化必须同步推进，不能先堆功能、最后统一补美化。
2. 任何新增页面都要先符合本规范，再进入联调。
3. 如果功能优先导致页面临时可用但视觉粗糙，必须在对应任务单中保留 UI/UX 收尾项，不能默认“以后再说”。
